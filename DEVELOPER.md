# Руководство разработчика onebase

## Структура проекта (пользовательское приложение)

```
my-project/
├── config/
│   └── app.yaml          # имя и версия приложения
├── catalogs/             # YAML-описания справочников
├── documents/            # YAML-описания документов
├── registers/            # YAML-описания регистров накопления
├── inforegs/             # YAML-описания регистров сведений
├── enums/                # YAML-описания перечислений
├── constants/            # YAML-файлы констант
├── reports/              # YAML-описания отчётов
├── printforms/           # YAML-описания печатных форм
├── processors/           # YAML-описания обработок
├── roles/                # YAML-описания ролей (права доступа)
└── src/                  # DSL-скрипты событий (*.os)
    ├── Документ.os           # ПриЗаписи документа
    ├── Документ.posting.os   # ОбработкаПроведения (только для posting: true)
    ├── ОбработкаX.proc.os    # точка входа обработки (Процедура Выполнить)
    └── МодульX.module.os     # общий модуль (функции/процедуры для повторного использования)
```

Запуск в файловом режиме (hot reload):
```bash
onebase dev --project ./my-project --db "postgres://localhost/mydb?sslmode=disable"
```

## Справочники (Catalogs)

Файл: `catalogs/<имя>.yaml`

```yaml
name: Номенклатура
title: Номенклатура товаров   # опционально: подпись для интерфейса
fields:
  - name: Наименование
    type: string
  - name: Артикул
    type: string
  - name: Цена
    type: number
  - name: Активен
    type: bool
  - name: ВидКонтрагента
    type: enum:ВидКонтрагента
```

`name` — идентификатор (используется в URL и DSL, обычно PascalCase).
`title` — человекочитаемое представление (аналог «Синонима» в 1С); если
задано, показывается в меню, заголовках и списках. Полезно для длинных
имён: `name: ПоступлениеНаРасчётныйСчёт` + `title: Поступление на
расчётный счёт`. Если `title` не задан — везде показывается `name`.
При конвертации из 1С (`onebase convert`) синоним объекта переносится в
`title` автоматически (если он отличается от имени).

### Типы полей

| Тип | PostgreSQL | Описание |
|---|---|---|
| `string` | TEXT | Строка |
| `number` | NUMERIC | Число |
| `date` | TIMESTAMPTZ | Дата и время |
| `bool` | BOOLEAN | Да/Нет |
| `reference:X` | UUID | Ссылка на справочник или документ X |
| `enum:X` | TEXT | Значение перечисления X |

### Настройка плитки списка

Плиточный вид списка по умолчанию строится автоматически: первое поле типа
`image` становится картинкой, первый обычный реквизит — заголовком, остальные
реквизиты выводятся строками ниже. Для ручной компоновки добавьте
`tile_view` в YAML справочника или документа:

```yaml
tile_view:
  image: Фото
  title: Наименование
  subtitle: Артикул
  fields: [Цена, Остаток]
```

`image` должен ссылаться на реквизит типа `image`; `title`, `subtitle` и
`fields` — на существующие поля шапки объекта. Если указать `fields: []`,
дополнительные строки под заголовком не выводятся.

Заданный набор `fields` применяется ко **всем** режимам списка — плитка,
страницы, лента и дерево показывают одни и те же колонки (Заголовок +
Подзаголовок + `fields`). Без `tile_view.fields` таблица по-прежнему выводит
все реквизиты.

## Документы (Documents)

Файл: `documents/<имя>.yaml`

```yaml
name: Поступление
title: Поступление товаров   # опционально: подпись для интерфейса (см. Справочники)
posting: true         # опционально: включить режим проведения
fields:
  - name: Номер
    type: string
  - name: Дата
    type: date
  - name: Поставщик
    type: reference:Контрагент
tableparts:
  - name: Товары
    fields:
      - name: Номенклатура
        type: reference:Номенклатура
      - name: Количество
        type: number
      - name: Цена
        type: number
      - name: Сумма
        type: number
```

Табличные части (`tableparts`) — вложенные списки строк внутри документа.

### Режим проведения (`posting: true`)

Если документ помечен `posting: true`, у него два DSL-файла:

| Файл | Вызывается | Назначение |
|---|---|---|
| `src/Документ.os` | при **сохранении** | валидация данных, вычисление полей |
| `src/Документ.posting.os` | при **проведении** | запись движений в регистры |

В интерфейсе для проводимых документов появляются кнопки «Провести» и «Отменить проведение».  
Движения в регистры записываются **только** через процедуру проведения.

## Регистры накопления (Registers)

Файл: `registers/<имя>.yaml`

```yaml
name: ТоварноеДвижение
dimensions:
  - name: Номенклатура
    type: string
resources:
  - name: Количество
    type: number
  - name: Сумма
    type: number
attributes:
  - name: Комментарий
    type: string
```

- **dimensions** — разрезы учёта
- **resources** — числовые показатели (Приход +, Расход −)
- **attributes** — дополнительные данные, не влияющие на итоги

Период движения берётся автоматически из первого поля типа `date` документа-регистратора.

### Итоги регистра (предрасчёт остатков)

Для регистров с большой историей движений можно включить предрасчёт итогов —
таблицу `итоги_<имя>` с текущим остатком по каждому набору измерений:

```yaml
name: ТоварноеДвижение
totals:
  enabled: true          # вести таблицу текущих итогов
dimensions: [...]
resources: [...]
```

- **totals.enabled** — платформа ведёт **помесячные обороты** в той же
  транзакции, что и движения (при проведении/отмене), поэтому итоги всегда
  согласованы. Из них читаются:
  - текущие `Остатки()` — SUM оборотов по всем месяцам;
  - `Остатки(&Момент)` / `Остатки(&Дата)` — обороты месяцев до момента + хвост
    движений внутри месяца момента (скан ограничен одним месяцем вместо всей
    истории);
  - `ОстаткиИОбороты(&Начало, &Конец)` — начальный остаток из итогов, приход/
    расход сканом периода `[Начало, Конец]`, конечный = начальный + приход −
    расход;
- итоги применяются к регистрам **без `attributes`** (их значения таблица итогов
  не хранит); регистр с атрибутами использует расчёт на лету, итоги для него не
  ведутся;
- итоги корректны при проведении задним числом, перепроведении и отмене;
- на обычном пути (fail-safe) остаются: отбор в аргументах `.Остатки(&момент,
  отбор)` — эквивалентно быстро через внешнее `ГДЕ` (`Остатки(&момент)` +
  `ГДЕ товар = …`); `ОстаткиИОбороты` без границ-дат или с периодичностью;
  `Обороты()`. Любой неохваченный случай — корректный обычный расчёт;
- полный пересчёт (после массовой правки в обход движка): `onebase recalc-totals
  --project <dir> --sqlite <файл> [--register <имя>]`.

Оборотные регистры (`kind: turnover`) остатков не имеют — `totals` к ним
неприменим.

## Регистры сведений (InfoRegisters)

Файл: `inforegs/<имя>.yaml`

```yaml
name: ЦеныНоменклатуры
periodic: true         # если true — первичный ключ включает период
dimensions:
  - name: Номенклатура
    type: reference:Номенклатура
resources:
  - name: Цена
    type: number
```

- **dimensions** — ключевые поля записи
- **resources** — значения, хранимые по ключу
- `periodic: true` — добавляет поле `period` в первичный ключ, фиксирует значение во времени

## Константы (Constants)

Файл: `constants/<любое_имя>.yaml` (все файлы из папки загружаются)

```yaml
constants:
  - name: НашаОрганизация
    type: reference:Контрагент
    label: Наша организация
  - name: ОсновнаяВалюта
    type: string
    default: "RUB"
    label: Основная валюта учёта
  - name: ТипКооператива
    type: enum:ТипыКооперативов   # enum-константа рендерится как <select>
    required: true                 # обязательна к заполнению
    label: Тип кооператива
```

Поле `required: true` делает константу обязательной. При сохранении формы
**Администрирование → Константы** сервер проверяет: непустое значение для
`required`, принадлежность значения домену перечисления (`enum:`) и
существование элемента справочника (`reference:`); при нарушении форма
возвращается с понятной ошибкой, а не сохраняет некорректное значение.

В DSL и шаблонах печатных форм доступны через `Константы.ИмяКонстанты`.  
Значения задаются в интерфейсе базы через раздел **Администрирование → Константы**.

## Отчёты (Reports)

Файл: `reports/<имя>.yaml`

```yaml
name: ОстаткиТоваров
title: Остатки товаров
params:
  - name: НаДату
    type: date
    label: На дату
  - name: ВидДвижения
    type: select
    label: Вид движения
    options:
      - ""
      - Приход
      - Расход
query: |
  ВЫБРАТЬ
    Номенклатура,
    СУММА(ВЫБОР КОГДА вид_движения = "Приход" ТОГДА количество ИНАЧЕ -количество КОНЕЦ) КАК Количество
  ИЗ РегистрНакопления.ТоварноеДвижение
  ГДЕ (&НаДату ЕСТЬ ПУСТО ИЛИ period <= &НаДату)
  СГРУППИРОВАТЬ ПО Номенклатура
  УПОРЯДОЧИТЬ ПО Номенклатура
```

### Типы параметров

| Тип | Описание |
|---|---|
| `string` | Текстовое поле |
| `number` | Числовое поле |
| `date` | Поле выбора даты |
| `bool` | Флажок (чекбокс). В запросе параметр — `true`/`false` |
| `select` | Выпадающий список (`options`) |
| `reference:Сущность` | Выбор элемента справочника |

Пустой параметр → NULL. Для опциональной фильтрации:
```
ГДЕ (&Параметр ЕСТЬ ПУСТО ИЛИ поле = &Параметр)
```

### Язык запросов

```
ВЫБРАТЬ [РАЗЛИЧНЫЕ]
  поле [КАК Псевдоним],
  СУММА(поле) КАК Итог,
  ВЫБОР КОГДА условие ТОГДА значение ИНАЧЕ другое КОНЕЦ КАК Поле
ИЗ РегистрНакопления.НазваниеРегистра
   | Справочник.НазваниеСправочника
   | Документ.НазваниеДокумента
[ГДЕ условие]
[СГРУППИРОВАТЬ ПО поле, ...]
[УПОРЯДОЧИТЬ ПО поле [УБЫВ|ВОЗР], ...]
```

**Агрегатные функции:** `СУММА`, `КОЛИЧЕСТВО`, `МИНИМУМ`, `МАКСИМУМ`, `СРЕДНЕЕ`  
**Логика:** `И`, `ИЛИ`, `НЕ`, `ЕСТЬ ПУСТО`  
**Параметры:** `&ИмяПараметра`

## Печатные формы (Print Forms) — макеты v2

Печатная форма OneBase — это **макет** `printforms/<имя>.layout.yaml`: набор
именованных областей с ячейками, спанами, границами по сторонам и параметрами
страницы. Один движок (`internal/sheet`) и одна пара рендереров (HTML + PDF с
кириллицей) обслуживают все три способа формирования:

| Способ | Файлы | Когда |
|---|---|---|
| **Декларативный** | только `<имя>.layout.yaml` с `binding:` | форма без кода — данные привязаны выражениями |
| **DSL** | `<имя>.os` + (опц.) `<имя>.layout.yaml` | нужна логика: циклы, условия, подзапросы |
| **Legacy** | `<имя>.yaml` старого формата (`title/header/table/footer`) | конвертируется на лету; мигрируйте командой `printforms migrate` |

При коллизии имени приоритет: декларативная > DSL > legacy (с предупреждением).

### Формат макета (.layout.yaml)

```yaml
name: РасходнаяНакладная
document: РеализацияТоваров        # документ/справочник-источник ("general" — без привязки)
page:                              # параметры страницы PDF (опционально)
  orientation: portrait            # portrait | landscape (рус. синонимы тоже)
  format: A4                       # A4 | A5 | A3 | Letter | Legal, либо кастом "229x162mm" (Ш×В в мм)
  margins: { top: 10, bottom: 10, left: 10, right: 10 }   # мм
columns:                           # глобальные ширины колонок (для всех областей)
  - width: 40px                    # px | mm; "%" печатью НЕ поддерживается; {} = авто
  - {}
areas:                             # упорядоченный список областей (sequence!)
  - name: Заголовок
    rows:
      - cells:
          - text: "Расходная накладная № {{Номер}} от {{Дата | date}}"
            bold: true
            align: center
            fontSize: 14
            colspan: 5
  - name: ШапкаТаблицы
    rows:
      - cells:
          - { text: "№",          bold: true, border: all }
          - { text: Наименование, bold: true, border: all }
  - name: Строка
    rows:
      - cells:
          - { parameter: Кол0, border: all }
          - { parameter: Кол1, border: all }
  - name: Подвал
    rows:
      - cells:
          - colspan: 5
            borders: { bottom: thin }   # границы по сторонам (приоритет над border)
```

**Поля ячейки:**

| Поле | Тип | Описание |
|---|---|---|
| `text` | string | Статичный текст; поддерживает интерполяцию `{{выражение｜формат}}` |
| `parameter` | string | Имя именованного параметра (привязка в `binding:` или из DSL) |
| `bold` / `italic` | bool | Начертание |
| `fontSize` | int | Размер шрифта, pt |
| `fontFamily` | string | `Times New Roman`/`serif` → PT Serif; иначе → PT Sans |
| `align` | string | `left` / `center` / `right` |
| `valign` | string | `top` / `middle` / `bottom` |
| `backColor` / `textColor` | string | CSS-цвет (`#ffeebb`) |
| `border` | string | legacy-пресет всей рамки: `none` / `thin` / `all` / `thick` |
| `borders` | object | per-side: `left/top/right/bottom` ∈ `none｜thin｜medium｜thick` (приоритет над `border`) |
| `colspan` / `rowspan` | int | Объединение по горизонтали / вертикали |

> **Совместимость макетов.** Старый формат `areas:` как **mapping** (ключ = имя
> области) по-прежнему читается; новый — **sequence** с полем `name:` (порядок
> гарантирован → используется как default `sequence` в binding). Сохранение
> пишет sequence.

> **Кастомный размер страницы.** Кроме именованных форматов, `format` принимает
> `"Ш×В"` в мм (`229x162mm`, `100x50`, `210,5x297`) — для нестандартных бланков
> (конверты, ярлыки). Размер **литеральный**, как CSS `@page size: W H`:
> `orientation` к нему не применяется (ширина/высота заданы явно). Работает и в
> DSL: `ТабДок.РазмерСтраницы = "229x162mm"`.

### Декларативная привязка данных (binding)

Без `.os`-кода: `binding:` связывает параметры с выражениями и тиражирует
область по строкам табличной части.

```yaml
binding:
  sequence: [Заголовок, ШапкаТаблицы, Строка, Итоги, Подвал]  # порядок вывода (default = порядок areas)
  repeat_header: ШапкаТаблицы          # повтор области на каждой странице PDF
  parameters:                          # параметр ← выражение (контекст документа)
    Покупатель: Покупатель.Наименование
    ИтогоСумма: "Итог.Товары.Сумма | number:2"
  repeat:
    - area: Строка                     # тиражируется по строкам ТЧ Товары
      source: Товары
      parameters:
        Кол0: "@row"                   # номер строки (1-based)
        Кол1: Номенклатура.Наименование
        Кол4: "Сумма | number:2"
```

**Язык выражений** (общий для `parameters`, `{{...}}` и legacy):

- `Поле` — поле строки ТЧ или документа; `Поле.ПодПоле` — поле ссылки
- `@row` — номер текущей строки ТЧ
- `Константы.Имя` — глобальная константа
- `Итог.<ТЧ>.<Поле>` — сумма числовой колонки табличной части
- `выражение | формат` — форматтер (`number:2`, `date`, …)

Параметр без записи в `parameters` автопривязывается к одноимённому полю.

**Многостраничная регламентная форма.** Рабочий пример — `examples/archive`
(внутренняя опись документов дела): `repeat_header` повторяет шапку таблицы на
каждой странице PDF, `repeat` тиражирует строки описи по табличной части; длинная
опись автоматически разбивается на страницы. Структурный HTML для таких форм не
нужен — многоуровневые шапки, границы по сторонам и разрывы страниц даёт сам макет.

### DSL-путь (ТабличныйДокумент)

`.os`-скрипт в `printforms/` строит документ программно. Первая строка —
привязка: `// Документ: РеализацияТоваров`. Точка входа — `Функция Сформировать()`,
возвращающая `ТабличныйДокумент`. Области берутся из парного `.layout.yaml`
через `ПолучитьОбласть`, параметры заполняются через `Область.Параметры.Имя`:

```
// Документ: РеализацияТоваров
Функция Сформировать()
    ТабДок = Новый ТабличныйДокумент
    Шапка = ТабДок.ПолучитьОбласть("Шапка")
    Шапка.Параметры.Покупатель = Покупатель.Наименование
    ТабДок.Вывести(Шапка)
    Для Каждого Стр Из Товары Цикл
        СтрОбл = ТабДок.ПолучитьОбласть("Строка")
        СтрОбл.Параметры.Товар = Стр.Номенклатура.Наименование
        ТабДок.Вывести(СтрОбл)
    КонецЦикла
    Возврат ТабДок
КонецФункции
```

Свойства страницы из DSL: `ТабДок.ОриентацияСтраницы`, `.РазмерСтраницы`,
`.ПоляПечати`. Картинки: `ТабДок.Нарисовать(<data-URI>)` / `Ячейка.Рисунок`.

### Рендеринг: HTML и PDF

- **HTML** — `GET …/print/{form}`. Кнопка «Печать» на карточке документа.
- **PDF** — `GET …/print/{form}/pdf`. Кириллица — встроенными шрифтами
  **PT Serif / PT Sans** (без транслитерации). Реализация — `internal/sheet/pdf.go`
  (рисуем по координатам: сетка колонок, авто-высоты строк, спаны, границы по
  сторонам, фоны, разрывы страниц с повтором `repeat_header`). Из DSL —
  `ТабДок.Записать(Имя, "pdf")` → base64.

Полный маршрут: `/ui/{kind}/{entity}/{id}/print/{form}[/pdf]`. Старый
`/print-dsl/…` → 301-редирект на единый `/print/…`.

**Картинки в ячейках.** `Cell.Picture` (DSL `Нарисовать` / `Ячейка.Рисунок`)
рендерится в HTML (`<img>`) и PDF (встроенный образ). Поддерживается **data-URI**
картинок (`data:image/png;base64,…`, PNG/JPEG/GIF) и http(s)-URL (только HTML —
в PDF внешние URL не загружаются). Файловые пути и произвольные схемы (`javascript:`,
`file:`) игнорируются. Картинка вписывается в ячейку с сохранением пропорций.

**Ограничение rowspan через разрыв страницы (PDF).** Ячейка с `rowspan>1`,
попадающая на разрыв страницы, рисуется за нижним полем и может наложиться на
шапку следующей страницы (MVP). `onebase check` выдаёт **предупреждение** (не
ошибку), если объединение по строкам есть в области, тиражируемой по ТЧ
(`binding.repeat`) — именно там разрыв почти неизбежен на длинном документе.
Полный перенос rowspan через страницу — в follow-up плана 64.

### Импорт макета из PDF

Конфигуратор → «Создать макет из PDF» (upload + № страницы) → POST
`/configurator/layout/import-pdf` создаёт **черновик** макета. Движок —
`internal/pdfimport` (вендоренный `pdfparse`, BSD-3): извлекает сетку линий,
регионы со спанами, тексты по содержимому стрима, ориентацию/формат страницы.
Пригоден для **векторных PDF с текстовым слоем** (УПД/счета/акты из 1С/Excel);
сканы → ошибка. Точность ±1мм; выравнивания и многострочность доводятся руками.
Недоверенный PDF обрабатывается под recover + таймаут + лимиты размера, в
admin-only зоне.

### Миграция legacy-форм

Старый плоский формат (`title/header/table/footer`) читается на лету
(`ConvertLegacy`: Title→заголовок, Header/Footer→области, Table→ШапкаТаблицы+
Строка+Итоги+binding). Перепишите файлы в макет v2 командой:

```bash
onebase printforms migrate --project examples/trade        # удаляет исходные .yaml
onebase printforms migrate --project examples/trade --keep # сохраняет исходники
```

`onebase check` помечает непустые legacy-формы предупреждением «выполните
printforms migrate».

### Визуальный редактор макетов в конфигураторе

Split-pane (YAML ↔ конструктор): клик по ячейке → панель свойств (текст,
параметр, шрифт, цвет, выравнивание, объединение, границы по сторонам). Линейка
ширин колонок и ручки высот строк, тогглы границ Л/В/П/Н, изменение порядка
областей, «Разрезать область перед строкой», панель «Данные» (дерево
реквизитов/ТЧ/констант → клик привязывает параметр), предпросмотр через POST
`/configurator/layout/preview`. Создание с нуля: «+ Печатная форма (макет)» у
сущности и «Создать макет» у `.os` без layout.

## Обработки (Processors)

Обработка — произвольный DSL-скрипт с параметрами, запускаемый вручную из интерфейса.

**Дескриптор:** `processors/<имя>.yaml`
```yaml
name: ПересчётЦен
title: Пересчёт цен номенклатуры
params:
  - name: Процент
    type: number
    label: Процент изменения цены
  - name: Группа
    type: string
    label: Группа (оставьте пустым для всех)
  - name: ОчищатьПередЗагрузкой
    type: bool
    label: Очистить документы перед загрузкой
    default: true
```

**Скрипт:** `src/<имя>.proc.os` (имя без учёта регистра первой буквы)
```
Процедура Выполнить()
    Процент = Параметры.Процент
    Если Процент = 0 Тогда
        Сообщить("Ошибка: процент не задан")
        Возврат
    КонецЕсли
    Сообщить("Пересчёт цен на " + Строка(Процент) + "%")
    Сообщить("Готово")
КонецПроцедуры
```

- Точка входа — `Процедура Выполнить()` (обязательно)
- Параметры доступны через объект `Параметры.ИмяПараметра`
- `Сообщить(текст)` — вывод в блок результата на странице обработки
- Раздел **Обработки** появляется в левом меню автоматически

### Типы параметров обработки

| Тип | Описание |
|---|---|
| `string` | Текстовое поле |
| `number` | Числовое поле |
| `date` | Поле выбора даты |
| `bool` | Флажок (чекбокс). В DSL `Параметры.Имя` — `Истина`/`Ложь` |

Любому параметру можно задать `default` — значение, подставляемое при первом
открытии формы обработки. Для `type: bool` пишите `default: true` или
`default: false`.

## Общие модули (Common Modules)

Общий модуль — библиотека функций/процедур, доступных из любого DSL-скрипта.

**Файл:** `src/<имя>.module.os`

```
Функция РассчитатьНДС(Сумма)
    Возврат Сумма * 0.2
КонецФункции

Функция СуммаПрописью(Сумма)
    Возврат "Сумма: " + Строка(Сумма) + " руб."
КонецФункции
```

- Экспортируются все `Процедура` и `Функция` верхнего уровня
- Вызываются по имени напрямую из любого `.os`-скрипта (документа или обработки)
- Поиск: сначала локальная процедура в файле, потом все общие модули

## DSL-скрипты

### Процедура ПриЗаписи / OnWrite

Файл: `src/<имя_документа>.os` — вызывается при **сохранении** документа.  
`Error(...)` / `Ошибка(...)` отменяет сохранение.

```
Процедура ПриЗаписи()
    Если this.Номер = "" Тогда
        Error("Номер обязателен");
    КонецЕсли;

    Движения.ТоварноеДвижение.Очистить();
    Для Каждого Строка Из this.Товары Цикл
        Строка.Сумма = Строка.Количество * Строка.Цена;
        Дв = Движения.ТоварноеДвижение.Добавить();
        Дв.ВидДвижения  = "Приход";
        Дв.Номенклатура = Строка.Номенклатура;
        Дв.Количество   = Строка.Количество;
        Дв.Сумма        = Строка.Сумма;
    КонецЦикла;
КонецПроцедуры
```

### Процедура ОбработкаПроведения / OnPost

Файл: `src/<имя_документа>.posting.os` — только для документов с `posting: true`.  
Вызывается при **проведении** документа.

```
Процедура ОбработкаПроведения()
    Движения.ОстаткиТоваров.Очистить();
    Для Каждого Строка Из this.Товары Цикл
        Дв = Движения.ОстаткиТоваров.Добавить();
        Дв.ВидДвижения  = "Расход";
        Дв.Номенклатура = Строка.Номенклатура;
        Дв.Склад        = this.Склад;
        Дв.Количество   = Строка.Количество;
        Дв.Сумма        = Строка.Сумма;
    КонецЦикла;
КонецПроцедуры
```

### Доступные конструкции

```
// Присваивание
x = выражение;

// Условие
Если условие Тогда
    ...
[Иначе
    ...]
КонецЕсли;

// Цикл по коллекции
Для Каждого Строка Из this.ТабличнаяЧасть Цикл
    ...
КонецЦикла;

// Числовой цикл
Для i = 1 По 10 Цикл
    ...
КонецЦикла;

// Цикл с предусловием
Пока условие Цикл
    ...
КонецЦикла;

// Ранний выход / возврат значения
Возврат;
Возврат выражение;

// Объявление процедуры
Процедура МояПроцедура(Параметр1, Параметр2)
    ...
КонецПроцедуры

// Объявление функции (возвращает значение)
Функция МояФункция(Аргумент)
    Возврат Аргумент * 2;
КонецФункции

// Арифметика: + - * /
// Сравнения:  = <> > < >= <=

// Поля документа
this.ИмяПоля

// Движения регистра
Движения.НазваниеРегистра.Очистить()
Дв = Движения.НазваниеРегистра.Добавить()
Дв.ВидДвижения = "Приход"   // или "Расход"
```

### Область видимости процедур

По умолчанию DSL сохраняет legacy-поведение: вызванная процедура может увидеть
локальные переменные вызывающей процедуры. Для новых конфигураций включайте
строгую лексическую область видимости:

```yaml
dsl:
  strict_lexical_scope: true
```

В строгом режиме процедура видит только свои параметры, локальные переменные,
переменные модуля и платформенные переменные запуска. Значения между
helper-процедурами передаются явно через параметры и результат функции. Перед
включением прогоните `onebase check --lint`; после включения `onebase check`
будет блокировать `dsl.cross-scope-read`.

### Встроенные функции

**Ошибки:** `Error(текст)` / `Ошибка(текст)` — отменить сохранение.  
**Вывод:** `Сообщить(текст)` / `Message(текст)` — вывод в обработках.

**Даты:** `Today()` / `ТекущаяДата()`, `Now()` / `ТекущаяДатаВремя()`, `Year(d)` / `Год(d)`, `Month(d)` / `Месяц(d)`, `Day(d)` / `День(d)`, `Date(...)` / `Дата(...)`, `AddDay(d,n)` / `ДобавитьДень(d,n)`, `AddMonth(d,n)` / `ДобавитьМесяц(d,n)`, `AddYear(d,n)` / `ДобавитьГод(d,n)`

Конструктор `Дата(Год, Месяц, День[, Час, Минута, Секунда])`; также принимает
строку (`Дата("2026-05-11")`) или число `ГГГГММДД` (`Дата(20260511)`).
Арифметика дат — в секундах, как в 1С: `Дата + 86400` → следующий день,
`Дата - Дата` → разница в секундах. Сравнение (`<`, `>`) — хронологическое.

**Строки:** `Str(n)` / `Строка(n)`, `Number(s)` / `Число(s)`, `Upper(s)` / `ВРег(s)`, `Lower(s)` / `НРег(s)`, `TrimAll(s)` / `СокрЛП(s)`, `Left(s,n)` / `Лев(s,n)`, `Right(s,n)` / `Прав(s,n)`, `Mid(s,start,len)` / `Сред(s,start,len)`, `StrLen(s)` / `СтрДлина(s)`, `StrFind(s,sub)` / `СтрНайти(s,sub)`

**Математика:** `Round(n,d)` / `Окр(n,d)`, `Abs(n)` / `Абс(n)`, `Int(n)` / `Цел(n)`, `Max(a,b)` / `Макс(a,b)`, `Min(a,b)` / `Мин(a,b)`

### Билингвальность

Все ключевые слова работают на русском и английском одновременно:

| Русский | English |

## Лаунчер информационных баз

`onebase start` открывает лаунчер — окно запуска информационных баз.

### Регистр баз: `~/.onebase/ibases.yaml`

```yaml
bases:
  - id: <uuid>
    name: Склад
    config_source: database   # или: file
    path: ""                  # только для file-режима
    db: postgres://localhost/sklad?sslmode=disable
    port: 8080
    created: 2026-04-24T00:00:00Z
```

### Два режима конфигурации

**`config_source: file`** — конфигурация в папке на диске:
- Удобно для разработки, git-friendly
- `onebase dev --project <путь>` — hot reload при изменении файлов
- Путь к папке виден в лаунчере

**`config_source: database`** — конфигурация в PostgreSQL (`_onebase_config`):
- Используется для готовых баз у пользователей
- Создаётся из лаунчера («Добавить» → галочка «Создать с нуля»)
- Редактирование: кнопка **«Конфигуратор»** → «Выгрузить» → правка → «Загрузить»
- Выгрузка попадает в `~/.onebase/workspace/<base-id>/`

### Рекомендуемый поток: Git → релиз → прод

Для командной разработки держите конфигурацию в файловом режиме:

```bash
onebase dev --project ./my-project --db "postgres://localhost/mydb?sslmode=disable"
git status
git commit
```

Git остаётся источником правды для разработчиков: код-ревью, CI, история
изменений и ветки живут там. Перед релизом прогоняйте проверку:

```bash
onebase check --project ./my-project
```

На прод накатывайте уже выбранную ревизию из Git в БД-режим:

```bash
onebase deploy \
  --project ./my-project \
  --db "postgres://user:pass@server/prod?sslmode=disable" \
  --message "release 1.4.0"
```

`deploy` загружает YAML/DSL-файлы в `_onebase_config`, применяет DDL-миграции
и создаёт одну запись в `_config_versions` с сообщением релиза. После этого
прод запускается без файлов проекта:

```bash
onebase run --config-source database --db "postgres://user:pass@server/prod?sslmode=disable"
```

Такой поток не заменяет Git. Он закрывает другую задачу: в самой продовой базе
видно, какая конфигурация была применена, и появляется техническая основа для
diff/rollback без доступа к репозиторию.

### Конфигуратор

Кнопка **Конфигуратор** в лаунчере открывает визуальный редактор конфигурации:

- **Конфигурация** — имя и версия приложения
- **Справочники / Документы / Регистры / Регистры сведений** — добавление, редактирование полей
- **Перечисления** — список значений
- **Константы** — имя, тип, подпись
- **Отчёты** — SQL-запрос и параметры (с возможностью добавлять/удалять параметры)
- **Общие модули** — просмотр DSL-кода
- **Обработки** — параметры обработок

Создаются автоматически при первом запуске базы:

| Таблица | Назначение |
|---|---|
| `_onebase_config` | Конфигурация в БД-режиме |

## Журнал регистрации

Журнал регистрации (аналог одноимённого механизма 1С) фиксирует действия в
служебной таблице `_audit`: создание, изменение (по одной записи на каждое
изменённое поле — со старым и новым значением), удаление, проведение и отмену
проведения документов, вход и выход пользователей.

Записи аудита привязываются к пользователю из HTTP-сессии. Если база работает
без авторизации (таблица `_users` пуста) — события всё равно пишутся, но с
пустым именем пользователя (в просмотре показывается как «(анонимно)»).

### Настройки

Открываются в **конфигураторе → меню → Журнал регистрации**. Это свойство
конкретной информационной базы (хранится в `_settings`), а не конфигурации.

- **Вести журнал регистрации** — главный переключатель. Выключен → не пишется
  ничего.
- **Регистрировать события** — отдельные флажки по типам:
  - Создание объектов
  - Изменение объектов
  - Удаление объектов
  - Проведение / отмена
  - Вход / выход пользователей

По умолчанию журнал включён, пишутся все события изменения данных и
проведение; вход/выход — выключен (шумно для однопользовательских баз).

Просмотр последних 100 записей — на той же странице конфигуратора.

> Аудит фиксирует операции, выполненные через веб-интерфейс. Фоновые операции
> (регламентные задания, миграция, восстановление из бэкапа) идут вне
> HTTP-контекста и в журнал не попадают.

## Пользователи и авторизация

По умолчанию (пустая таблица `_users`) — доступ без пароля.

Как только создан хотя бы один пользователь — при входе требуется логин и пароль.

### Создание пользователей

В веб-интерфейсе базы: **Администрирование → Пользователи → + Добавить**.

Поля: логин, полное имя, пароль, флаг «Администратор».

### Роли

- **Администратор** (`is_admin = true`) — полный доступ + управление пользователями
- **Обычный пользователь** — доступ к данным базы

### Авторизация через лаунчер

При нажатии «Предприятие» в лаунчере — если база защищена, перед открытием появится форма входа. После успешного входа открывается веб-интерфейс уже с активной сессией.

### Сессии и параллельная работа

Пользователь может иметь несколько активных сессий одновременно (план 78): новый вход
не завершает прежние, вход в конфигуратор не выбивает сессию Предприятия. Сессии живут
в таблице `_sessions` (TTL 24 часа, истёкшие подчищаются при каждом логине); помимо
токена и срока хранится служебная мета: `public_id` (безопасный идентификатор для UI,
не токен), `kind` (`enterprise`/`configurator`), `created_at`, `last_seen_at`
(обновляется middleware не чаще раза в 5 минут), `ip`, `user_agent`.

Управление: экран «Активные пользователи» показывает по строке на сессию и умеет
завершить одну сессию (по `public_id`) или все сессии пользователя; в профиле есть
«Выйти со всех устройств». Смена пароля администратором завершает все сессии
пользователя, самостоятельная — все, кроме текущей. События пишутся в аудит:
`session_kick`, `session_kick_all`, `password_change_sessions_revoked`,
`session_displaced`.

Опциональная политика «Максимум сессий на пользователя» (ключ
`auth.max_sessions_per_user` в `_settings`, поле на экранах активных сессий):
0/пусто — безлимит (по умолчанию), 1 — фактически поведение прежних версий,
N — на вырост. При превышении лимита новый вход **вытесняет** самую давнюю по
активности сессию Предприятия (не отказывает во входе — брошенная сессия при
TTL 24 ч заблокировала бы пользователя); сессии конфигуратора не считаются и
не вытесняются.

Кнопка «Новое окно» в лаунчере (`POST /bases/{id}/start-isolated?mode=auto|native|browser`)
открывает базу в отдельном окне с изолированным профилем из
`~/.onebase/browser-profiles/<base-id>/profile-N/` — свой cookie-jar позволяет войти
под другим пользователем. Режим `native` (GUI-сборка под Windows) — второе нативное
WebView2-окно: лаунчер запускает сам себя (`onebase window --url ...`) с переменной
`ONEBASE_WEBVIEW_PROFILE`, которую читает патч `webview.h` в `third_party/webview_go`
(vendored, см. README там же). Режим `browser` — внешний Chromium-браузер с
`--user-data-dir`. `auto` (по умолчанию) выбирает native, если сборка умеет, иначе
браузер; выбор доступен в меню «▾» у кнопки. Профили работают как «рабочие места»:
клик занимает младший свободный профиль (занятость — по lock-файлу Chromium, включая
`EBWebView/lockfile` нативных окон), профиль помнит вход. Очистка — «Очистить
изолированные профили» (`POST /bases/{id}/profiles/clean`), удаляются только незанятые.

## Транзакции в DSL

DSL-скрипты обработок могут явно управлять PostgreSQL-транзакцией через три функции:

```
НачатьТранзакцию()            // начать транзакцию (или SAVEPOINT при вложении)
ЗафиксироватьТранзакцию()     // COMMIT (или RELEASE SAVEPOINT)
ОтменитьТранзакцию()          // ROLLBACK (или ROLLBACK TO SAVEPOINT)

// Английские псевдонимы:
BeginTransaction() / CommitTransaction() / RollbackTransaction()
```

**Типичный паттерн — атомарный импорт:**

```
Процедура Выполнить()
    НачатьТранзакцию();
    Попытка
        Для Каждого Строка Из ПолучитьДанные() Цикл
            ОбработатьСтроку(Строка);
        КонецЦикла;
        ЗафиксироватьТранзакцию();
    Исключение
        ОтменитьТранзакцию();
        Сообщить("Откат: " + ОписаниеОшибки());
    КонецПопытки;
КонецПроцедуры
```

**Вложенные транзакции** реализованы через `SAVEPOINT`:

```
НачатьТранзакцию();          // BEGIN
    НачатьТранзакцию();      // SAVEPOINT sp1
    ОтменитьТранзакцию();    // ROLLBACK TO SAVEPOINT sp1
ЗафиксироватьТранзакцию();   // COMMIT
```

**Инъекция в тест/обработчик:**

```go
state := interpreter.NewTxState(ctx)
extra := interpreter.NewTxFunctions(state, db) // db реализует interpreter.TxDB
extra["МояФункция"] = interpreter.BuiltinFunc(func(args []any, ...) (any, error) {
    // Использовать state.Ctx() вместо исходного ctx — внутри транзакции
    db.Pool().Exec(state.Ctx(), "INSERT INTO ...", ...)
    return nil, nil
})
interp.Run(proc, nil, extra)
```

## Управляемые блокировки (БлокировкаДанных)

Если два документа проводятся параллельно и затрагивают **одни и те же**
данные (одна Номенклатура на одном Складе), расчёт остатков одного может
не увидеть движения другого. `БлокировкаДанных()` сериализует такие
проведения: пока первое держит блокировку по набору измерений, второе с
пересекающимся набором ждёт.

### Синтаксис

```
Процедура ОбработкаПроведения()
    Блокировка = БлокировкаДанных();

    Для Каждого Строка Из this.Товары Цикл
        Элемент = Блокировка.Добавить("ОстаткиТоваров");
        Элемент.УстановитьЗначение("Номенклатура", Строка.Номенклатура);
        Элемент.УстановитьЗначение("Склад", this.Склад);
    КонецЦикла;

    Блокировка.Заблокировать();   // дальше — монопольный доступ к этим ключам

    // ... чтение остатков и запись движений ...

    Блокировка.Разблокировать();  // опционально — см. ниже
КонецПроцедуры
```

### API

| Метод | Описание |
|---|---|
| `БлокировкаДанных()` / `DataLock()` | Создать объект блокировки |
| `.Добавить(ИмяРегистра)` / `.Add(...)` | Добавить элемент блокировки → возвращает элемент |
| `Элемент.УстановитьЗначение(Измерение, Значение)` / `.SetValue(...)` | Задать значение измерения для элемента |
| `.Заблокировать()` / `.Lock()` | Взять блокировки по всем добавленным элементам |
| `.Разблокировать()` / `.Unlock()` | Отпустить блокировки |

### Поведение

- Блокировки **гранулярные**: ключ — `регистр + набор значений измерений`.
  Два проведения с непересекающимися наборами идут параллельно; с
  пересекающимся — сериализуются.
- Ключи берутся в детерминированном (отсортированном) порядке — взаимной
  блокировки (deadlock) между двумя проведениями не возникает.
- `.Разблокировать()` вызывать **не обязательно**: платформа отпускает все
  удерживаемые блокировки автоматически при завершении проведения.
- Область действия — **один процесс onebase**. Параллельные HTTP-запросы
  внутри процесса сериализуются; для SQLite этого достаточно, так как сам
  SQLite допускает только одного писателя.

## Работа с JSON в DSL

```
// Парсинг JSON-строки
данные = ПрочитатьJSON("{""имя"":""Иван"",""возраст"":30}");
Сообщить(данные.Получить("имя"));        // Иван

// Массив
arr = ПрочитатьJSON("[1,2,3]");
Сообщить(arr.Количество());              // 3

// Вложенные структуры
список = ПрочитатьJSON("{""товары"":[{""имя"":""A""}]}");
Для Каждого Товар Из список.Получить("товары") Цикл
    Сообщить(Товар.Получить("имя"));
КонецЦикла;

// Сериализация
м = Новый Соответствие;
м.Вставить("имя", "Иван");
м.Вставить("сумма", 1000);
текст = ЗаписатьJSON(м);    // → {"имя":"Иван","сумма":1000}
```

**Соответствие типов JSON → DSL:**

| JSON | DSL |
|---|---|
| object | Соответствие |
| array | Массив |
| string | строка |
| number (целое) | число |
| number (дробное) | число |
| true/false | Булево |
| null | Неопределено |

**Английские псевдонимы:** `ReadJSON` / `WriteJSON`

**Строки DSL поддерживают `""` как экранированную кавычку** (аналог 1С):

```
// "" → один символ "
текст = "say ""hello""";    // → say "hello"
json  = "{""a"":1}";        // → {"a":1}
```

## Предопределённые элементы справочников

Статические записи справочника, которые всегда присутствуют в БД и не могут быть удалены пользователем. Задаются в YAML, синхронизируются при каждом запуске.

### YAML

```yaml
name: Валюта
fields:
  - name: Наименование
    type: string
  - name: Код
    type: string
predefined:
  - name: Рубль
    fields:
      Наименование: Российский рубль
      Код: RUB
  - name: Доллар
    fields:
      Наименование: Доллар США
      Код: USD
```

### DSL-доступ

```
// ПредопределённыеЗначения.<Справочник>.<Имя> → UUID записи
валюта = ПредопределённыеЗначения.Валюта.Рубль;
this.ВалютаУчёта = ПредопределённыеЗначения.Валюта.Рубль;

Если this.Валюта = ПредопределённыеЗначения.Валюта.Рубль Тогда
    // ...
КонецЕсли;

// Английский псевдоним:
v = PredefinedValues.Валюта.Рубль;
```

### Поведение

- При запуске `onebase dev/run` — upsert каждого предопределённого элемента по `_predefined_name`. UUID никогда не меняется; поля обновляются.
- В UI списка справочника — предопределённые элементы отмечены иконкой ★.
- Попытка удалить или пометить предопределённый элемент → ошибка.
- В БД добавляются колонки `_predefined_name TEXT` и `_is_predefined BOOLEAN` только для справочников с `predefined:` блоком.

### Ссылки между предопределёнными элементами

В ссылочном поле предопределённого элемента можно указать **имя другого
предопределённого элемента** — платформа подставит его UUID при синхронизации.

```yaml
# catalogs/Организация.yaml
name: Организация
fields:
  - name: Наименование
    type: string
predefined:
  - name: ГоловнойОфис
    fields:
      Наименование: Головной офис

# catalogs/Склад.yaml
name: Склад
fields:
  - name: Наименование
    type: string
  - name: Организация
    type: reference:Организация
predefined:
  - name: ОсновнойСклад
    fields:
      Наименование: Основной склад
      Организация: ГоловнойОфис      # ← имя предопределённого из справочника Организация
```

- **Self-reference** — ссылка на предопределённый элемент **того же**
  справочника (например, `БазовыйТип: Закупочная`).
- **Cross-reference** — ссылка на предопределённый элемент **другого**
  справочника (как в примере выше).
- Справочники синхронизируются в порядке зависимостей: тот, на чьи
  предопределённые ссылаются, обрабатывается раньше.
- Если имя не соответствует ни одному предопределённому элементу целевого
  справочника — синхронизация завершается ошибкой (ловит опечатки).
- Циклические self-reference зависимости → ошибка с указанием цепочки.

## Журналы документов (Document Journals)

Журнал объединяет несколько документов в единую таблицу через `UNION ALL`.

Файл: `journals/<имя>.yaml`

```yaml
name: ВсеОперации
title: Все операции
documents:
  - ПоступлениеТоваров
  - РеализацияТоваров
columns:
  - field: Дата
    label: Дата
    format: date          # явный формат (авто-определяется для "дата" по имени)
  - field: Контрагент
    label: Контрагент
    fallback: [Поставщик, Покупатель]   # COALESCE из нескольких полей
  - field: Сумма
    label: Сумма
filters:
  - field: Дата
    type: date_range
  - field: Контрагент
    type: reference:Контрагент   # показывает выпадающий список
  - field: Номер
    type: string                 # текстовый поиск
```

**Семантика fallback**: для колонки «Контрагент» с `fallback: [Поставщик, Покупатель]` в SQL строится `COALESCE(поставщик_id, покупатель_id)`. Значения UUID разрешаются в имена через `resolveJournalRefs`.

**URL:** `/ui/journal/<lowercase-name>`

## Подсистемы (Subsystems)

Подсистемы — разделы навигации в верхнем таб-баре. Позволяют сгруппировать объекты по бизнес-областям (Продажи / Закупки / Склад).

Файл: `subsystems/<имя>.yaml`

```yaml
name: Продажи
title: Продажи
order: 10          # порядок в таб-баре
roles: [Менеджер]  # необязательно: whitelist ролей (см. ниже)
contents:
  documents:  [РеализацияТоваров]
  catalogs:   [Контрагент, Номенклатура, Валюта]
  reports:    [ОстаткиТоваров]
  processors: [ПересчётЦен]
  pages:      [ПанельРуководителя]   # план 66
  inforegs:   []
  registers:  []
```

Один объект может входить в несколько подсистем (аналогично 1С). Если папка `subsystems/` отсутствует или пуста — интерфейс работает в режиме плоской навигации (как раньше).

**Как работает:**
- При наличии подсистем в UI появляется горизонтальный таб-бар под топбаром.
- Клик на таб → URL параметр `?subsystem=ИмяПодсистемы` → боковая nav фильтруется по contents.
- Если `?subsystem` не указан — отображается первая подсистема (по `order`).
- Без подсистем (`subsystems/` пуста) — плоская nav со всеми объектами.

**Видимость по правам и ролям:**
- Раздел показывается пользователю, только если в его `contents` есть хотя бы один доступный объект (`read` для справочников/документов/регистров, `run` для отчётов/обработок; журнал доступен, если читается хотя бы один его документ; страница — по своим `roles`). Открытый деплой без пользователей и администратор видят всё.
- Необязательный whitelist `roles: [Роль1, Роль2]` дополнительно ограничивает раздел перечисленными ролями — та же семантика, что у страниц (план 66) и HTTP-сервисов (план 61). Пустой/отсутствующий список = видимость только по правам.
- Раздел без `contents` (только рабочий стол) виден всем: гейтить нечем — при необходимости используйте `roles:`.
- Прямой заход `?subsystem=Скрытый` отвечает 403; вход при скрытой глобальной «Главной» редиректит в первый **видимый** раздел.
- Виджеты рабочего стола, чей источник данных пользователю недоступен, не рендерятся вовсе (карточка и опустевший ряд пропадают); настоящие ошибки (битый запрос/SQL) остаются видимыми. У actions-виджета кнопки создания фильтруются по праву `write`.
- Опечатки в `roles:` ловит `onebase check --lint` (предупреждение `rbac.unknown-role`; роль может существовать только в БД, поэтому это не жёсткая ошибка).

## Выполнение команд ОС (`ВыполнитьКоманду`, план 67)

DSL может запускать процессы на сервере и дожидаться завершения. Это
**исполнение кода на сервере (RCE)** — самая мощная и опасная возможность,
поэтому она **выключена по умолчанию** и гейтится отдельным флагом базы
`exec.enabled` (Система → Настройки → «Разрешить выполнение команд ОС»), как
предохранитель сети. В песочнице (ИИ-генерация, marketplace) запрещена всегда;
после восстановления из бэкапа флаг сбрасывается в выкл.

```bsl
// Команда и аргументы — РАЗДЕЛЬНО, без shell (инъекция невозможна).
Рез = ВыполнитьКоманду("git", Массив("status", "--porcelain"), 30);
Сообщить("Код возврата: " + Рез.КодВозврата);
Сообщить(Рез.СтандартныйВывод);
Если Рез.КодВозврата <> 0 Тогда
    Сообщить(Рез.ОшибочныйВывод);
КонецЕсли;
```

Сигнатура: `ВыполнитьКоманду(Команда, Аргументы = Неопределено, Таймаут = 30, РабочийКаталог = "")`.
Результат — Структура `{КодВозврата, СтандартныйВывод, ОшибочныйВывод, Завершилась}`.

- **Без shell**: команда и `Аргументы` (Массив строк) передаются раздельно — `;`,
  `$(…)`, `|` в аргументах попадают в процесс дословно, не интерпретируются.
- **Таймаут обязателен** (по умолчанию 30 с, потолок 10 мин): по истечении процесс
  убивается, `Завершилась = Ложь`, `КодВозврата = -1`.
- Вывод обрезается до 1 МиБ на поток; каждый запуск пишется в журнал (действие `exec`).
- Если возможность выключена — вызов поднимает ошибку (ловится `Попыткой`).

Включайте только на доверенной/локальной базе.

## Страницы (Pages)

Страница — произвольное представление, которое открывается из меню и формируется
обработчиком на встроенном языке. В отличие от управляемых форм (привязаны к
объекту) и виджетов (фиксированные плитки) — это свободный экран. Описывается
**структурой блоков**, а не строкой HTML: UI рендерит блоки в общую оболочку с
автоэкранированием и i18n.

Файлы:
- `pages/<имя>.yaml` — метаданные;
- `src/<имя>.page.os` — обработчик `Процедура ПриФормировании(Страница, Параметры) Экспорт`.

```yaml
# pages/Панель.yaml
name: Панель
title: Панель руководителя
titles: { en: Manager dashboard }
icon: layout-dashboard
roles: [Руководитель]   # (необяз.) кому видна; пусто — всем
params: [период]        # (необяз.) объявленные параметры строки запроса
```

Открывается по `/ui/page/<Name>`; регистрируется в меню строкой `pages:` в
`contents` подсистемы.

**Построитель `Страница`** (блоки рендерятся в порядке добавления):

| Метод | Блок |
|---|---|
| `.Заголовок(текст)` | заголовок секции |
| `.Абзац(текст)` / `.Текст(текст)` | абзац |
| `.Показатель(подпись, значение[, формат])` | KPI; формат `money`/`number`/`percent` |
| `.Таблица(заголовок)` → `.Колонки(...)`, `.ДобавитьСтроку()` → `.Установить(колонка, значение)`, `.Ссылка(колонка, url)` | таблица |
| `.Список(заголовок)` → `.Пункт(текст[, url])` | список |
| `.График(заголовок, вид)` → `.Категории(...)`, `.Серия(имя, Массив)` | график (ECharts), вид `bar`/`line`/`pie` |
| `.Кнопка(подпись, url)` / `.Ссылка(подпись, url)` | кнопка-ссылка |
| `.КнопкаДействие(подпись, "ИмяПроцедуры")` | кнопка, вызывающая серверную процедуру-действие |
| `.Разделитель()` | разделитель |
| `.ДобавитьСыройHTML(html)` | произвольный HTML (санитизируется; на ответственности автора) |

`СсылкаНаОбъект(...)` возвращает URL карточки `/ui/<вид>/<сущность>/<id>` —
для кликабельных ячеек/пунктов. Две формы: `СсылкаНаОбъект(Ссылка)` (объект-ссылка,
например из `НайтиПоКоду`) и `СсылкаНаОбъект("ИмяСущности", Ид)`. В результатах
запроса колонка `Ссылка` компилируется в `id` и приходит UUID-строкой без типа,
поэтому давайте ей псевдоним и указывайте сущность явно (см. пример).

```bsl
// src/Панель.page.os
Процедура ПриФормировании(Страница, Параметры) Экспорт
    Страница.Заголовок("Сводка");
    Страница.Показатель("Открыл", ИмяПользователя(), "");

    Таблица = Страница.Таблица("Номенклатура");
    Таблица.Колонки("Наименование", "Артикул");
    Запрос = Новый Запрос;
    // Ссылка → id; псевдоним КАК Ид даёт полю имя в результате.
    Запрос.Текст = "ВЫБРАТЬ Ссылка КАК Ид, Наименование, Артикул ИЗ Справочник.Номенклатура";
    Для Каждого Стр Из Запрос.Выполнить() Цикл
        Строка = Таблица.ДобавитьСтроку();
        Строка.Установить("Наименование", Стр.Наименование);
        Строка.Ссылка("Наименование", СсылкаНаОбъект("Номенклатура", Стр.Ид));
        Строка.Установить("Артикул", Стр.Артикул);
    КонецЦикла;
КонецПроцедуры
```

**Кнопки-действия.** `.КнопкаДействие("Пересчитать", "Пересчитать")` рендерится не
ссылкой, а кнопкой, которая отправляет `POST /ui/page/<Name>/action/Пересчитать` и
вызывает серверную процедуру-действие из того же `src/<имя>.page.os`:

```bsl
Процедура Пересчитать(Страница, Параметры) Экспорт
    // побочный эффект: провести, пересчитать, выгрузить…
    Сообщить("Итоги пересчитаны за " + Параметры.Получить("период"));
КонецПроцедуры
```

Резолв процедуры — по имени, как у `ПриФормировании` (Экспорт-процедура из
`.page.os`). После действия страница перерисовывается заново через
`ПриФормировании` (POST-Redirect-GET с сохранением `Параметры`), а `Сообщить(...)`
показывает общий бар сообщений. Текущие параметры строки запроса доезжают и в
действие. PRG исключает повторный запуск действия при обновлении страницы (F5) —
важно для проведения/пересчёта.

Доступ к ролям проверяется как у HTTP-сервисов; `onebase check` проверяет наличие
`ПриФормировании` и компилирует запросы обработчика. Пример: `examples/trade`
(страница «Панель» в разделе «Отчёты»).

**Локализация подписей.** Статические авторские подписи блоков (заголовки, подписи
показателей, заголовки таблиц/списков/графиков и колонок, тексты пунктов и кнопок)
при рендере переводятся через словарь локали (русский-как-ключ). Чтобы подпись
переводилась, добавьте её в `<проект>/locales/<язык>.json` (например
`{"Номенклатура": "Items"}`). Для русской локали (и без словаря) — без изменений.
**Данные не переводятся**: значения показателей, ячейки таблиц, данные графиков и
URL приходят из запросов. Пример: `examples/trade/locales/en.json`.

Динамически собранные подписи (`"Отчёт за " + Период`) ключа целиком не имеют —
для их статической части используйте `НСтр("ru = 'Отчёт за '; en = 'Report for '")
+ Период`. В обработчике страницы (и формы) `НСтр` без явного кода языка берёт язык
текущего пользователя; вне запроса — `"ru"`.

## Виртуальные таблицы регистров накопления

В языке запросов доступны виртуальные таблицы, которые вычисляют агрегированные данные на лету.

### Синтаксис

```
// Остатки на дату (или на текущий момент, если параметр пустой)
ИЗ РегистрНакопления.НазваниеРегистра.Остатки(&НаДату)

// Обороты за период
ИЗ РегистрНакопления.НазваниеРегистра.Обороты(&Начало, &Конец)

// Остатки и обороты совмещённо
ИЗ РегистрНакопления.НазваниеРегистра.ОстаткиИОбороты(&Начало, &Конец)

// Срез последних (для регистров сведений)
ИЗ РегистрСведений.НазваниеРегистра.СрезПоследних(&НаДату)
```

### Колонки результата

**Остатки (`Остатки`):**
| Колонка | Тип | Описание |
|---|---|---|
| измерение (по имени) | — | Разрез учёта |
| `<Ресурс>Остаток` | NUMERIC | Остаток по ресурсу |

**Обороты (`Обороты`):**
| Колонка | Тип | Описание |
|---|---|---|
| измерение (по имени) | — | Разрез учёта |
| `<Ресурс>Приход` | NUMERIC | Сумма приходов |
| `<Ресурс>Расход` | NUMERIC | Сумма расходов |
| `<Ресурс>Оборот` | NUMERIC | Приход − Расход |

### Пример отчёта

```yaml
name: ОстаткиТоваров
query: |
  ВЫБРАТЬ
    Номенклатура,
    КоличествоОстаток,
    СуммаОстаток
  ИЗ РегистрНакопления.ОстаткиТоваров.Остатки(&НаДату)
  ГДЕ КоличествоОстаток > 0
  УПОРЯДОЧИТЬ ПО Номенклатура
```

### Реализация

Виртуальные таблицы реализованы как PostgreSQL-функции (planner functions), созданные в `internal/storage/registers.go`. Транслятор в `internal/query/query.go` разворачивает `РегистрНакопления.X.Остатки(...)` в вызов `рег_x_balances(...)`.

## Автонумерация документов

Документы могут автоматически получать уникальный номер при сохранении.

### YAML

```yaml
name: Заявка
numerator: ЗаявкиПоГодам    # ссылка на нумератор
posting: true
fields:
  - name: Номер
    type: string
  ...
```

Файл нумератора: `numerators/<имя>.yaml`

```yaml
name: ЗаявкиПоГодам
prefix: "ЗАЯ-{YYYY}-"    # {YYYY}, {MM}, {DD}, {HH}, {mm}, {ss}
pad: 5                    # ширина числовой части (00001)
period: year              # сброс счётчика: year | month | day | never
```

### Поведение

- При сохранении документа, у которого `Номер` пустой, платформа автоматически генерирует следующий номер.
- Счётчик хранится в таблице `_numerators` (инкремент через `FOR UPDATE` — атомарно).
- `period: year` — счётчик сбрасывается в январе каждого года.
- Префикс поддерживает маски даты: `ПРИХ-{YYYY}{MM}-` → `ПРИХ-202601-00001`.

### DSL (ручная нумерация)

```
// Явно запросить следующий номер (если не используется автонумерация):
this.Номер = НаименованиеНумератора.СледующийНомер();
```

## Менеджеры справочников и документов в DSL

Глобалы `Справочники` / `Catalogs` и `Документы` / `Documents` дают доступ к
записи, поиску и удалению объектов.

### Создание и запись

```
// Справочник
Зап = Справочники.Контрагент.Создать();
Зап.Наименование = "ООО Ромашка";
Ссылка = Зап.Записать();          // → ссылка на записанную запись

// Документ
Док = Документы.ПоступлениеТоваров.Создать();
Док.Номер = "ПТ-00001";
Стр = Док.Товары.Добавить();
Стр.Номенклатура = Ном; Стр.Количество = 10;
Док.Записать();                    // только запись
Док.Провести();                    // запись + проведение
```

`Записать()` повторяет поведение веб-интерфейса: проставляет номер документа
(автонумерация, если есть строковый реквизит `Номер` и он не задан явно),
затем вызывает обработчик `ПриЗаписи()` (`OnWrite`) до сохранения — так
вычисляются расчётные реквизиты (СуммаНДС, итоги). `Провести()` дополнительно
вызывает `ОбработкаПроведения()` (`OnPost`) и формирует движения регистров.

### Поиск

```
к = Справочники.Контрагент.НайтиПоНаименованию("ООО Ромашка");
т = Справочники.ТипЦен.НайтиПоКоду("РОЗН");
д = Документы.ПоступлениеТоваров.НайтиПоНомеру("ПТ-00001");
// не найдено → возвращается значение, равное Неопределено (nil)
```

### Удаление

```
Ссылка.Удалить();                              // напрямую по ссылке
Документы.ПоступлениеТоваров.Удалить(Ссылка);   // менеджерный вариант
Справочники.Контрагент.Удалить(Ссылка);
```

- Предопределённые элементы и непустые группы удалить нельзя — будет ошибка.
- `Ссылка.ПолучитьОбъект()` возвращает рабочий дескриптор объекта
  (поддерживает `Удалить()`, `Наименование`, `УникальныйИдентификатор`).
- Ссылки из результатов запроса не привязаны к менеджеру — для их удаления
  используйте менеджерный вариант `Документы.Тип.Удалить(Ссылка)`.

### Реквизиты по ссылке — `ЗначениеРеквизитаОбъекта`

Ссылка несёт только UUID и наименование. Чтобы получить любой другой
реквизит объекта, используйте функцию `ЗначениеРеквизитаОбъекта(Ссылка,
"Реквизит")` (английский алиас — `ObjectAttributeValue`):

```
Строка.СтавкаНДС = ЗначениеРеквизитаОбъекта(Строка.Номенклатура, "СтавкаНДС");

// ссылочный реквизит возвращается ссылкой — цепочки работают:
Процент = ЗначениеРеквизитаОбъекта(
    ЗначениеРеквизитаОбъекта(Ном, "СтавкаНДС"), "Процент");
```

- Каждый вызов — один запрос к БД; видит данные открытой DSL-транзакции.
- Пустая ссылка или отсутствующая запись → `nil`.
- Несуществующее имя реквизита → ошибка (ловит опечатки).
- Доступ через точку (`Ссылка.Реквизит`) **не** поддерживается — это
  сознательное решение, как и рекомендация 1С использовать функцию.

## Объект Запрос в DSL

Позволяет выполнять произвольные запросы на языке запросов OneBase прямо из DSL-скриптов.

### Синтаксис

```
Запрос = Новый Запрос;
Запрос.Текст = "ВЫБРАТЬ Наименование, Цена ИЗ Номенклатура ГДЕ Цена > &МинЦена";
Запрос.УстановитьПараметр("МинЦена", 1000);
Результат = Запрос.Выполнить();

Для Каждого Строка Из Результат Цикл
    Сообщить(Строка.Наименование + ": " + Строка(Строка.Цена));
КонецЦикла;
```

### API объекта Запрос

| Метод / свойство | Описание |
|---|---|
| `Запрос.Текст` | Текст запроса |
| `Запрос.УстановитьПараметр(имя, значение)` | Задать значение параметра `&Имя` |
| `Запрос.Выполнить()` | Выполнить → Массив Структур (строк) |

Поля строки результата доступны через точку: `Строка.ИмяПоля`.

### Пример с Попытка/Исключение

```
Попытка
    Результат = Запрос.Выполнить();
Исключение
    Сообщить("Ошибка: " + ОписаниеОшибки());
    Возврат;
КонецПопытки;
```

## Иерархические справочники

Справочник может иметь древовидную структуру — каждый элемент указывает на родителя.

### YAML

```yaml
name: Номенклатура
hierarchical: true       # включить иерархию
fields:
  - name: Наименование
    type: string
  ...
```

### Поведение

- Добавляется системная колонка `parent_id UUID` (ссылка на элемент того же справочника).
- В интерфейсе список отображается как дерево с раскрывающимися узлами.
- Для создания группы — кнопка «Новая группа», для элемента — «Создать».
- Хлебные крошки при открытии элемента показывают путь от корня.
- В запросах иерархия прозрачна: `ВЫБРАТЬ ... ИЗ Справочник.Номенклатура` возвращает все элементы плоским списком.

### DSL

```
// Получить родителя текущего элемента
Родитель = this.Родитель;
```

## Регламентные задания (Scheduled Jobs)

Автоматически выполняемые DSL-процедуры по расписанию cron.

### Файл задания

`scheduled/<имя>.yaml`

```yaml
name: ОбновлениеКурсов
title: Обновление курсов валют
schedule: "0 9 * * 1-5"    # cron: будни в 9:00
processor: ЗагрузкаКурсов   # имя процедуры (src/<имя>.proc.os)
params:
  Источник: "ЦБ РФ"
enabled: true
on_error: log               # log | stop
timeout: 300                # секунд (по умолчанию 300)
```

### Поле `schedule`

Стандартный формат cron (5 полей): `минута час день_месяца месяц день_недели`.

| Пример | Описание |
|---|---|
| `*/1 * * * *` | каждую минуту |
| `0 * * * *` | каждый час |
| `0 9 * * 1-5` | будни 09:00 |
| `0 0 1 * *` | 1-е число каждого месяца в 00:00 |

### Параметры с шаблонами

```yaml
params:
  Дата: "{{today}}"                 # текущая дата
  Начало: "{{today|minus_days:7}}"  # 7 дней назад
  Месяц: "{{today|start_of_month}}" # начало месяца
  Метка: "{{now}}"                  # текущее время с секундами
```

### Управление из UI

**Администрирование → Регламентные задания:**
- Список заданий с расписанием и статусом последнего запуска
- Кнопка «Запустить» — немедленное выполнение вне расписания
- История выполнений (последние 50) с длительностью и выводом

### Таблица истории: `_scheduled_runs`

| Колонка | Тип | Описание |
|---|---|---|
| `id` | UUID | Идентификатор запуска |
| `job_name` | TEXT | Имя задания |
| `started_at` | TIMESTAMPTZ | Время начала |
| `finished_at` | TIMESTAMPTZ | Время завершения |
| `status` | TEXT | `running` / `success` / `error` / `timeout` |
| `output` | TEXT | Весь вывод `Сообщить(...)` |
| `error` | TEXT | Текст ошибки (если статус `error`) |
| `duration_ms` | INTEGER | Длительность в мс |

## Регистр бухгалтерии

Двойная запись (Дт/Кт) для ведения бухгалтерского учёта.

### План счетов

Файл: `accounts/<имя>.yaml`

```yaml
name: Основной
title: Основной план счетов
accounts:
  - code: "10"
    name: Материалы
    kind: active      # active | passive | active_passive
  - code: "60"
    name: Расчёты с поставщиками
    kind: passive
  - code: "90"
    name: Продажи
    kind: active_passive
    children:
      - code: "90.1"
        name: Выручка
        kind: passive
      - code: "90.2"
        name: Себестоимость
        kind: active
```

### Регистр бухгалтерии

Файл: `accountregs/<имя>.yaml`

```yaml
name: БухУчёт
title: Бухгалтерский учёт
accounts: Основной       # ссылка на план счетов
resources:
  - name: Сумма
    type: number
subconto:                # аналитические разрезы на проводках (опционально)
  - name: Контрагент
    type: reference:Контрагент
  - name: Номенклатура
    type: reference:Номенклатура
```

**Субконто** — аналитика на проводках (по аналогии с 1С): по каким разрезам
детализируется остаток/оборот счёта. Порядок объявления задаёт нумерацию
`Субконто1`, `Субконто2`, … Тип субконто — как у обычного поля
(`reference:<Сущность>`, `string`, `enum:<…>`). Каждый субконто хранится в
колонке `субконто<N>` таблицы регистра; имя колонки стабильно при переименовании
поля. `onebase check` проверяет, что `reference`-субконто указывает на
существующую сущность.

### Движения в DSL (`.posting.os`)

```
Процедура ОбработкаПроведения()
  Движения.БухУчёт.Очистить();

  Дв = Движения.БухУчёт.Добавить();
  Дв.СчётДт = "41";              // код счёта (Дт)
  Дв.СчётКт = "60";              // код счёта (Кт)
  Дв.Сумма  = this.СуммаДокумента;
  Дв.Период = this.Дата;

  // Субконто — две взаимозаменяемые формы адресации:
  Дв.Субконто1 = this.Поставщик;            // по номеру (краткая запись)
  Дв.Субконто.Номенклатура = Стр.Товар;     // по имени
КонецПроцедуры
```

Незаданные субконто пишутся как NULL. Присваивание несуществующему субконто
(`Субконто9` при двух объявленных или неизвестное имя `Субконто.Договор`) —
**ошибка проведения**, а не молчаливая потеря данных.

### Виртуальные таблицы в запросах

```
// Остатки на дату
ИЗ РегистрБухгалтерии.БухУчёт.Остатки(&НаДату)

// Обороты за период
ИЗ РегистрБухгалтерии.БухУчёт.Обороты(&Начало, &Конец)

// Разворот по субконто с фильтром по счёту
ВЫБРАТЬ Субконто1 КАК Контрагент, СуммаОстаток
ИЗ РегистрБухгалтерии.БухУчёт.Остатки(&НаДату, Счёт = "60")
```

**Колонки `Остатки`:** `Счёт`, `Наименование`, `Субконто1`…`СубконтоN`, `<Ресурс>Остаток`, `<Ресурс>Дт`, `<Ресурс>Кт`  
**Колонки `Обороты`:** `Счёт`, `Наименование`, `Субконто1`…`СубконтоN`, `<Ресурс>Дт`, `<Ресурс>Кт`

Если у регистра объявлены субконто, остатки/обороты разворачиваются по ним
(GROUP BY включает колонки субконто). В фильтре виртуальной таблицы доступны имена
`Счёт`, `Субконто1`…`СубконтоN`, а также субконто по имени поля.

### Схема таблицы `акк_<имя>`

| Колонка | Тип | Описание |
|---|---|---|
| `id` | UUID | Идентификатор движения |
| `period` | TIMESTAMPTZ | Дата движения |
| `регистратор` | UUID | ID документа-регистратора |
| `регистратор_тип` | TEXT | Тип документа |
| `счётдт` | TEXT | Код счёта Дт |
| `счёткт` | TEXT | Код счёта Кт |
| ресурсы (по имени) | NUMERIC | Суммы ресурсов |
| `субконто1`…`субконтоN` | по типу субконто | Аналитические разрезы (UUID для `reference`) |

Счета хранятся в таблице `_accounts (plan TEXT, code TEXT, name TEXT, kind TEXT, parent TEXT)`.

## Демо-режим

Демо-режим предназначен для публичного показа базы: жёлтый баннер предупреждает
пользователей, что данные временные, а автосброс по расписанию возвращает базу
к эталонному состоянию.

### Настройка `config/app.yaml`

```yaml
demo:
  enabled: true
  reset_backup: backups/demo_initial.obz   # путь относительно папки проекта
  reset_schedule: "0 2 * * *"              # cron: каждую ночь в 02:00
  message: "Данные сбрасываются каждую ночь в 02:00"
```

| Поле | По умолчанию | Описание |
|---|---|---|
| `enabled` | `false` | Включить демо-режим |
| `reset_backup` | — | Путь к `.obz`-бэкапу эталонного состояния |
| `reset_schedule` | `"0 2 * * *"` | cron-расписание сброса |
| `message` | `"Данные сбрасываются каждую ночь в 02:00"` | Текст в баннере |

Если `reset_backup` не указан — сброс регистрируется в расписании, но ничего не делает
(баннер всё равно отображается).

### Подготовка эталонного бэкапа

1. Запустить базу, наполнить демо-данными.
2. В конфигураторе: **Бэкап → Создать полный бэкап** (формат `.obz`).
3. Скопировать файл в папку проекта, например `backups/demo_initial.obz`.
4. Прописать путь в `config/app.yaml`.

### Роль «Демо»

Создайте файл `roles/демо.yaml` — роль без права удаления:

```yaml
name: Демо
description: Демо-доступ — чтение и создание без удаления
permissions:
  catalogs:
    Контрагент: [read, write]
    Номенклатура: [read, write]
  documents:
    РеализацияТоваров: [read, write, post, unpost]
    ПоступлениеТоваров: [read, write, post, unpost]
  registers:
    ОстаткиТоваров: [read]
  reports:
    ОстаткиТоваров: [run]
```

### Создание демо-пользователя

1. **Система → Пользователи → + Добавить**: логин `demo`, без флага «Администратор».
2. Назначить роль **Демо** через кнопку **Роли**.
3. Нажать 🔒 (кнопка-замок) — запретить смене пароля.

### Что сбрасывает автосброс

При срабатывании задания `DemoReset` (`backup.DemoReset`):

| Что | Действие |
|---|---|
| Таблицы справочников, документов, регистров | Очищаются и восстанавливаются из `.obz` |
| `_constants` (константы) | Восстанавливаются из `.obz` |
| `_attachments` | Восстанавливаются из `.obz` |
| `_users`, `_sessions` | **Не трогаются** — пользователи и сессии сохраняются |
| `_roles`, `_user_roles` | **Не трогаются** — права сохраняются |
| `_scheduled_runs` | **Не трогается** — история заданий сохраняется |

### Что видит демо-пользователь

| Возможность | Демо-роль |
|---|---|
| Просмотр данных | ✅ |
| Создание и редактирование записей | ✅ |
| Проведение документов | ✅ |
| Удаление записей | ❌ (нет `delete` в роли) |
| Смена пароля | ❌ (флаг 🔒 на пользователе) |
| Регламентные задания | ❌ (только администратор) |
| Консоль запросов и кода | ❌ (только администратор) |

### Баннер в интерфейсе

При `demo.enabled: true` в верхней части каждой страницы отображается:

```
⚠️ ДЕМО-РЕЖИМ — Данные сбрасываются каждую ночь в 02:00
```

запусков. Можно запустить вручную кнопкой «Запустить».

## Поиск и пагинация в списках

Параметры запроса для `GET /ui/catalog/<имя>` и `GET /ui/documents/<имя>`:

| Параметр | Тип | Описание |
|---|---|---|
| `q` | string | Полнотекстовый поиск: ILIKE `%q%` по всем полям типа `string` |
| `filter[Поле]` | string | Точный фильтр по полю (по умолчанию `=`, для строк — ILIKE) |
| `filter[Поле_from]` | date/number | Начало диапазона |
| `filter[Поле_to]` | date/number | Конец диапазона |
| `sort` | string | Имя поля для сортировки |
| `dir` | `asc` / `desc` | Направление сортировки |
| `page` | int | Номер страницы (начиная с 1) |
| `limit` | int | Размер страницы (1–1000, по умолчанию 100) |

**Ответ шаблона** получает поля: `Total int`, `Page int`, `TotalPages int`, `HasPrev bool`, `HasNext bool`, `PrevPage int`, `NextPage int`.

**Go API (`internal/storage/crud.go`):**

```go
type ListParams struct {
    Filters   map[string]FilterValue
    Sort      string
    Dir       string       // "asc" | "desc"
    ParentStr string       // для иерархических справочников
    Search    string       // ILIKE по всем string-полям
    Limit     int          // 0 = без ограничения
    Offset    int          // для пагинации
}

// List() возвращает строки с применением всех условий
rows, err := db.List(ctx, "Номенклатура", entity, params)

// CountList() возвращает COUNT(*) без LIMIT/OFFSET
total, err := db.CountList(ctx, "Номенклатура", entity, params)
```

## Планы обмена (data exchange, план 86)

Синхронизация данных между базами OneBase файловыми пакетами. Объект
`exchange/<имя>.yaml`:

```yaml
# exchange/ФилиалыЦентр.yaml
name: ФилиалыЦентр
title: Филиалы ↔ Центр
conflict: by_time            # by_time | by_node_priority | hook
content:                     # состав обмена: <Вид>.<Имя>
  - Справочник.Номенклатура
  - Справочник.Контрагент
  - Документ.РеализацияТоваров
nodes:                       # участники; priority — для by_node_priority
  - { code: center, name: Центральная, priority: 10 }
  - { code: fil01,  name: Филиал №1,   priority: 1 }
```

- **Код текущего узла** не хранится в YAML (конфигурация у всех участников
  одинаковая) — задаётся на каждой базе: `onebase exchange init --plan X --node center`.
- **Регистрация изменений.** Запись/проведение/пометка удаления объекта из
  состава автоматически ставит его в очередь выгрузки всем узлам, кроме текущего
  (внутри транзакции записи). Опорой служит колонка `_version`.
- **Выгрузка.** `onebase exchange dump --plan X --to fil01 --out pkg.obx` собирает
  незапподтверждённые изменения в версионированный JSON-пакет `.obx`.
- **Загрузка.** `onebase exchange load --in pkg.obx` идемпотентна: объект
  применяется, если его нет или входящая версия строго новее (документы приходят
  **непроведёнными** — движения на приёмник не переносятся). Повторная доставка
  того же пакета ничего не меняет.
- **Конфликт** (встречная правка одного объекта) разрешается правилом `conflict`:
  `by_time` — победил позже изменённый; `by_node_priority` — узел с большим
  `priority`; `hook` — процедура `ПриКонфликтеОбмена(Локальный, Входящий) Экспорт`
  в общем модуле (`src/*.module.os`) возвращает Истину, если побеждает входящее.
  Хук работает на Go-путях приёма (онлайн push, CLI load/sync); в DSL-пути
  `ЗагрузитьПакет` правило hook откатывается к `by_time`. `onebase check`
  требует наличие процедуры при `conflict: hook`.
- **Из DSL:** `Пакет = ПланыОбмена.ФилиалыЦентр.ВыгрузитьИзменения("fil01");` и
  `ПланыОбмена.ФилиалыЦентр.ЗагрузитьПакет(Пакет);` (не оборачивать в
  `Транзакцию()` — операции открывают свою).

- **Онлайн-обмен по сети.** Вместо файла — прямой обмен по HTTP. У узлов задайте
  `url` (адрес базы, `${env:VAR}` поддерживается) и общий токен плана
  (`onebase exchange init --token <секрет>`, одинаковый на всех участниках). Сервер
  каждой базы публикует приёмные эндпоинты `POST /exchange/<план>/push` и
  `GET /exchange/<план>/pull?to=<узел>` (Bearer-токен плана; без токена 403, неверный
  401). Команда `onebase exchange sync --plan X --with <узел>` отправляет свои
  изменения партнёру и забирает его изменения для себя (pull несёт подтверждение —
  очередь дренируется). Подтверждение приёма (ack) пиггибэком очищает очередь и в
  файловом цикле (обратный пакет подтверждает прямой).

Служебные таблицы: `_exchange_changes` (очередь), `_exchange_peers` (счётчики
сообщений/подтверждений). `onebase check` валидирует состав, узлы и правило.

## Новые CLI-команды (справочно)

| Команда | Описание |
|---|---|
| `onebase exchange init --project <dir> --sqlite <f> --plan <имя> --node <код> [--token <секрет>]` | Задать код текущего узла базы (и, опц., токен онлайн-обмена) |
| `onebase exchange dump --plan <имя> --to <узел> --out <pkg.obx>` | Выгрузить изменения для узла в файл пакета |
| `onebase exchange load --in <pkg.obx>` | Загрузить пакет обмена в базу (идемпотентно) |
| `onebase exchange sync --plan <имя> --with <узел>` | Онлайн-обмен с узлом по HTTP (push+pull) |
| `onebase exchange status --plan <имя>` | Очередь и счётчики сообщений по узлам |
| `onebase deploy --project <dir> --db <dsn> --message <text>` | Накатить файловую конфигурацию в БД-режим, применить миграции и записать версию релиза |

| `onebase init --template <name> <dir>` | Создать проект из шаблона |

## CLI-контракт для ИИ-инструментов

Для внешних агентов, IDE-интеграций и отладки генерации используйте связку:

```bash
onebase ai-guide --project <dir>          # человеко- и agent-readable справочник
onebase describe --project <dir>          # полный JSON-контракт конфигурации
onebase examples --list                   # доступные canonical snippets
onebase examples document                 # пример YAML документа
onebase check --project <dir>             # validation gate после правок
```

`onebase describe` возвращает `schemaVersion: 2` и рассчитан на машинное потребление.
В контракт входят:

- справочники, документы, регистры, регистры сведений, планы счетов и регистры бухгалтерии;
- отчёты, обработки, виджеты, журналы, подсистемы, страницы, HTTP-сервисы, регламентные задания;
- RBAC-роли из `roles/*.yaml`;
- формы, события форм, параметры, табличные части и source `{file,line}`;
- модули DSL и процедуры с параметрами, `export`-флагом и source location;
- `builtins` и полный `language` из `internal/dsl/langref`.

`onebase examples [kind]` нужен как низкорисковая подсказка формата. Поддерживаемые `kind`:
`catalog`, `document`, `register`, `inforeg`, `enum`, `processor`, `report`, `widget`,
`form`, `page`, `service`, `role`, `query`, `posting`. Русские алиасы вроде
`справочник`, `документ`, `запрос`, `роль` тоже поддерживаются.
