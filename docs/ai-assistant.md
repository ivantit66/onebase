# ИИ-помощник (LLM) в конфигурации

OneBase умеет обращаться к большим языковым моделям прямо из DSL. Это позволяет
встраивать в конфигурацию анализ данных (что закупить, ABC-анализ, объяснение
отчётов), распознавание документов и другие сценарии. См. план `Plans/48`.

## Настройка

Конфигуратор → меню → **ИИ-помощник**. Настройки задаются формой из трёх частей:

- **endpoints** — провайдеры и ключи (`kind`: `gemini` | `anthropic` | `openai` |
  `compatible`; `base_url` для z.ai/локальных моделей; `api_key`).
- **models** — модели с привязкой к endpoint и флагом `vision` для мультимодальных.
- **profiles** — маршрутизация: задача → упорядоченная цепочка моделей. При
  лимитах/временных ошибках движок переключается на следующую модель цепочки.

При сохранении форма пишет тот же структурный конфиг в `_settings`. Эквивалентный JSON
для Gemini в документах и GLM через z.ai в текстовых задачах:

```json
{
  "enabled": true,
  "endpoints": [
    {"name": "google", "kind": "gemini", "api_key": "..."},
    {"name": "z_ai", "kind": "anthropic", "base_url": "https://api.z.ai/api/anthropic", "api_key": "..."}
  ],
  "models": [
    {"name": "gemini-2.5-flash", "endpoint": "google", "vision": true},
    {"name": "gemini-2.0-flash", "endpoint": "google", "vision": true},
    {"name": "glm-4.6", "endpoint": "z_ai"}
  ],
  "profiles": [
    {"task": "документы", "models": ["gemini-2.5-flash", "gemini-2.0-flash"]},
    {"task": "анализ", "models": ["glm-4.6"]}
  ],
  "default_profile": "анализ"
}
```

Кнопка **Проверить** делает пробный запрос по выбранной задаче — проверяет ключи и
маршрут до сохранения. Ключи хранятся в служебной таблице базы (`_settings`) и не
попадают в экспорт конфигурации.

## Конфиг через app.yaml (деплой и демо)

Так как ключи лежат в `_settings` и **не входят в `.obz`**, при деплое демо или
прод-базы из бэкапа ИИ-помощник окажется ненастроенным. Чтобы донести конфиг вместе
с конфигурацией (без утечки ключа в выгрузку), задайте секцию `llm` в
`config/app.yaml`, а ключи — через `${env:ПЕРЕМЕННАЯ}`:

```yaml
name: МояБаза
llm:
  enabled: true
  endpoints:
    - {name: z_ai, kind: anthropic, base_url: https://api.z.ai/api/anthropic, api_key: "${env:ZAI_KEY}"}
    - {name: google, kind: gemini, api_key: "${env:GEMINI_KEY}"}
  models:
    - {name: glm-4.6, endpoint: z_ai}
    - {name: gemini-2.5-flash, endpoint: google, vision: true}
  profiles:
    - {task: анализ, models: [glm-4.6]}
    - {task: чат, models: [glm-4.6]}
    - {task: документы, models: [gemini-2.5-flash]}
  default_profile: анализ
ai:
  data_scope: rbac       # admin_only | rbac | all
  daily_token_cap: 50000 # 0 или отсутствие = без лимита
```

При старте сервера (`onebase run`) этот конфиг применяется к базе, а `${env:...}`
подставляется из окружения. **Когда секция `llm` задана в `app.yaml`, она
авторитетна** — перезаписывает `_settings` при каждом запуске, поэтому правьте конфиг
в файле, а не через UI. Ключи (`ZAI_KEY`, `GEMINI_KEY`) держите в переменных окружения
сервиса/контейнера — они не попадут ни в git, ни в `.obz`.

Блок `ai:` задаёт non-secret политику доступа при деплое. `data_scope: rbac`
разрешает инструменты данных только пользователям с флагом «Доступ ИИ-чата к
данным» или ролям с `permissions.ai_data_access: true`; сами источники запроса
проверяются по их `read`-правам. Если секций `llm`/`ai` нет — поведение прежнее:
настройка через конфигуратор, хранение только в `_settings`.
При выгрузке универсального `.obz` безопасные ключи `ai.data_scope` и
`ai.daily_token_cap` переносятся вместе с базой; `llm.config` и API-ключи не
экспортируются.

## Чат в пользовательском режиме: данные и действия

Плавающий чат 🤖 доступен на любой странице базы (кнопка появляется, когда
помощник настроен). Пользователям с доступом к данным (см. `data_scope` выше)
модель отвечает по фактическим данным базы через инструменты `описание_данных`
и `выполнить_запрос`, а также умеет **готовить действия**:

- **создать документ** или **элемент справочника** — черновик по вашим словам
  («заведи заказ для Ромашки: 2 стола, 4 стула»). Ссылочные поля ИИ подбирает
  по наименованию или находит запросом; запись происходит **только после
  нажатия кнопки «Создать»** в карточке подтверждения, документ **не
  проводится**. Права и строковый доступ проверяются как при ручном вводе.
- **открыть форму** — кнопка перехода к списку, конкретной записи, отчёту или
  обработке («открой список заказов»).

Окно чата можно растянуть за левую кромку (двойной клик по ней — вернуть
ширину по умолчанию); выбранная ширина запоминается в браузере.

Создание объектов через чат журналируется в аудите ИИ (`чат-создание`).

## Функции DSL

| Функция | Назначение |
|---|---|
| `ЗапросИИ(Промпт)` / `ЗапросИИ(Промпт, Параметры)` | Текстовый запрос, возвращает строку. |
| `ЗапросИИДжейсон(Промпт)` / `(Промпт, Параметры)` | То же, но просит модель вернуть строгий JSON. |
| `РаспознатьДокумент(Путь, Промпт)` | Vision-запрос: распознаёт изображение/PDF с диска (профиль `документы`). |
| `РаспознатьИзображение(ДанныеBase64, ТипMIME, Промпт)` | То же, но из данных в памяти (загруженный файл/вложение), без записи на диск. |

`Параметры` — `Структура` с необязательными полями:

- `Задача` — профиль маршрутизации (по умолчанию `анализ`).
- `Система` — системная инструкция (роль модели).
- `Формат` — `"json"`, чтобы запросить JSON-ответ.
- `Температура` — число.

При выключенном/ненастроенном помощнике функции выдают понятную ошибку (можно
поймать через `Попытка … Исключение`).

## Пример: рекомендация к закупке

```bsl
Процедура Выполнить()
  // 1. Готовим данные штатным запросом (остатки + продажи за период).
  Запрос = Новый Запрос;
  Запрос.Текст =
    "ВЫБРАТЬ Номенклатура, КоличествоОстаток
    |ИЗ РегистрНакопления.ТоварноеДвижение.Остатки(&НаДату)";
  Запрос.УстановитьПараметр("НаДату", ТекущаяДата());
  Данные = Запрос.Выполнить().Выгрузить();

  // 2. Просим ИИ проанализировать и предложить закупку.
  Промпт = "Проанализируй остатки и продажи, предложи что и сколько закупить
           |с кратким обоснованием. Данные: " + ВJSON(Данные);
  П = Новый Структура("Задача, Система", "анализ", "Ты опытный товаровед.");
  Ответ = ЗапросИИ(Промпт, П);

  Сообщить(Ответ);
КонецПроцедуры
```

## Пример: умная загрузка документа (vision → черновик)

Распознавание скана/фото накладной и создание черновика поступления. Профиль
`документы` (по умолчанию `gemini-2.5-flash` с фолбэком) поддерживает изображения.

```bsl
Процедура РаспознатьНакладную(ДанныеФайлаBase64)
  Промпт = "Извлеки из накладной поставщика и строки (наименование, количество, цена).
           |Верни строго JSON: {""Поставщик"":"""", ""Строки"":[{""Наименование"":"""",""Количество"":0,""Цена"":0}]}";
  ОтветJSON = РаспознатьИзображение(ДанныеФайлаBase64, "image/jpeg", Промпт);
  Данные = ПарсJSON(ОтветJSON);

  // Создаём черновик документа (без проведения) — человек проверяет и проводит сам.
  Док = Документы.Поступление.Создать();
  Док.Поставщик = НайтиКонтрагента(Данные.Поставщик);
  Для Каждого Стр Из Данные.Строки Цикл
    Нс = Док.Товары.Добавить();
    Нс.Номенклатура = ПодобратьНоменклатуру(Стр.Наименование); // fuzzy-match по справочнику
    Нс.Количество   = Стр.Количество;
    Нс.Цена         = Стр.Цена;
  КонецЦикла;
  Док.Записать(); // черновик; проведение — вручную после проверки
КонецПроцедуры
```

Подбор номенклатуры (`ПодобратьНоменклатуру`) и контрагента — обычный `Запрос` по
справочникам; при неоднозначности оставляйте поле пустым для ручного выбора.
