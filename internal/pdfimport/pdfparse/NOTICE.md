# pdfparse — вендоренный форк PDF-парсера

Этот пакет — копия `github.com/dslipak/pdf` v0.0.2 (форк `rsc.io/pdf`),
вендоренная в OneBase для нужд импорта макетов печатных форм из PDF (план 64,
этап 6). Лицензия исходного кода — BSD-3-Clause (см. `LICENSE`,
Copyright (c) 2009 The Go Authors).

## Зачем вендоринг

Публичного API хватило бы для извлечения текста, но **извлечение кириллицы из
PDF, сгенерированных fpdf** (наш серверный рендер и синтетические фикстуры),
требует патча самого CMap-декодера. Поэтому пакет скопирован и пропатчен.

## Внесённые изменения относительно upstream

1. **Имя пакета** `pdf` → `pdfparse` (избегаем коллизии имён и подчёркиваем форк).

2. **Исправление декодирования широких identity-bfrange ToUnicode CMap**
   (`page.go`, метод `cmap.Decode`). fpdf пишет вырожденный ToUnicode вида:

   ```
   1 beginbfrange
   <0000> <FFFF> <0000>
   endbfrange
   ```

   Исходный rsc.io/pdf при масштабировании назначения инкрементировал только
   **последний байт** строки-назначения, из-за чего код `041D` («Н», U+041D)
   декодировался в U+001D — терялся старший байт. Добавлены помощники
   `offsetBE`/`addRangeOffsetBE`: смещение `(code - lo)` прибавляется ко всем
   байтам назначения как к big-endian целому с переносом. 1С-PDF используют
   явные `bfchar` и работали и без патча; патч чинит именно широкие диапазоны.

3. **Устранение мёртвых вызовов `fmt.Sprint(...)`/`fmt.Sprintf(...)`**, чей
   результат отбрасывался (`lex.go`, `read.go`) — `go vet` помечал их
   `unusedresult`. Поведение не меняется (в upstream это уже no-op: `errorf`
   лишь форматирует строку, panic закомментирован).

Тесты патча — в `cmap_offset_test.go`.
