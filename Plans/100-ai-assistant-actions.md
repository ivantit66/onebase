# Этап 100 — Кликабельные действия в ответах ИИ-ассистента

**Статус:** 🔜 Предложено (проектный набросок)

> Источник: пользователь заметил, что ассистент в ответе советует действия
> («оформить документ **Заказ поставщику**», «обработка **План закупок**,
> раздел *Закупки*»), но это просто текст — по нему некуда кликнуть. После
> markdown-рендера ответов (этап markdown, PR к нему) ссылки в ответе уже
> становятся кликабельными; не хватает, чтобы ассистент их **отдавал**.

## Контекст

Пользовательский ИИ-ассистент (`/ui/ai/chat`, `internal/ui/ai_chat.go`) отвечает
свободным текстом. Модель уже получает срез конфигурации (сущности, отчёты,
обработки — `aiassist`/`projectSchemaText`), то есть **знает имена объектов**, о
которых говорит. Инфраструктура открытия уже есть:

- markdown-рендер (соседний PR) превращает `[текст](/ui/…)` в ссылку и
  санитайзит `href` (только `http(s)`/относительные);
- вкладочная оболочка `obOpenInShell` (`ui.js`) перехватывает клики по `/ui/…`
  и открывает форму/отчёт/обработку **новой вкладкой** рядом (issue #129/#130).

Значит для «кликабельных действий» достаточно, чтобы ассистент выдавал
**канонические `/ui/`-ссылки** на объекты, которые он упоминает.

## Канонические URL

| Объект | URL |
|---|---|
| Новый документ | `/ui/document/<Документ>/new` |
| Список сущности | `/ui/<сущность>` |
| Отчёт | `/ui/report/<Отчёт>` |
| Обработка | `/ui/processor/<Обработка>` |
| Страница | `/ui/page/<Страница>` |
| Журнал | `/ui/journal/<журнал-в-нижнем-регистре>` |
| Карточка записи | `/ui/<сущность>/<Вид>/<id>` (как в виджетах) |

## Варианты реализации

### A. Модель отдаёт ссылки (рекомендуется как основа)
В системный промпт ассистента добавить правило: *когда советуешь действие над
объектом из конфигурации — оформляй его markdown-ссылкой на канонический URL*
(таблица выше), а не просто упоминай имя. Модель уже видит список объектов в
контексте, а рендер+оболочка доводят клик до открытия вкладки.

- Плюсы: почти нулевая цена, переиспользует markdown-рендер и `obOpenInShell`;
  модель сама выбирает, что кликабельно.
- Минусы: недетерминированно (модель иногда забудет ссылку); имя объекта в URL
  должно быть точным — подсказать модели «бери имя из контекста дословно».

### B. Пост-линковка известных имён (дополнение)
Сервер (или рендер) пост-обрабатывает ответ: находит **точные** вхождения имён
объектов из конфигурации и оборачивает в ссылку. Только точное совпадение по
словарю имён (без фаззи) — иначе ложные срабатывания.

- Плюсы: детерминированно, работает даже когда модель не поставила ссылку.
- Минусы: конфликты имён (документ и справочник с похожим именем), риск линковать
  имя в неподходящем контексте; нужна аккуратная токенизация.

### C. Структурированные действия (крупная форма, опционально)
Ответ ассистента возвращает отдельный список `actions: [{label, url}]`, который
UI рисует **кнопками** под сообщением. Требует, чтобы модель вызывала инструмент
`предложить_действие(label, тип, объект)` (сервер валидирует объект и строит
безопасный URL). Самое надёжное и «богатое», но самый большой объём.

**Рекомендация:** A как база (дёшево, сразу работает поверх markdown-рендера) +
B точечной пост-линковкой для самых частых объектов; C — если понадобится ряд
кнопок-действий с подтверждением.

## Безопасность

- Ссылки только на **свой** `/ui/…` (относительные, same-origin) — markdown-рендер
  уже режет `javascript:`/внешние схемы; для ответов ассистента дополнительно
  допускать только префикс `/ui/`.
- Действия по ссылке — это GET-навигация на форму (открытие), а не выполнение;
  запись/проведение остаются за явными кнопками формы. Никаких side-effect-URL.
- Имя объекта в URL валидировать по словарю конфигурации (вариант B/C), чтобы
  ссылка не вела на несуществующий/чужой объект.

## Изменения в коде (ориентир)

- **Промпт (A):** `internal/aiassist` / `internal/ui/ai_chat.go` — правило про
  канонические ссылки + перечень шаблонов URL в системном промпте (у ассистента
  уже есть список объектов в контексте).
- **Пост-линковка (B):** хелпер, строящий словарь `имя → URL` из
  `proj.Entities/Reports/Processors/Pages/Journals`, и обёртка ответа перед
  возвратом (или в markdown-рендере на клиенте по переданному словарю).
- **Действия-кнопки (C):** инструмент `предложить_действие` в `ai_tools.go` +
  поле `actions` в ответе `/ui/ai/chat` + рендер кнопок в `ui.js`.

## Тесты

- Пост-линковка: имя объекта → корректный `/ui/…`, отсутствие ложных
  срабатываний на подстроках, конфликт имён разрешается детерминированно.
- Санитайзинг: ответ с `javascript:`/внешним URL не даёт кликабельной ссылки.
- (C) Инструмент строит URL только для существующих объектов; несуществующий →
  отказ.

## Вне объёма

- Выполнение действий (запись/проведение) прямо из чата — намеренно нет
  (только навигация-открытие).
- Фаззи-сопоставление имён — только точные совпадения по словарю.
