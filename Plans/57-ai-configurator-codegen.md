## Контекст

Инфраструктура уже есть: `internal/llm` (мультипровайдер, фолбэк, tool-use), `onebase
ai-guide` (генерит `AGENTS.md` с сигнатурами builtin'ов), понятный декларативный формат
(YAML-метаданные + DSL `.os`), и валидатор `onebase check`, который **компилирует и
исполняет** запросы модулей. Это идеальная почва, чтобы снять главный барьер входа:
**«опиши задачу по-русски → получи рабочий каркас конфигурации»**.

Три пользовательских сценария:
1. **Генерация каркаса** — «нужен учёт заявок с товарами и статусами» → справочники +
   документ с ТЧ + регистр + форма.
2. **Объяснение ошибок** — вывод `onebase check` человеческим языком + предложенный фикс.
3. **Подсказка запроса** — «средний чек по менеджерам за месяц» → текст запроса OneBase.

> **Предусловие безопасности:** сгенерированный DSL/конфигурацию нельзя исполнять «вслепую».
> Этап 0 (песочница) — обязательная база; без неё фича опасна (особенно с marketplace, план 60).

## Этап 0 — Песочница DSL (фундамент)

Изолированное исполнение недоверенного DSL (сгенерированного ИИ или установленного из
реестра). Часть кирпичей уже есть:
- файловый sandbox (`interpreter.SetFileSandbox`, `filesandbox.go`) — ограничение FS;
- лимиты рекурсии/итераций (`limits.go`, `maxWhileIter`), wall-clock timeout (follow-up плана 47).

Достроить **профиль ограничений** на запуск (а не глобально — связь с планом 52, `execCtx`):
- запрет/whitelist сетевых builtin'ов (HTTP-клиент) и email;
- CPU/таймаут и потолок памяти коллекций на запуск;
- запрет файловых операций вне песочницы (уже есть, привязать к профилю).

```go
// internal/dsl/interpreter — профиль на запуск (живёт в execCtx из плана 52)
type SandboxProfile struct {
    AllowNet, AllowFile, AllowEmail bool
    MaxWallClock time.Duration
    MaxLoopIters int
}
```

## Этап 1 — `describe`-контекст для модели

Чтобы модель не выдумывала, ей нужен срез текущей конфигурации. Переиспользовать
`onebase describe` (вся структура в JSON) + `ai-guide` (builtins). Сжать в компактный
системный контекст: список объектов с полями (как `aiSchemaText` в `ai_tools.go:77`, но
для конфигуратора и шире — ТЧ, проводки, формы).

| Файл | Изменение |
|---|---|
| `internal/llm` / `internal/aiassist` | сборка контекста: `describe` + builtins → промпт |
| `internal/cli/aiguide.go` | переиспользовать генерацию справки builtins |

## Этап 2 — Генерация каркаса (tool-use с записью под подтверждение)

Расширить tool-use инструментами **записи в конфигурацию** — но только в **песочницу-черновик**,
с обязательным предпросмотром и подтверждением пользователя перед применением.

Инструменты модели (исполняются в конфигураторе, под admin):
- `создать_объект(тип, имя, YAML)` — кладёт черновик в staging, не в рабочую конфигурацию;
- `проверить_конфигурацию()` — запускает `onebase check` на staging, возвращает ошибки модели
  (она сама исправляет — цикл, ограниченный `MaxToolIterations`);
- `показать_объект(имя)` — чтение существующего для контекста.

Поток: запрос пользователя → модель генерит черновики → авто-`check` в песочнице →
самокоррекция → **показ diff пользователю** → «Применить»/«Отклонить». Применение — через
существующий путь записи конфигурации (`configdb`/файлы).

| Файл | Изменение |
|---|---|
| `internal/launcher/ai_*.go` | инструменты staging-записи + предпросмотр diff + apply |
| `internal/configcheck` | вызов `check` на staging-наборе (in-memory), структурированный результат |
| `internal/launcher/configurator_tmpl.go` (после плана 55) | UI: чат генерации + diff-вью + кнопки apply/reject |

## Этап 3 — Объяснение ошибок `check` и подсказка запросов

- **Объяснение `check`:** прогнать вывод `configcheck` через профиль `конфигуратор`
  (`llm`), вернуть человеческое объяснение + предложенный фикс (как дополнительная
  кнопка рядом с ошибкой).
- **Подсказка запроса:** инструмент «построй запрос по описанию» поверх `describe`-контекста
  (этап 1) + конструктора запросов (уже есть). Результат — текст в консоль запросов.

## Тесты

- `sandbox_profile_test.go` — запрещённые builtin'ы (net/file/email) блокируются по профилю;
  таймаут/лимит итераций срабатывают.
- генерация: на фиксированном промпте модель-заглушка (фейковый `llm.Runner`) возвращает
  YAML → `check` на staging проходит → diff формируется; ошибочный YAML → цикл коррекции.
- объяснение ошибок: вывод `check` → непустой человекочитаемый текст (с фейковым LLM).
- **применение только после подтверждения** — без apply рабочая конфигурация не меняется.

## Verification

1. В конфигураторе: «нужен справочник Клиенты и документ Заявка с товарами» → черновики →
   авто-`check` зелёный → diff → «Применить» → объекты появились, база стартует.
2. Сломать YAML вручную, нажать «Объясни ошибку» → понятное объяснение + фикс.
3. «средний чек по менеджерам за месяц» → валидный запрос, выполняется в консоли.
4. Сгенерированный код с сетевым вызовом в песочнице-черновике — заблокирован профилем.
