# План 66 — Страницы (произвольные представления на DSL)

Произвольная страница — самостоятельное представление, которое автор конфигурации
**программирует** на встроенном языке, когда готовых виджетов/форм не хватает.

- Метаданные: `pages/<имя>.yaml` (`name`, `title`/`titles`, `icon`, `roles`, `params`).
- Обработчик: `src/<имя>.page.os`, `Процедура ПриФормировании(Страница, Параметры) Экспорт`.
- Маршрут: `/ui/page/<имя>` (`internal/ui/handlers_page.go`).
- В меню: через `contents.pages: [...]` в `subsystems/<раздел>.yaml`.

## Страницы vs панели подсистем

Легко спутать: и то, и другое — «сводный экран». Разница — **декларативный
дашборд из готовых виджетов** против **произвольного экрана, собранного кодом**.

| | Панель подсистемы (`home_page`) | Страница (план 66) |
|---|---|---|
| Как описывается | Декларативно, YAML | Императивно, DSL-код |
| Из чего собрана | Виджеты `widgets/*.yaml` (отдельные переиспользуемые объекты) | Блоки построителя `Страница` (живут внутри процедуры) |
| Логика | один `query` на виджет, без кода | циклы, условия, несколько запросов, любые вычисления |
| Параметры | нет | да (`?param=…` → структура `Параметры`) |
| Где живёт | блок `home_page` внутри `subsystems/X.yaml` | `pages/X.yaml` + `src/X.page.os` |
| Открытие | автоматически при выборе подсистемы | по URL `/ui/page/X` (опц. в меню через `contents.pages`) |
| Сколько | одна на подсистему | сколько угодно, не привязаны к разделам |

Аналогия:

- **Панель подсистемы** — «собрать дашборд из готовых плиток»: быстро, но
  ограничено типами виджетов (`kpi`/`chart`/`actions`/`list`). Для обзорного
  экрана раздела из типовых KPI / графиков / списков.
- **Страница** — «написать свой мини-отчёт/форму кодом»: когда нужна нестандартная
  таблица со ссылками на объекты, своя логика выборки, параметры, сложная
  компоновка.

Тот же дашборд-движок рисует и глобальный рабочий стол (`config/home_page.yaml`),
и панели подсистем — это один механизм на разных уровнях. Страница стоит особняком:
отдельный объект навигации со своим адресом.

## Построитель «Страница» (блоки)

`internal/dsl/interpreter/page_builtins.go` — объект `Страница` наполняется блоками:

- `Заголовок(текст)`, `Абзац`/`Текст(текст)`, `Разделитель()`.
- `Показатель(подпись, значение, формат)` — формат `money`/`percent`/`number`.
- `Кнопка`/`Ссылка(текст, url)`.
- `Таблица(заголовок)` → `Колонки(...)`, `ДобавитьСтроку()` → `Установить(колонка, значение)`, `Ссылка(колонка, url)`.
- `Список(заголовок)` → `Пункт(текст, url)`.
- `График(заголовок, вид)` → `Категории(...)`, `Серия(имя, массив)` — рендер через ECharts, как виджеты.
- `ДобавитьСыройHTML(html)` — escape-hatch, санитизируется (`sanitizePageHTML`), «на ответственности автора».

`СсылкаНаОбъект(...)` (`internal/dslvars/dslvars.go`) — URL карточки
`/ui/<вид>/<сущность>/<id>`, две формы:

- `СсылкаНаОбъект(Ссылка)` — вид/сущность берутся из объекта-ссылки (например из `НайтиПоКоду`);
- `СсылкаНаОбъект("Сущность", Ид)` — для результатов запроса, где колонка `Ссылка`
  компилируется в `id` и приходит UUID-строкой без типа (давайте ей псевдоним: `ВЫБРАТЬ Ссылка КАК Ид …`).

## Состояние

Сделано:

- Метаданные + загрузка (`internal/page`), реестр (`GetPage`/`GetPageProcedure`).
- Роутер с проверкой ролей и параметрами строки запроса → структура `Параметры`.
- Построитель со всеми блоками выше; форматирование KPI; график через ECharts.
- Валидация `onebase check` (`internal/configcheck/pages.go`): наличие обработчика,
  процедуры `ПриФормировании`, контроль дублей имён, компиляция текстов запросов.
- langref (`internal/dsl/langref`) + раздел «Страницы» в `DEVELOPER.md`.
- Тесты: `internal/page/page_test.go`, `internal/ui/handlers_page_test.go`,
  `internal/dsl/interpreter/page_builtins_test.go`.
- Демо: `examples/trade` — страница «Панель» (раздел «Отчёты», `/ui/page/Панель`).
- **Кнопки-действия** (доработка 1, сделано): `Страница.КнопкаДействие(текст, "Имя")`
  → POST `/ui/page/{name}/action/{action}` → процедура-действие из `.page.os`
  (резолв как `ПриФормировании`) → PRG-редирект на страницу, `Сообщить()` через
  общий бар. Поле `PageBlock.Action`, маршрут `s.pageAction`, шаблон `page-custom`,
  langref (объект «Страница»), DEVELOPER.md, тесты, демо «Обновить сводку».
- **Конфигуратор** (доработка 2, сделано): объект «страница» создаётся/правится/
  удаляется в визуальном конфигураторе — узел «Страницы» в дереве, панель-редактор
  (заголовок, иконка, роли, параметры + код обработчика с «Проверить» через общий
  `runCheck('dsl',…)`). Файлы пишутся под исходным регистром имени (`pages/<Имя>.yaml`
  + `src/<Имя>.page.os`), работает и в file-, и в database-режиме. См. ниже.
- **i18n подписей блоков** (доработка 3, сделано): авторские подписи блоков
  переводятся через `s.tr` при рендере; данные из запросов не трогаются. См. ниже.

Осталось / возможные доработки:

- **`НСтр("текст")`** (опц.): явная отметка статической части динамически собранной
  подписи (`НСтр("Отчёт за ") + Период`), которую авто-уровень целиком в ключ не
  свернёт. Нужен доступ к языку запроса из интерпретатора — отдельная задача.

## План доработок

### 1. Кнопки-действия (вызов процедуры обработчика) — ✅ СДЕЛАНО

Цель: кнопка на странице вызывает серверную процедуру (провести, экспортировать,
пересчитать…), а не только переходит по URL.

Опора — события управляемых форм (`internal/ui/handlers_managed_events.go`):
фронт шлёт POST, сервер резолвит процедуру и исполняет через `interp.Run`, в ответ
— JSON с `messages`/`values`/`error`. Ключевое: `Registry.GetPageProcedure(pageName,
procName)` уже резолвит **любую** Экспорт-процедуру из `src/<имя>.page.os` по имени —
инфраструктура резолва готова, как у `ПриФормировании`.

Подход:
- DSL: метод `Страница.КнопкаДействие(текст, "ИмяДействия")` → блок `button` с полем
  `Action` (имя процедуры) вместо `URL`.
- Обработчик — `Процедура ИмяДействия(Страница, Параметры) Экспорт` в том же `.page.os`.
- Роутер: `POST /ui/page/{name}/action/{action}` — `GetPageProcedure` → `interp.Call`
  с теми же dslVars (`Страница`, `Параметры`, `MovementsCollector`), сбор `Сообщить()`.
- Результат: проще всего — после действия заново выполнить `ПриФормировании` и
  перерисовать страницу + флеш-сообщения (full reload); позже — AJAX-обновление блоков.
- Файлы: `page_builtins.go` (метод + поле `Action`), `handlers_page.go` (маршрут),
  шаблон `page-custom` (кнопка как submit/AJAX), langref + DEVELOPER.md, тест.
- Объём: ~0.5–1 день.

### 2. Поддержка страниц в конфигураторе лаунчера — ✅ СДЕЛАНО

Цель: создавать/редактировать страницы в визуальном конфигураторе (раньше — только в файлах).

Опора оказалась ближе к **обработке** (`processors/*.yaml` + `src/*.proc.os`), а не к
управляемым формам: страница — это тоже пара «метаданные YAML + обработчик `.os` в
`src/`», правится прямо в дереве (а не отдельной страницей-редактором, как managed-формы).

Сделано:
- **Реестр/проекция**: `cfgPage` + `data.Pages` в `configurator.go`; `loadCfgData`
  наполняет из `proj.Pages` (уже грузится загрузчиком) + `readPageSources` читает
  `src/*.page.os` (ключ — имя в нижнем регистре, как лукап). `.page.os` исключён из
  `readOSSources`, чтобы не оседать в модулях объектов.
- **Дерево + редактор**: узел «Страницы» (`cfgNewObj('page')`) и панель `page-<Имя>`
  в `configurator_tmpl.go` — заголовок, иконка, роли, параметры (списки через запятую)
  и код обработчика (`pre`/`textarea`, `startEdit`/`endEdit`), кнопки Сохранить/
  Проверить/Удалить. «Проверить» переиспользует общий `runCheck('dsl',…)` →
  `CheckDSLSource` (синтаксис обработчика).
- **Маршруты/обработчики**: `POST …/configurator/page` и `…/page-delete`
  (`pages_handlers.go`). Сохранение пишет обе сущности через `saveConfigFile`
  (file + database), удаление — через `deleteConfigFile`. Переводы заголовка
  (`titles`) переносятся из текущей конфигурации, чтобы визуальная правка их не
  затирала (полноценная правка `titles` — это i18n, п.3).
- **Создание**: «+» в дереве (kind=`page`) кладёт `pages/<Имя>.yaml` + скелет
  `src/<Имя>.page.os` с `Процедура ПриФормировании`. Имя файла — в исходном регистре
  (как демо `pages/Панель.yaml`), чтобы правка перезаписывала тот же файл без дублей
  на регистрозависимых ФС.
- **page.SaveFile/Marshal**: добавлены в `internal/page` (+ `omitempty` на
  необязательных полях для компактного YAML).
- **Тесты**: `page_test.go` (Marshal/SaveFile round-trip), `pages_handlers_test.go`
  (рендер дерева+панели, `splitConfigList`, `readPageSources`, и полный цикл
  new→save→delete в file-режиме).

### 3. i18n подписей блоков — ✅ СДЕЛАНО (авто-уровень)

Цель: статические подписи блоков переводятся для нерусских локалей.

Опора — `i18n.Bundle.T(lang, key)` (русский-как-ключ: непереведённый ключ возвращается
как есть → для `ru` и без словаря это no-op), `s.tr(lang, key)` в `server.go`, язык из
`resolveLang` (пользователь → база → Accept-Language).

Сделано (авто-уровень):
- `Server.localizePageBlocks(lang, blocks)` (`handlers_page.go`) прогоняет через `s.tr`
  **только авторские подписи**: `heading.Text`, `paragraph.Text`, `kpi.Label`,
  `table.Title`/`table.ColumnLabels`, `list.Title`/`item.Text`, `chart.Title`,
  `button.Text`. Вызывается в `s.page` после `builder.Blocks()`. **Данные не трогаем**:
  `kpi.Value`, ячейки таблиц (`Rows`), данные графика, URL.
- **Колонки таблиц**: ключ адресации ячеек и отображаемый заголовок разведены —
  добавлено поле `PageBlock.ColumnLabels` (копия `Columns`), переводится оно; `Columns`
  остаются ключами для `index $row.Cells` в шаблоне (иначе перевод заголовка
  рассогласовал бы ячейки). Шаблон `page-custom`: `<th>` из `ColumnLabels`, тело — из
  `Columns`.
- Ключи автор добавляет в `<проект>/locales/*.json` (внешний словарь поверх
  встроенного). Демо: `examples/trade/locales/en.json` переводит страницу «Панель».
- Динамически собранные подписи (`"Отчёт за " + Период`) ключа целиком не имеют →
  проходят без перевода (для них — опциональный `НСтр`, см. выше).
- Тесты: `handlers_page_test.go` — перевод подписей + неприкосновенность данных, рендер
  HTML (переведённый `<th>` при сохранных ячейках), no-op для `ru`/nil-Bundle/пустого
  языка.

Не делалось: явный `НСтр("текст")` (вынесен в «возможные доработки»).
