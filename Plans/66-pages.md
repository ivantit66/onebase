# Этап 66 — Страницы (произвольные представления на DSL)

**Статус:** ✅ Реализовано (базовая версия + доработки 1–3, см. конец файла)

> Источник: предложение пользователя — «ввести понятие *страницы*, которую можно
> открывать из меню и формировать алгоритмом на встроенном языке как HTML,
> возможно со ссылками на объекты базы; кастомизировать интерфейс при наличии
> фантазии».

## Контекст

Сейчас прикладной разработчик может рисовать собственный UI тремя способами, и у
каждого — своя ниша и свой потолок:

| Механизм | Что даёт | Чего не хватает для «страницы» |
|---|---|---|
| **Виджеты** (план 31/39, `internal/widget`) | KPI/список/график/действия на рабочем столе, декларативно поверх запроса | только фиксированные типы плиток; нельзя свободная композиция/текст/разметка |
| **Управляемые формы** (план 37) | формы объектов (реквизиты, ТЧ, команды) | привязаны к сущности; не «свободный экран» |
| **HTTP-сервисы** (план 61, `internal/httpservice`) | DSL-обработчик возвращает произвольный контент (`ОтветТекст`/`ОтветJSON`) | это «голый» HTTP-ответ вне оболочки приложения (нет шапки/навигации/темы/i18n) и нет пункта в меню |

Чего **нет** — объекта «**страница**»: именованного экрана, который (а) открывается
из меню раздела, (б) формируется обработчиком на встроенном языке, (в) рендерится
**внутри оболочки приложения** (та же шапка, навигация, тема, локализация), (г)
умеет ссылаться на объекты базы. Это аналог «дополнительного рабочего места» /
произвольной формы-документа в 1С, но в безопасной для веб-платформы форме.

### Ключевое проектное решение: структура, а не строка HTML

Соблазнительная реализация — «обработчик возвращает строку HTML, мы её вставляем».
Так делать **не следует** как основной путь — это противоречит уже сделанным в
платформе инвестициям:

- **XSS.** Склейка HTML строками с данными из базы превращает любую запись с
  `<script>` в дыру. В движке уже есть осознанная защита от этого (например
  `internal/webhook/render_escape_test.go`). Сырой HTML из DSL перекладывает
  экранирование на совесть каждого автора конфигурации (включая ИИ-генерацию и
  marketplace из плана 60).
- **Единство UI.** Произвольный HTML мимо шаблонов теряет тёмную тему,
  адаптивную вёрстку (план 45 PWA), доступность и **i18n** — а в локализацию
  вложены десятки `internal/i18n/locales/*.json`.
- **Обход движка.** Сырой HTML обходит навигацию и RBAC уровня форм.

Поэтому страница описывается **структурой блоков** (как виджеты — `widget.Result`),
которую оболочка рендерит в общем стиле с автоэкранированием и переводами. Для
тех, кому действительно нужен произвольный HTML, оставляем явный escape-hatch
(`ДобавитьСыройHTML`) с обязательной санитизацией и без инлайнового JS.

> **Прототип сегодня, без этого плана (фаза 0).** Идею можно пощупать уже сейчас:
> HTTP-сервис (план 61) с обработчиком, возвращающим `ОтветТекст(200, "<html>…",
> "text/html")`, плюс пункт меню-ссылка на `/hs/<root>/...`. Это нулевая
> стоимость, чтобы понять, какие блоки реально нужны, прежде чем строить
> структурный рендер. Минусы (вне оболочки, нет i18n/темы, ручное экранирование)
> — ровно то, что закрывает этот план.

## Синтаксис / UX

### 1. Метаданные страницы — `pages/<имя>.yaml`

```yaml
name: ПанельРуководителя
title: Панель руководителя
titles:
  en: Manager dashboard
icon: layout-dashboard
roles: [Руководитель]      # (необяз.) кому видна/доступна; пусто — всем аутентифицированным
params:                    # (необяз.) объявленные параметры строки запроса (?период=...)
  - период
```

### 2. Обработчик — `src/<имя>.page.os`

Имя файла (без учёта регистра) совпадает с `name`. Обработчик получает **построитель
страницы** `Страница` (мутируемый объект, как `Движения` в проведении) и
`Параметры` (Структура значений из строки запроса), наполняет страницу блоками:

```bsl
Процедура ПриФормировании(Страница, Параметры) Экспорт
    Страница.Заголовок("Сводка на " + Строка(ТекущаяДата()));

    // KPI-блок из запроса
    Выручка = ОстатокДенег();   // обычная функция DSL/запрос
    Страница.Показатель("Выручка за месяц", Выручка, "money");

    // Таблица
    Т = Страница.Таблица("Топ-должники");
    Т.Колонки("Контрагент", "Долг");
    Для Каждого Стр Из ДолжникиЗапрос() Цикл
        Стр2 = Т.ДобавитьСтроку();
        Стр2.Установить("Контрагент", Стр.Контрагент);
        Стр2.Установить("Долг", Стр.Сумма);
        Стр2.Ссылка("Контрагент", СсылкаНаОбъект(Стр.КонтрагентСсылка)); // кликабельная ячейка
    КонецЦикла;

    // Текст и действия
    Страница.Абзац("Данные обновляются при каждом открытии.");
    Страница.Кнопка("Создать счёт", "/ui/document/Счёт/new");
КонецПроцедуры
```

В обработчике доступен **полный набор DSL-переменных**, как в обработках и
HTTP-сервисах: `Запрос`, `Справочники`/`Документы`, `Перечисления`, `Константы`,
`ТекущийПользователь()`/`ИмяПользователя()`, HTTP/Email (под предохранителем
плана 62) и т.д. Страница исполняется **на сервере по GET-запросу** —
побочные эффекты (запись/проведение) допустимы, но не рекомендуются; для действий
используйте кнопки-ссылки на формы/сервисы или кнопки-действия (доработка 1).

### 3. Появление в меню — `subsystems/<раздел>.yaml`

Страница регистрируется в навигации раздела наравне с отчётами и обработками —
новой строкой `pages:` в `contents`:

```yaml
name: Руководителю
contents:
  pages:   [ПанельРуководителя]
  reports: [ОборотноСальдовая]
```

Открывается по `/ui/page/ПанельРуководителя` (с сохранением `?subsystem=` для
подсветки раздела, как ссылка «Главная» в плане 39).

## Объекты DSL

Новый инжектируемый объект-построитель `Страница` и функция-хелпер
`СсылкаНаОбъект`.

### Построитель `Страница`

| Метод | Назначение |
|---|---|
| `.Заголовок(текст)` | заголовок секции (h2) |
| `.Абзац(текст)` / `.Текст(текст)` | текстовый блок (экранируется) |
| `.Показатель(подпись, значение [, формат])` | KPI-плитка; формат `money`/`percent`/`number` (как у виджетов) |
| `.Таблица(заголовок)` → объект таблицы | `.Колонки(...)`, `.ДобавитьСтроку()` → строка с `.Установить(колонка, значение)` и `.Ссылка(колонка, url)` |
| `.Список(заголовок)` → `.Пункт(текст[, url])` | список (опц. со ссылками) |
| `.График(заголовок, вид)` → `.Категории(...)`, `.Серия(имя, Массив)` | график (ECharts), вид `bar`/`line`/`pie` |
| `.Кнопка(подпись, url)` / `.Ссылка(подпись, url)` | навигационная кнопка/ссылка |
| `.КнопкаДействие(подпись, "ИмяПроцедуры")` | кнопка, вызывающая серверную процедуру-действие (доработка 1) |
| `.Разделитель()` | визуальный разделитель |
| `.ДобавитьСыройHTML(текст)` | **escape-hatch**: произвольный HTML; санитизируется (вырезается `<script>`/обработчики `on*`/`javascript:`); помечается в доке «на ответственности автора» |

### `СсылкаНаОбъект(...)` → строка-URL

Возвращает канонический URL карточки объекта `/ui/<вид>/<сущность>/<id>` — то же
соглашение, что уже используют виджеты (`internal/widget/runner.go`). Две формы:
`СсылкаНаОбъект(Ссылка)` (объект-ссылка, например из `НайтиПоКоду`) и
`СсылкаНаОбъект("Сущность", Ид)` — для результатов запроса, где колонка `Ссылка`
компилируется в `id` и приходит UUID-строкой без типа (давайте ей псевдоним:
`ВЫБРАТЬ Ссылка КАК Ид …`).

## Безопасность

- **Автоэкранирование** всех текстов/ячеек по умолчанию (рендер через
  `html/template`, не конкатенация строк).
- **`ДобавитьСыройHTML`** — единственный путь к произвольному HTML, с санитизацией
  (вырезаются `<script>`/`<style>`, обработчики `on*=`, `javascript:`-URI). По
  умолчанию страница без сырого HTML.
- **RBAC.** `roles:` в метаданных проверяется до запуска обработчика (как у
  HTTP-сервисов, план 61): нет роли → 403, нет в меню. Администратор — всегда.
- **Песочница и предохранитель.** Обработчик исполняется через тот же путь, что
  обработки (`dslvars.Common.Build()` + UI-специфичные переменные), значит
  сетевой предохранитель (план 62) и файловый сэндбокс действуют без доработок.
- **Только чтение по смыслу.** Страница открывается по GET; запись/проведение из
  обработчика `ПриФормировании` не рекомендуются. Для побочных эффектов —
  кнопки-действия (доработка 1): POST + PRG, что исключает повтор при F5.

## Хранилище / SQL

Изменений схемы БД нет — страница целиком в метаданных проекта
(`pages/<имя>.yaml` + `src/<имя>.page.os`), как HTTP-сервисы.

## Изменения в коде (базовая версия)

**UI:** `internal/ui/handlers_page.go` — маршрут `/ui/page/{имя}` → проверка ролей →

`internal/ui/server.go` — регистрация маршрута и навигации раздела.

**Метаданные навигации:** `internal/metadata/subsystem.go` — `Pages []string` в
`SubsystemContents`.

**Валидация:** `internal/configcheck/pages.go` — `CheckPages` (наличие обработчика,
процедуры `ПриФормировании`, контроль дублей; компиляция запросов модулей).

**Хот-релоад/CLI/доки:** `reg.LoadPages(...)` в `run`/`dev` (`--watch`); langref;
раздел «Страницы» в `DEVELOPER.md`; пример `examples/trade` (страница «Панель»).

## Чем отличается от 1С

В 1С произвольный экран делают «Общей формой» либо HTML-документом в поле формы
(`ПолеHTMLДокумента`) — мощно, но небезопасно и платформозависимо. Здесь —
**структурные блоки с автоэкранированием и i18n из коробки**, регистрация в меню
одной строкой, хот-релоад (`--watch`), и явный (санитизируемый) escape-hatch для
тех, кому нужен сырой HTML.

## Доработки после первой версии (план 66.1–66.3) — ✅ СДЕЛАНО

Базовая версия (выше) реализована в PR #82. Ниже — доработки, добавленные следом.

### 1. Кнопки-действия (вызов процедуры обработчика) — ✅

`Страница.КнопкаДействие(текст, "Имя")` → блок `button` с полем `Action`. Фронт
шлёт `POST /ui/page/{name}/action/{action}`; роутер резолвит **любую**
Экспорт-процедуру из `.page.os` (как `ПриФормировании`) через `GetPageProcedure`,
исполняет с теми же dslVars (`Страница`, `Параметры`, `MovementsCollector`),
собирает `Сообщить()`. Затем PRG-редирект на страницу (перерисовка через
`ПриФормировании`, сохранение `Параметры`), что исключает повтор действия при F5.
Файлы: `page_builtins.go`, `handlers_page.go` (`s.pageAction`), шаблон
`page-custom`, langref, DEVELOPER.md, тесты, демо «Обновить сводку».

### 2. Поддержка страниц в конфигураторе лаунчера — ✅

Объект «страница» создаётся/правится/удаляется в визуальном конфигураторе по
образцу **обработки** (метаданные YAML + обработчик `.os` в `src/`), а не
управляемой формы: правится прямо в дереве.
- `cfgPage` + `data.Pages` в `configurator.go`; `loadCfgData` из `proj.Pages` +
  `readPageSources` (`src/*.page.os`); `.page.os` исключён из `readOSSources`.
- Узел «Страницы» в дереве (`cfgNewObj('page')`) и панель-редактор
  (`configurator_tmpl.go`): заголовок, иконка, роли, параметры + код обработчика с
  «Проверить» (общий `runCheck('dsl',…)` → `CheckDSLSource`).
- Маршруты `POST …/configurator/page` и `…/page-delete` (`pages_handlers.go`):
  сохранение/удаление через `saveConfigFile`/`deleteConfigFile` (file + database);
  `titles` переносятся из текущей конфигурации.
- Создание кладёт `pages/<Имя>.yaml` + скелет `src/<Имя>.page.os` (исходный регистр
  имени — без дублей на регистрозависимых ФС). Добавлены `page.Marshal`/`SaveFile`
  (+ `omitempty`). Тесты: `pages_handlers_test.go` (рендер + цикл new→save→delete).

### 3. i18n подписей блоков — ✅

**Авто-уровень.** `Server.localizePageBlocks(lang, blocks)` (`handlers_page.go`)
прогоняет через `s.tr` (русский-как-ключ) **только авторские подписи**:
`heading.Text`, `paragraph.Text`, `kpi.Label`, `table.Title`/`table.ColumnLabels`,
`list.Title`/`item.Text`, `chart.Title`, `button.Text`. **Данные не трогаются**:
`kpi.Value`, ячейки таблиц, данные графика, URL. Колонки таблиц: ключ адресации
ячеек и отображаемый заголовок разведены — `PageBlock.ColumnLabels` переводится,
`Columns` остаются ключами для `index $row.Cells`. Ключи автор кладёт в
`<проект>/locales/*.json`; демо `examples/trade/locales/en.json`.

**Явный уровень (`НСтр`).** Функция `НСтр(ИсходнаяСтрока[, КодЯзыка])` (формат
`"ru = '…'; en = '…'"`) раньше по умолчанию брала `"ru"` и не знала языка
интерфейса. Теперь `interpreter.NewNStrFunc(defaultLang)` + инжект в `buildDSLVars`
с языком запроса (`withLang` в `pageProcEnv`) — `НСтр` без явного кода переводит на
язык пользователя (вне запроса — язык базы). Для статической части склеенных
подписей: `НСтр("ru = 'Отчёт за '; en = 'Report for '") + Период`.

### Ещё возможно (необязательно)

- Предпросмотр страницы в конфигураторе.
