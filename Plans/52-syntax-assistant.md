# План 52: Синтакс-помощник OneBase (справка по встроенному языку)

## Контекст

У OneBase **нет аналога 1С-овского «Синтакс-помощника»** — ни справки в редакторе
конфигуратора, ни окна-справочника с методами и сигнатурами. Разработчик конфигурации,
пишущий DSL (`.os`), не может узнать «какие есть функции/методы и как они вызываются»,
не читая исходники платформы или примеры в `examples/`.

Что есть сегодня:
- **`onebase ai-guide`** (`internal/cli/aiguide.go`) генерирует `AGENTS.md` из живого
  реестра функций (`builtinGroups()` → `interpreter.New*Functions()`,
  `KnownBuiltinNames()`), но печатает **только имена** функций; прямо стоит дисклеймер
  «сигнатуры смотрите в примерах». Это экспорт для **внешних** ИИ-агентов, не для
  человека в конфигураторе.
- **README → «DSL (язык модулей)»** — обзорная шпаргалка ключевых конструкций.
- **Редактор Monaco** в конфигураторе (`internal/launcher/configurator_tmpl.go`):
  подсветка с **захардкоженным** списком (~30 имён, строка ~2000) и автодополнение,
  предлагающее **только ~16 ключевых слов** (строка ~2030). Ни функций, ни сигнатур,
  ни hover, ни подсказки параметров. Реестра нет — список руками и устаревает.
- Встроенный **ИИ-помощник** (`internal/ui/ai_chat.go`, `ai_tools.go`) имеет инструменты
  только по **данным** (`описание_данных`, `выполнить_запрос`) — синтаксис языка он не
  знает. ИИ-путь признан неподходящим: справка нужна и без ИИ-агента.

**Корневая причина.** Метаданных сигнатур/описаний у встроенных функций нет нигде:
функции — это `map[string]func(...)` (`builtins.go:30`), `KnownBuiltinNames()`
(`builtins.go:389`) отдаёт **только имена**. Методы объектов (`Запрос.Выполнить`,
`Массив.Добавить`…) вообще не каталогизированы — реализованы через интерфейс
`MethodCallable.CallMethod` (`internal/dsl/interpreter/env.go:12`) со `switch` внутри
каждого типа (`chart.go`, `catalogs_proxy.go`, `collections.go` …).

**Цель.** Полноценный 1С-опыт справки по языку в конфигураторе: IntelliSense в редакторе
(автодополнение/hover/подсказка параметров) **и** отдельное окно-справочник, оба питаются
из **единого реестра дескрипторов**, который заодно обогащает `ai-guide` сигнатурами.
Единый источник = «не устаревает» и для внешнего ИИ, и для человека.

## Решения по развилкам (согласовано)

- **Объём:** и то, и другое — IntelliSense в редакторе **и** окно-справочник (полный
  1С-опыт).
- **Охват контента:** глобальные функции + конструкции языка + **методы объектов** +
  **язык запросов** (максимум).
- **Полнота — гибрид:** жёсткий гейт (`go test` падает) там, где есть с чем сверять
  (глобальные функции против `KnownBuiltinNames()`); мягкий отчёт (`t.Log`, в духе
  `i18ncheck`) по методам объектов и языку запросов (реестра нет); плюс дешёвая
  структурная валидация по всем дескрипторам — тоже в гейте. **Лишний дескриптор**
  (ссылка на несуществующую функцию) **валит тест** — это баг.
- **Архитектура — вариант A:** Go-реестр дескрипторов в новом пакете
  `internal/dsl/langref`. Один источник; `interpreter` его не импортирует (циклов нет);
  потребители — `cli/ai-guide` (напрямую) и конфигуратор (через JSON-эндпоинт).
- **Поле `Deprecated`/`SinceVersion`:** YAGNI для v1; добавить тривиально позже.
- **`ai-guide`:** переводится на `langref`; старая группировка через фабрики
  (`builtinGroups()`/`collect()`) **удаляется полностью** (иначе два источника
  группировки).
- **Контекст `Объект.` в автодополнении:** best-effort по совпадению имени объекта,
  **без вывода типов переменных** (динамический язык; type-inference — не v1).
- **Окно-справочник:** закреплённая боковая панель (справа), сворачиваемая; открыть/закрыть
  кнопкой и **F1**, Esc — закрыть; по умолчанию скрыта. Плавающий/открепляемый режим —
  потом (YAGNI).
- **«Вставить в редактор»:** включаем — клик по записи вставляет сигнатуру в текущую
  позицию курсора активного Monaco.
- **Локализация описаний:** русский, v1 без перевода (EN-алиасы уже есть для матчинга и
  поиска); подписи интерфейса — через `{{t $.Lang …}}` как везде.
- **Зависимости:** новых нет (stdlib + уже подключённые chi/Monaco); сборка без CGo не
  меняется.

## Архитектура (поток данных)

```
Реестр дескрипторов  internal/dsl/langref          (ЕДИНЫЙ ИСТОЧНИК)
   { Имя, Алиасы, Вид, Объект, Сигнатура, Параметры[], Возврат, Описание, Пример, Группа }
        │
        ├──► onebase ai-guide / AGENTS.md       (cli импортирует langref напрямую;
        │                                         теперь С СИГНАТУРАМИ + методы + запросы)
        │
        └──► GET /bases/{id}/configurator/langref → JSON   (launcher)
                 │
                 ├──► Monaco: подсветка + автодополнение + hover + подсказка параметров
                 └──► Окно-справочник (дерево объектов/групп → деталь + «Вставить»)
```

`langref` тяжёлых зависимостей не имеет; `interpreter` его **не** импортирует. Связь с
реестром функций — только в тесте полноты (Часть B).

## Часть A. Пакет `internal/dsl/langref` — модель и организация

Ядро (`langref.go`):

```go
package langref

type Kind string
const (
    KindFunc    Kind = "func"     // глобальная функция: Сообщить(...)
    KindMethod  Kind = "method"   // метод объекта: Запрос.Выполнить()
    KindKeyword Kind = "keyword"  // конструкция: Если … Тогда … КонецЕсли
    KindQuery   Kind = "query"    // язык запросов: ВЫБРАТЬ, .Остатки()
)

type Param struct {
    Name, Type, Doc string
    Optional        bool
}

type Descriptor struct {
    Name      string   // канон. рус. имя, lower-case — ключ матчинга
    Display   string   // для показа: "СтрЗаменить"
    Aliases   []string // англ. синонимы: "StrReplace"
    Kind      Kind
    Object    string   // для методов: "Запрос", "Массив", "ТабличныйДокумент"
    Signature string   // готовая строка сигнатуры
    Params    []Param
    Returns   string   // тип возврата ("" если нет)
    Doc       string   // 1–3 предложения
    Example   string   // опционально, короткий пример
    Group     string   // для дерева: "Строки", "Даты", "HTTP" …
}

func All() []Descriptor                       // агрегатор всех файлов
func ByName(name string) (Descriptor, bool)   // регистронезависимо, по имени и алиасам
func Objects() []string                       // уникальные Object (для дерева)
func Groups() []string                        // уникальные Group функций
```

Раскладка по файлам (мелкие, сфокусированные):
- `functions.go` — глобальные функции, сгруппированы как нынешний `builtinGroups()`
  (Строки, Даты, Число, Коллекции, HTTP, Email, Транзакции, Диаграммы,
  ТабличныйДокумент, Файлы, ИИ-помощник).
- `methods.go` — методы объектов (Запрос, Движения, Массив, Структура, Соответствие,
  ТабличныйДокумент, Диаграмма, ссылки/`Ref` …). Источник — ручной каталог по `switch`
  в `CallMethod`-реализациях интерпретатора.
- `keywords.go` — конструкции языка (Процедура/Функция, Если/Тогда/Иначе/КонецЕсли,
  Для Каждого/Цикл, Пока, Попытка/Исключение, Возврат, Прервать, Продолжить, Новый,
  Истина/Ложь/Неопределено и операторы И/ИЛИ/НЕ).
- `query.go` — слова языка запросов (ВЫБРАТЬ, ИЗ, ГДЕ, УПОРЯДОЧИТЬ ПО, СГРУППИРОВАТЬ ПО,
  соединения, КАК, ВОЗР/УБЫВ), агрегаты (СУММА/КОЛИЧЕСТВО/МИНИМУМ/МАКСИМУМ/СРЕДНЕЕ),
  виртуальные таблицы `.Остатки()/.Обороты()/.СрезПоследних()`.
- `langref.go` — типы + агрегатор `All()` + функции поиска.

## Часть B. Полнота и валидация (гибрид)

`langref/completeness_test.go`:

**Жёсткий гейт (падает `go test`):**
- Каждое имя из `interpreter.KnownBuiltinNames()` имеет дескриптор с `Kind==KindFunc`
  (сверка по `Name` + `Aliases`, регистронезависимо). Нет → «нет описания: `<имя>`».
- Каждый дескриптор-функция ссылается на существующее имя реестра. Нет → «лишний
  дескриптор: `<имя>`» (ловит опечатки и удалённые функции).
- Структурная валидация по **всем** дескрипторам: непустые `Name`/`Display`/`Doc`;
  у `KindMethod` непустой `Object`; нет дублей `(Object, Name)`.

**Мягкий отчёт (`t.Log`, не блокирует):**
- Сводка охвата того, что автоматически не сверить: «методы объектов: N по M объектам;
  язык запросов: K слов; конструкции: P».

Юнит-тесты `langref_test.go`: `ByName` — регистронезависимость, разрешение по алиасу,
методы по `(Object, Name)`; `Objects()/Groups()` — без дублей, отсортированы.

## Часть C. Апгрейд `ai-guide` (AGENTS.md с сигнатурами)

`internal/cli/aiguide.go`:
- `generateAIGuide()` читает `langref.All()` вместо самостоятельного сбора имён.
- Раздел «## Язык DSL» рендерится из дескрипторов: на функцию — `Signature` + `Doc`,
  сгруппировано по `Group`. Добавляются подразделы `### Методы объектов`
  (по `Object`) и `### Язык запросов` — которых раньше в guide не было.
- Дисклеймер «сигнатуры смотрите в примерах» убирается.
- `builtinGroups()`, `collect()`, тип `builtinGroup` и вызовы `interpreter.New*Functions()`
  для группировки — **удаляются** (роль группировки переходит в `langref.Group`).
- Поведение команды `onebase ai-guide [--output] [--claude]` и `onebase init` (кладёт
  текст в `AGENTS.md`) — без изменений, текст лишь богаче.

## Часть D. Редактор конфигуратора — JSON-эндпоинт + Monaco-провайдеры

**Эндпоинт.** Новый `GET /bases/{id}/configurator/langref` → хендлер
`configuratorLangref` (`internal/launcher`, рядом с группой `configurator/*` в
`server.go`, под той же cfg-аутентификацией) отдаёт `langref.All()` как JSON.
Конфигуратор грузит его один раз при инициализации Monaco и кеширует в JS; тот же кеш
питает окно-справочник (Часть E).

**Изменения в `configurator_tmpl.go`:**
1. **Подсветка** — список `builtins` в Monarch-токенайзере (сейчас ~30 имён вручную)
   наполняется из JSON (все имена функций/методов/слов запросов).
2. **Автодополнение** (`registerCompletionItemProvider`) — вместо 16 ключевых слов все
   дескрипторы: `insertText` со сниппетом параметров
   (`СтрЗаменить(${1:Строка}, ${2:Что}, ${3:Чем})`), `detail` = `Signature`,
   `documentation` = `Doc` + `Example`, иконка по `Kind`.
3. **Hover** (`registerHoverProvider`, сейчас нет) — наведение на слово → `ByName` →
   сигнатура + описание.
4. **Подсказка параметров** (`registerSignatureHelpProvider`, сейчас нет) — по `(` и `,`
   список `Params` с подсветкой активного.
5. **Контекст `Объект.`** — best-effort: если токен перед `.` совпадает с известным
   `Object` — показываем методы только его; иначе — общий список. Промахи не ломают
   набор.

## Часть E. Окно-справочник (как F1 в 1С)

Закреплённая сворачиваемая боковая панель (справа), данные — из того же `/langref` JSON,
рендер на клиенте. Отдельный `*_tmpl.go` (чтобы не раздувать `configurator_tmpl.go` —
в проекте так уже сделано для forms).

```
┌─ Синтакс-помощник ───────────────────────────[поиск: строку____] [✕]─┐
│ ОБЪЕКТЫ / ГРУППЫ          │  СтрЗаменить(Строка, Что, Чем)            │
│ ▾ Глобальные функции      │  Возвращает: строка                       │
│    ▸ Строки               │  Заменяет все вхождения «Что» на «Чем».    │
│    ▸ Даты                 │  Параметры:                               │
│ ▾ Методы объектов         │   Строка : строка — исходный текст         │
│    ▾ Массив               │   Что    : строка — что искать            │
│       Добавить            │   Чем    : строка — на что заменить       │
│ ▾ Конструкции языка       │  Пример: Н = СтрЗаменить("а-б-в","-","+");  │
│ ▾ Язык запросов           │                        [Вставить в редактор]│
└───────────────────────────────────────────────────────────────────────┘
```

Поведение:
- Открытие — кнопка в панели конфигуратора + **F1**; Esc — закрыть; по умолчанию скрыта.
- Слева дерево: `Глобальные функции → Group`, `Методы объектов → Object`, `Конструкции
  языка`, `Язык запросов`. Справа — деталь: сигнатура, возврат, описание, таблица
  параметров, пример.
- Поиск вверху фильтрует по имени/алиасу/описанию.
- **«Вставить в редактор»** — вставляет сигнатуру в текущую позицию курсора активного
  Monaco.
- Подписи интерфейса — `{{t $.Lang …}}`; содержимое (описания) — русский.

  группировки.
- **Правка:** `internal/launcher/configurator_tmpl.go` — провайдеры Monaco, загрузка
  `/langref`, кнопка/тоггл панели, обработчик F1/Esc.

## Границы (не входит в v1)

- Вывод типов переменных для контекста `.` (только best-effort по имени объекта).
- Плавающий/открепляемый режим окна-справочника (только закреплённая сворачиваемая панель).
- Перевод описаний на английский (только русский; EN — через алиасы для поиска).
- Поле `Deprecated`/`SinceVersion` в дескрипторе.
- JS e2e-тесты (в проекте нет JS-тест-раннера) — JS покрыт только смоук-рендером +
  ручной проверкой.
- Автоматическая сверка методов объектов и слов запросов с кодом (реестра нет — только
  мягкий отчёт охвата).

## Порядок реализации (ориентир для плана)

1. Пакет `langref`: типы, агрегатор, поиск, тесты полноты/валидации (пустой каталог →
   тест сразу включает гейт).
2. Наполнение `functions.go` (по реестру — гейт ведёт), затем `keywords.go`, `query.go`,
   `methods.go`.
3. `ai-guide` на `langref`, удаление фабрик, тесты guide.
4. Эндпоинт `/configurator/langref` + хендлер-тест.
5. Monaco-провайдеры (автодополнение/hover/подсказка параметров/подсветка) + смоук-тест.
6. Окно-справочник + «Вставить в редактор» + смоук-тест.
7. `onebase check`/`go test ./...` зелёные; ручная проверка IntelliSense и панели.
