# План 88 — Полевой доступ / маскирование ПДн + аудит раскрытия

Дата проектирования: 2026-07-21.
Статус: ⬜ Проект (мотивирован оценкой ТЗ КЦ ICE-32, пробел G1:
`OneService/onebase-cc-platform-traceability-v0.1.md`). Закрывает
CC-SEC-002 (в части «поле»), CC-SEC-003, CC-SEC-004, CC-FR-P02-008, CC-FR-P04-009.

## Контекст

План 79 дал **строковый** доступ (какие строки видит пользователь). Полевого нет:
внутри видимой строки все реквизиты отдаются целиком. Пакет `internal/access`
компилирует только row-предикаты; ни `auth.Permission`, ни рендер форм/списков,
ни REST не умеют «показать телефон замаскированным по роли».

ТЗ КЦ требует ровно этого:

- CC-SEC-003 — маскирование телефона/адреса и иных утверждённых полей по роли и
  контексту;
- CC-SEC-004 — если политика допускает раскрытие поля, операция требует
  право/основание и фиксируется в AuditEvent;
- CC-FR-P02-008 / CC-FR-P04-009 — полный телефон/адрес раскрывается по матрице с
  аудитом; автоопределившийся номер подставляется только в допустимое поле.

Обход «раздельные поля + ролевой гейт видимости» работает, но размазывает ПДн по
схеме и не даёт единого аудита раскрытий. Нужен платформенный механизм рядом с
планом 79.

## Цель

- декларативная **полевая политика** в ролях: для каждого поля — `full` / `mask` /
  `hide`; та же грамматика объектов, что у `row_access` (`catalogs`, `documents`,
  `registers`, `inforegs`);
- маскирование применяется **единообразно** во всех путях чтения (UI-формы/списки,
  REST v1/v2, reference picker, печать, экспорт, отчёты/AI) — как это сделал
  план 79 для строк;
- **контролируемое раскрытие**: явное действие «Раскрыть» требует отдельного права
  и основания, возвращает полное значение и пишет `AuditEvent`;
- **fail-closed** и маскирование **на сервере** (нельзя отдать полное значение в
  HTML/JSON и спрятать в CSS/JS);
- SQLite/dev и PostgreSQL/prod — одинаковая семантика.

## Не цели первого среза

- не шифровать ПДн в БД (это отдельная задача хранения, не контроль доступа);
- не маскировать по значению строки (row-контекст уже даёт план 79);
- не строить UI-редактор политик в первом PR (YAML + lint достаточно для MVP);
- не заменять этим планом задачу минимизации ПДн в конвертах событий (CC-DATA-111).

## Модель в ролях

Рядом с `permissions.row_access`:

```yaml
name: Оператор КЦ
permissions:
  catalogs:
    Клиент: [read]
  field_access:
    catalogs:
      Клиент:
        Телефон:        { read: mask_tail, keep: 4 }   # •••••1122
        МобильныйТелефон: { read: mask_tail, keep: 4 }
        Адрес:          { read: mask_city }             # только город
        Паспорт:        { read: hide }                  # не отдавать вовсе
    documents:
      Заявка:
        Телефон: { read: mask_tail, keep: 4 }
```

Стратегии маскирования (`read:`):

- `full` — полное значение (по умолчанию, если поле не перечислено);
- `mask_tail` + `keep: N` — оставить последние N символов, префикс → `•`;
- `mask_city` — для адреса: оставить город, скрыть улицу/дом (парсер адреса —
  прикладной, платформа даёт хук нормализации);
- `mask_all` — всё значение → фиксированная маска;
- `hide` — поле отсутствует в выдаче (и в списке, и в форме, и в API).

Право на раскрытие — отдельная операция `disclose` в object-level правах:

```yaml
  documents:
    Заявка: [read, disclose]     # роль может запросить полное значение по основанию
```

## Семантика (зеркалит план 79)

1. Пользователей нет → поведение открытое, как сейчас.
2. `admin` → по умолчанию `full`, но раскрытие всё равно аудируется; флаг
   `mask_admin: true` в развёртывании подчиняет и админа маске (CC-SEC-006 —
   админ не получает ПДн автоматически; значение по умолчанию решается на ревью).
3. Object-level `read` остаётся первым фильтром; нет `read` — нет и поля.
4. Несколько ролей → **наименее ограничивающая** побеждает (если хоть одна даёт
   `full`, поле полное) — та же OR-семантика, что у `row_access`.
5. Политика невалидна/непригодна → **fail-closed**: поле маскируется, ошибка в
   lint/лог.

## Где применять

Точки — те же, что закрывал план 79 (иначе появится путь обхода):

- **UI список/форма** (`autogen templates.go`, managed `templates_managed.go`):
  замаскированные поля рендерятся маской; `hide` — колонки/поля нет;
- **REST v1/v2** (`internal/api`): list/get маскируют проекцию; `hide` не попадает
  в JSON;
- **reference picker** (`/ui/_ref-options`): отображаемое поле маскируется, если
  оно чувствительное;
- **печать/экспорт/вложения** (`internal/printform`, экспорт списков): маска
  применяется до генерации PDF/Excel;
- **отчёты, query-компилятор, AI** (`internal/query`): самый рискованный контур —
  как в плане 79D, сначала **fail-closed gate**: non-admin запрос, чьи `Sources`
  содержат объект с активной field-policy на выбранное поле, отклоняется понятной
  ошибкой, пока компилятор не умеет маскировать проекцию в SQL; отдельным этапом —
  маскирование выражений проекции (не постфильтр в Go).

## Раскрытие (CC-SEC-004)

- **UI:** замаскированное поле показывает афорданс «Раскрыть». Клик →
  `POST /ui/{kind}/{entity}/{id}/disclose` c `{ field, reason }`. Сервер проверяет
  object-level `disclose`, пишет `AuditEvent` и возвращает полное значение
  (инлайн, без перезагрузки формы).
- **REST:** `GET /v2/{kind}/{entity}/{id}/field/{name}?disclose=1&reason=…` —
  тот же гейт и аудит.
- **Аудит раскрытия НЕ хранит само значение** (CC-DATA-111): `action=disclose`,
  `entity/id/field/reason/actor/at`, `new_value = "(раскрыто)"`. В `_audit`
  добавляется колонка `reason TEXT`.
- Раскрытие без права → 403 и негативный role-тест (CC-SEC-007).

## Архитектура кода

### auth
Расширить `auth.Permission`:
```go
type Permission struct {
    // …существующее…
    RowAccess   RowAccess
    FieldAccess FieldAccess `yaml:"field_access"`   // kind → entity → field → FieldPolicy
}
type FieldPolicy struct { Read string; Keep int }   // full|mask_tail|mask_city|mask_all|hide
```
Роли без `field_access` работают как раньше.

### access
Пакет `internal/access` (рядом с row-логикой):
- `FieldDecision(user, kind, entity, field) → full|mask|hide (+ strategy)`;
- merge ролей по OR/least-restrictive;
- валидация против metadata (поле существует, стратегия применима к типу);
- `MaskValue(strategy, keep, value) any` — чистые функции маскирования.

### storage / helper
- `MaskRecord(dec map[field]FieldDecision, row map[string]any)` — применить к
  прочитанной строке; общий для list/get/picker/print/export;
- `_audit`: колонка `reason`, действие `disclose`.

### ui / api
- вызвать `MaskRecord` на каждом пути чтения (см. «Где применять»);
- эндпоинты `disclose` (UI + REST) с гейтом + аудитом.

### query / configcheck
- fail-closed gate для отчётов/AI по масковым полям;
- lint `field_access.*`: неизвестный объект/поле, неизвестная стратегия, стратегия
  не подходит типу поля, `disclose` без `read`.

## Storage-чокпоинт (defense-in-depth) — как 79F

Маскирование Go-side держится на дисциплине: новый list/get-хендлер, забывший
`MaskRecord`, тихо сольёт ПДн. Поэтому:
- **тест-сканер** путей чтения падает, если хендлер для объекта с активной
  field-policy не проходит через `MaskRecord` (аналог `assertRowFiltersApplied`);
- строгий рантайм-флаг: строка с непримечённой маской в dev/строгом режиме → ошибка.

## Тесты

- две роли (маска / полное); список/форма/REST/picker/печать/экспорт — все
  маскируют; `hide` не течёт ни в HTML, ни в JSON;
- least-restrictive: роль с `full` перекрывает роль с `mask`;
- раскрытие: без права → 403; с правом → 200 + `AuditEvent` без значения;
- отчёт/AI по масковому полю у non-admin → безопасный отказ (fail-closed);
- админ по умолчанию `full`; `mask_admin: true` подчиняет админа;
- пользователей нет → открыто (обратная совместимость);
- сканер путей чтения ловит хендлер без `MaskRecord`.
