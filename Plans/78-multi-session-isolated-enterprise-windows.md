# Issue draft: параллельные сессии и изолированные окна Предприятия

**Title:** `feat(auth,launcher): параллельные сессии и изолированные окна Предприятия`

## Контекст

Сейчас OneBase мешает трём сценариям:

1. Один и тот же пользователь не может иметь несколько активных сессий: `auth.Repo.CreateSession` перед созданием новой сессии удаляет все старые сессии пользователя. Ту же `CreateSession` вызывает и логин конфигуратора (`internal/launcher/cfgauth.go`), поэтому вход в конфигуратор выбивает сессию Предприятия того же пользователя (и наоборот).
2. Два пользователя в одном браузерном профиле конфликтуют через cookie `onebase_session`, потому что пользовательский режим открывается на одном origin `http://localhost:<port>`.
3. Cookie не привязаны к порту: `onebase_session` с `Path=/` на хосте `localhost` делят все базы и сам лаунчер, так что логин в базу B затирает cookie базы A даже для разных баз. Изолированные профили чинят это только для изолированных окон. Пер-базовые имена cookie (`onebase_session_<port>`) — отдельный follow-up вне этого плана: они ломают передачу сессии конфигуратор→Предприятие через общий cookie домена (`/auth/bootstrap`, `_enterpriseURL()` в `configurator.js`).

Для реальной многопользовательской работы, проверки ролей и GUI-сценариев нужно поддержать:

- несколько активных сессий одного пользователя;
- несколько независимых окон пользовательского режима для разных пользователей;
- работу этого сценария из обычного лаунчера и из `onebase-gui.exe`.

## Цель

Сделать штатный сценарий: из лаунчера можно открыть несколько независимых окон "Предприятия" одной базы и войти в них под разными пользователями, без взаимного перетирания cookie и без выбивания старых сессий.

## Не цели

- Не запускать несколько серверных процессов одной базы на разных портах.
- Не менять модель хранения одной информационной базы.
- Не передавать сессионный токен через URL.
- Не делать полноценный SSO/OAuth.
- Не переписывать весь GUI на другой WebView-фреймворк в рамках первого этапа.

## Предлагаемый подход

Комбинация двух изменений:

1. Разрешить несколько активных сессий пользователя на сервере.
2. Добавить запуск изолированного окна "Предприятия" через отдельный браузерный профиль.

Для `onebase-gui.exe` первый практичный вариант: оставлять сам лаунчер во встроенном WebView, а пользовательский режим открывать во внешнем Chromium/Edge/Chrome с отдельным `user-data-dir`. Текущая зависимость `github.com/webview/webview_go` не даёт публичного параметра `UserDataFolder`, поэтому изолированные профили внутри WebView2 лучше оставить отдельной подзадачей.

## Фаза 1. Серверная модель сессий

### 1.1. Изменить создание сессии

Файл: `internal/auth/users.go`

- Убрать удаление всех старых сессий из `CreateSession`.
- Перед созданием новой сессии удалять только истёкшие сессии.
- Не менять срок жизни сессии в рамках этой задачи: оставить 24 часа, если не будет отдельного решения.
- Кандидат к решению при реализации: скользящее продление `expires_at` в `TouchSession` (п. 1.3). После мультисессий фиксированные 24 часа станут заметнее — несколько окон, умирающих вразнобой посреди работы, — а продление при touch стоит одну строку UPDATE.

Текущее поведение:

```go
DELETE FROM _sessions WHERE user_id = ...
```

Новое поведение:

```go
DELETE FROM _sessions WHERE expires_at <= now()
INSERT INTO _sessions ...
```

### 1.2. Расширить служебные данные сессии

Файл: `internal/auth/users.go`

Добавить idempotent-миграции для `_sessions`:

- `public_id TEXT` — безопасный идентификатор сессии для UI, не сам токен;
- `kind TEXT` — вид сессии: `enterprise` | `configurator`. Нужен админке (отличить окно Предприятия от конфигуратора у одного логина — user-agent не поможет, браузер тот же) и политике лимита сессий (п. 1.6). Заполняется при создании: auth-хендлеры пишут `enterprise`, `cfgLoginSubmit` — `configurator`;
- `created_at TIMESTAMP`;
- `last_seen_at TIMESTAMP`;
- `ip TEXT`;
- `user_agent TEXT`.

Причина: после разрешения нескольких сессий админке нужно отличать "окно 1" от "окно 2", не показывая токены.

Ловушки SQLite — учесть в миграции:

- `ALTER TABLE ADD COLUMN ... UNIQUE` в SQLite запрещён — уникальность `public_id` обеспечивать отдельным `CREATE UNIQUE INDEX IF NOT EXISTS ix_sessions_public_id`;
- `DEFAULT CURRENT_TIMESTAMP` в `ADD COLUMN` тоже запрещён — все новые колонки nullable без default, значения проставляет INSERT;
- заодно добавить индексы `ix_sessions_user_id` (Kick*/лимит) и `ix_sessions_expires_at` (очистка истёкших);
- `EnsureSchema` сейчас глотает любые ошибки ALTER (`users.go`), а вызывается конкурентно из процесса базы и из лаунчера на одном SQLite-файле: молча проглоченный `SQLITE_BUSY` оставит колонку несозданной, и INSERT с новыми полями упадёт. Игнорировать только ошибку «duplicate column», остальные — возвращать/логировать.

Вариант без ломки существующих баз:

- новые колонки nullable;
- новые сессии заполняют все поля;
- старые сессии отображаются с пустыми `public_id/kind/ip/user_agent`.

### 1.3. Обновить API репозитория

Файлы:

- `internal/auth/users.go`
- `internal/auth/handlers.go` — два вызова `CreateSession` (LoginSubmit, LoginJSON);
- `internal/launcher/cfgauth.go` — **третий вызов `CreateSession`** (логин конфигуратора, `cfgLoginSubmit`): смена сигнатуры обязана затронуть и его, поэтому PR №1 честно называется `auth+launcher`.

Добавить методы:

- `CreateSession(ctx, userID string, meta SessionMeta) (token string, err error)` или совместимый overload через новый метод `CreateSessionWithMeta`; в `SessionMeta` — `Kind`, `IP`, `UserAgent`;
- `DeleteExpiredSessions(ctx) error`;
- `KickSession(ctx, publicID string) error`;
- `KickUser(ctx, login string) error` оставить как "завершить все сессии пользователя";
- `KickOtherSessions(ctx, userID, currentToken string) error` для "выйти со всех устройств кроме текущего";
- `TouchSession(ctx, token string, now time.Time) error` с throttling: in-memory map по токену, не чаще раза в 5 минут — SQLite single-writer, лаунчер и база пишут в один файл из двух процессов (WAL + busy_timeout уже есть, но лишние записи ни к чему). Вызывать в `auth.Middleware` после успешного `LookupSession` **и** в `cfgAuthMiddleware` лаунчера — иначе сессии конфигуратора будут выглядеть «мёртвыми» в `last_seen_at`.

`ActiveSessions` должен возвращать несколько строк одного логина и поля:

- `PublicID`;
- `Kind`;
- `Login`;
- `FullName`;
- `IsAdmin`;
- `CreatedAt`;
- `LastSeenAt`;
- `ExpiresAt`;
- `IP`;
- `UserAgent`.

### 1.4. Инвалидировать сессии при смене пароля

Файлы:

- `internal/launcher/admin_handlers.go`

- явно отразить поведение в коде и тестах.

Это компенсирует главный риск пункта 1: старые сессии не должны переживать смену пароля.

### 1.5. Обновить logout и аудит

Изменения:

- Audit-события `login`/`logout` уже пишутся (`handlers.go`, LoginSubmit/LoginJSON/Logout) — добавить только новые:

  - password_change_sessions_revoked;

### 1.6. Опция: лимит одновременных сессий пользователя

Предлагалось отдельным маленьким PR после PR №1; по факту реализована в том же
PR #255 (решение автора: «добить» фичу целиком). Колонка `kind` из п. 1.2
заложена там же.

Фрейминг — не «режим совместимости старый/новый», а политика безопасности: «Максимум одновременных сессий на пользователя». Бизнес-ценность — борьба с шарингом учёток (оператор не должен работать с одной учёткой на двух рабочих местах); прежнее поведение становится частным случаем политики.

- Ключ `auth.max_sessions_per_user` в `_settings` — механизм готов, образцы: `internal/storage/network_switch.go` (`net.enabled`), `ai_audit.go` (`ai.daily_token_cap`). Это операционная политика инстанса, поэтому `_settings`, а не `app.yaml`.
- Пусто/0 = безлимит (по умолчанию — иначе сценарии этого плана мертвы из коробки); 1 = фактически прежнее поведение; N — на вырост при той же реализации.
- Семантика при достижении лимита — **вытеснение самой старой сессии по `last_seen_at`, не отказ во входе**: брошенная сессия (браузер закрыт без «Выйти») при TTL 24 ч заблокировала бы пользователя до вмешательства админа.
- Лимит считает и вытесняет только сессии того же `kind` (enterprise); сессии конфигуратора не считает и не трогает — иначе вернётся баг «вход в конфигуратор выбивает Предприятие».
- Аудит-событие `session_displaced`.
- UI: числовое поле в обеих админках (пользовательский режим и конфигуратор), рядом с настройками аудита.
- Замечание: изолированные окна для **разных** пользователей (фаза 3) работают при любом лимите — вытесняются только сессии того же пользователя.

## Фаза 3. Изолированный запуск Предприятия из лаунчера

### 3.1. UX в лаунчере

Файл: `internal/launcher/templates.go`

Добавить рядом с кнопкой "Предприятие" дополнительное действие:

- "Предприятие" — текущее поведение, открыть обычный URL;
- "Новое изолированное окно" — открыть пользовательский режим в отдельном браузерном профиле.

Минимальный вариант UI:

- отдельная кнопка рядом с "Предприятие";
- или выпадающее меню у кнопки "Предприятие".

Текст должен быть понятен пользователю: "Новое окно (другой пользователь)".

### 3.2. Backend endpoint лаунчера

Файлы:

- `internal/launcher/server.go`
- `internal/launcher/handlers.go`
- новый файл `internal/launcher/isolated_browser.go`

Добавить endpoint:

```text
POST /bases/{id}/start-isolated
```

Поведение:

1. Запустить базу, если она ещё не запущена.
2. Дождаться `/health`.
3. Сформировать URL `http://localhost:<port>/ui`.
4. Создать или выбрать отдельный профиль браузера.
5. Запустить внешнее окно браузера с этим профилем.
6. Вернуть JSON `{ "ok": true }` или `{ "error": "..." }`.

### 3.3. Хранение профилей

Каталог:

```text
~/.onebase/browser-profiles/<base-id>/<profile-id>/
```

Поля профиля:

- base id;
- profile id;
- display name, например `Окно 1`;
- created_at;
- last_used_at;
- browser kind, если определён.

Правило переиспользования — модель «рабочих мест» (иначе «не создавать профиль на каждый клик» и «новое окно» противоречат друг другу):

- клик «Новое изолированное окно» открывает младший свободный (не запущенный сейчас) профиль; если все заняты — создаёт следующий;
- профиль помнит вход (cookie персистентна): повторное открытие «Окна 1» вернёт пользователя без логина. Это фича (окно = рабочее место), но её надо явно задокументировать, чтобы пользователь не ждал экран логина.

Нужна очистка:

- команда в лаунчере "Очистить изолированные профили";
- автоматическая очистка профилей, не использованных 30 дней, если это не рискованно;
- не удалять профиль, пока браузерный процесс жив. Ловушка: PID запущенного дочернего процесса — ненадёжный признак занятости: Chromium при повторном запуске с тем же `--user-data-dir` делегирует существующему процессу и сразу выходит. Практичный вариант — пытаться удалить и пропускать при ошибке (Windows не даст удалить залоченный профиль) либо проверять lock-файл Chromium в каталоге профиля.

### 3.4. Запуск браузера

Новый код: `internal/launcher/isolated_browser_*.go`

Windows:

- приоритет: Edge, Chrome, Chromium;
- искать стандартные пути:
  - `%ProgramFiles%\Microsoft\Edge\Application\msedge.exe`;
  - `%ProgramFiles(x86)%\Microsoft\Edge\Application\msedge.exe`;
  - `%LocalAppData%\Microsoft\Edge\Application\msedge.exe`;
  - `%ProgramFiles%\Google\Chrome\Application\chrome.exe`;
  - `%ProgramFiles(x86)%\Google\Chrome\Application\chrome.exe`;
  - `%LocalAppData%\Google\Chrome\Application\chrome.exe`.

Linux:

- `google-chrome`;
- `google-chrome-stable`;
- `chromium`;
- `chromium-browser`;
- `microsoft-edge`.

macOS:

- `open -na "Google Chrome" --args ...`;
- `open -na "Microsoft Edge" --args ...`;
- `open -na "Chromium" --args ...`.

Аргументы:

```text
--user-data-dir=<profile-dir>
--no-first-run
--new-window
--app=http://localhost:<port>/ui
```

Если `--app` создаёт проблемы на платформе, fallback:

```text
--new-window http://localhost:<port>/ui
```

### 3.5. Fallback

Если Chromium-браузер не найден:

1. Вернуть понятную ошибку в UI.
2. Предложить fallback:
   - открыть обычное окно;
   - открыть вторую сессию через `http://127.0.0.1:<port>/ui` как ограниченный вариант на две сессии.

Важно: `localhost`/`127.0.0.1` не заменяет изолированные профили, потому что даёт только ограниченное количество независимых origins.

## Фаза 4. GUI-режим (`onebase-gui.exe`)

### 4.1. Первый этап

Файл: `internal/launcher/window_webview.go`

Сам лаунчер остаётся во встроенном WebView.

"Новое изолированное окно" из GUI запускает внешний Chromium/Edge/Chrome с отдельным `user-data-dir`.

Плюсы:

- не нужно форкать `webview_go`;
- одинаковая логика для обычного лаунчера и GUI;
- изоляция cookie работает надёжно.

Минус:

- пользовательский режим открывается во внешнем браузерном окне, а не во втором native WebView-окне.

### 4.2. Отдельная будущая подзадача

Если нужен именно второй native WebView, первый кандидат — **без форка `webview_go`**:

- WebView2 читает документированную переменную окружения `WEBVIEW2_USER_DATA_FOLDER`. Второе нативное окно = запуск самого себя подпроцессом (скрытая команда вроде `onebase-gui window --url ...`) с этой переменной, указывающей на каталог профиля.

**Проверено при реализации (2026-07-07): env-кандидат НЕ работает.** Vendored
`webview.h` из `webview_go` сам вычисляет userDataFolder как `%APPDATA%\<имя exe>`
(`GetModuleFileNameW` + `SHGetFolderPathW(CSIDL_APPDATA)`, строки ~3318–3328) и
передаёт его **явным параметром** `CreateCoreWebView2EnvironmentWithOptions` —
явный параметр сильнее переменной окружения (подтверждено живым экспериментом:
профиль создался в `%APPDATA%\<exe>`, а не в каталоге из переменной). Побочный
вывод: все окна одного exe делят один WebView2-профиль, поэтому «нативные
изолированные окна» без патча биндинга невозможны в принципе.

Рабочие варианты (эта подзадача):

- пропатчить/форкнуть `webview_go` (или только его `webview.h`): уважать свою
  переменную окружения при вычислении userDataFolder — патч в одном месте;
- или заменить `webview_go` на обёртку, где можно задавать WebView2 environment/profile;
- сделать платформенные реализации для Windows/macOS/Linux.

Это не включать в первый PR, чтобы не раздувать задачу.

- `ActiveSessions` возвращает несколько строк для одного логина (с `Kind`);
- миграция `_sessions` идемпотентна и работает на SQLite и PostgreSQL (уникальный индекс `public_id` отдельным CREATE INDEX);
- лимит сессий (п. 1.6, отдельный PR): вытеснение старейшей по `last_seen_at`, только своего `kind`, конфигуратор не задет, аудит `session_displaced`.

## Риски и минусы

### Безопасность

Риск: после разрешения нескольких сессий украденная старая сессия не исчезает при новом логине.

Компенсации:

- ревокация старых сессий при смене пароля;
- "выйти со всех устройств";
- админский kick session/kick all;
- аудит session revoke;
- очистка истёкших сессий.

### Кросс-базовый конфликт cookie остаётся для обычных окон

Риск: изолированные профили чинят перетирание cookie только в изолированных окнах; обычные окна разных баз в одном браузере по-прежнему затирают друг друга (cookie не привязаны к порту, см. «Контекст» п. 3).

Компенсации:

- зафиксировано как известное ограничение в контексте и документации;
- follow-up вне плана: пер-базовые имена cookie (`onebase_session_<port>`) + переработка bootstrap-флоу конфигуратор→Предприятие, который сейчас полагается на общий cookie домена.

### Рост `_sessions`

Риск: таблица будет расти.

Компенсации:

- `DeleteExpiredSessions`;
- запуск очистки при login;
- возможно периодическая очистка при старте сервера.

### Админка станет неоднозначной

Риск: один логин будет виден несколько раз.

Компенсации:

- показывать created_at/last_seen/ip/user_agent;
- отдельные действия "завершить сессию" и "завершить все".

### Платформенная сложность запуска браузера

Риск: разные пути Chrome/Edge/Chromium на Windows/Linux/macOS.

Компенсации:

- browser launcher за интерфейсом;
- unit-тесты detection;
- понятный fallback;
- в первом этапе поддержать Windows как основной GUI target, Linux/macOS best-effort.

### Профили занимают место на диске

Риск: `~/.onebase/browser-profiles` будет расти.

Компенсации:

- очистка старых профилей;
- UI-действие очистки;
- хранить профили per base/per window, не создавать новый профиль при каждом клике без необходимости.

### GUI-паритет

Риск: изолированное окно в первом этапе будет внешним браузером, а не native WebView.

Компенсация:

- явно описать это в issue и документации;
- вынести полноценный WebView2 `UserDataFolder` в отдельную задачу.

## Критерии приёмки

- Один и тот же пользователь может иметь две активные сессии одновременно.
- Вход в конфигуратор не завершает сессию Предприятия того же пользователя (и наоборот).
- Два разных пользователя могут одновременно работать в одной базе из лаунчера без перетирания cookie.
- Сценарий работает из `onebase-gui.exe`.
- Админ видит несколько сессий одного пользователя отдельными строками и различает Предприятие/Конфигуратор (`kind`).
- Админ может завершить одну сессию или все сессии пользователя.
- Смена пароля инвалидирует старые сессии согласно принятой политике.
- Истёкшие сессии очищаются.
- Токены не попадают в URL, логи и UI.
- (Отдельный PR) Лимит `auth.max_sessions_per_user` вытесняет старейшую сессию того же вида; 0/пусто = безлимит по умолчанию.
- Тесты auth и launcher/UI обновлены.
- Документация обновлена, включая `docs/features.md` (status: testing).
