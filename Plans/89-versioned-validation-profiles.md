# План 89 — Версионируемые профили валидации и версии правил

Дата проектирования: 2026-07-21.
Статус: ⬜ Проект, **низкий платформенный приоритет** (мотивирован оценкой ТЗ КЦ
ICE-32, пробел G2: `OneService/onebase-cc-platform-traceability-v0.1.md`).
Закрывает CC-RULE-001…004, CC-RULE-007, CC-RULE-008 (TBD-BLOCKING «показать
механизм версий/effective-dating/предпросмотра/публикации»), CC-FR-P02-003/004,

> **Честная переоценка (21.07.2026).** В отличие от плана 88 (маскирование ПДн —
> истинный платформенный пробел, сериализация в REST/печати/экспорте идёт внутри
> движка и из `.os` недостижима), **бо́льшая часть этого плана делается на

> рекомендуемый заход — сначала переиспользуемый **паттерн конфигурации**, а
> вынос в движок — опционально и позже, если паттерн приживётся. Приоритет:
> **план 88 → в движок; план 89 → прикладной паттерн.**

## Контекст

Сейчас прикладная валидация живёт в **императивных хуках** `.os` (`OnWrite`,
серверные события форм). Это работает, но не отвечает требованиям ТЗ КЦ к
**управляемым правилам**:

- CC-RULE-002 — правило должно иметь код, версию, владельца, область, дату
  начала/окончания, статус публикации, ссылку на решение;
- CC-RULE-003 — изменение опубликованного правила создаёт новую версию и не
  меняет смысл сохранённой истории;
- CC-RULE-004 — проверки делятся на `BLOCK`/`WARN`/`INFO` с кодом и понятным
  текстом;
- CC-RULE-007 — откат на прежнюю опубликованную версию без удаления новой;
- CC-RULE-008 (`TBD-BLOCKING`) — OneBase должна **показать** механизм версий,
  effective-dating, тестового предпросмотра и публикации;
- CC-FR-P02-003/004 — заявка не принимается, пока не выполнен **validation
  profile** её типа/филиала/направления; обязательность/видимость зависят от
  правила, а не зашиты в форму.

Кирпичи уже есть: декларативные объекты конфигурации (`internal/metadata`),
DSL-интерпретатор для выражений, **периодические регистры сведений** как
готовый примитив effective-dating (СрезПоследних по `(period, …)`), серверная
валидация констант (required/enum/reference), версионирование конфигурации
(план 60), RBAC (план 04). Настолько же, насколько кирпичей уже хватает для
прикладной сборки, платформенная часть — это удобство и консистентность, а не
недостающая возможность (см. ниже).

## Альтернатива: сначала прикладной уровень (рекомендуемый заход)

Прежде чем трогать движок — что уже собирается **чистой конфигурацией**, без
единой строки Go:

- **Проверки.** Хук `OnWrite`/`ПередЗаписью` уже умеет остановить запись
  (`ВызватьИсключение`) и показать необязательное предупреждение (`Сообщить`) —
  это BLOCK и WARN/INFO соответственно.
- **Правила как данные.** Справочник `ПравилаВалидации` (код, объект, поле,
  уровень, текст, выражение) + цикл по нему в хуке. Правило меняет бизнес-
  пользователь, а не программист.
- **Даты действия и история версий.** Периодический регистр сведений
  (СрезПоследних) даёт «активную на дату» версию правила и хранит прошлые, не
  переписывая их — ровно семантика CC-RULE-002/003.
- **Публикация/черновик.** Флаг статуса в справочнике + фильтр «только
  опубликованные активные» в хуке.

1С, к слову, тоже держит сложную валидацию в коде (`ОбработкаПроверкиЗаполнения`)
плюс флаг «Заполнение» в метаданных — декларативного «движка правил» там нет.

**Что из плана действительно выигрывает от движка** (и только это стоит
платформизации, если она вообще делается):

| Возможность | Почему движок | Оценка |
|---|---|---|
| Гарантия срабатывания на **всех** путях записи (UI/REST/DSL/импорт), а не только где автор повесил хук | Как storage-чокпоинт плана 79 для доступа | средне |
| Готовый цикл черновик→публикация→откат + UI-редактор + аудит из коробки | Не переписывать в каждой конфигурации | удобство |
| Типизированный результат `{код, уровень, поле}` вместо свободного текста `Сообщить` | Аккуратные коды ошибок (CC-ERR-001) | мелочь |

Вывод: **делать сначала как переиспользуемый паттерн** (справочник правил + хук +
периодический регистр + пример в `examples/`), закрывая CC-RULE-008 демонстрацией
на прикладном уровне. Поднимать в движок (разделы ниже) — отдельным решением,
если паттерн приживётся и захочется общий редактор/гарантию путей. Разделы
«Синтаксис/Хранилище/Движок/Интеграция» ниже описывают именно **платформенный
вариант** — как целевую форму, а не как первый шаг.

## Цель (платформенного варианта)

- декларативный объект **профиль валидации**: именованный версионируемый набор
  полевых правил (обязательность/видимость/допустимость + уровень
  `BLOCK`/`WARN`/`INFO`), выбираемый по контексту (тип/филиал/направление);
- **версия правила** с владельцем, областью, периодом действия
  (`effective_from`/`effective_to`), статусом (`draft`/`published`/`retired`),
  ссылкой на решение;
- жизненный цикл **черновик → предпросмотр/тест → публикация → откат**;
- движок применяет **активную на бизнес-момент** версию (effective-dating);
- типизированный результат проверки (код, уровень, поле, текст) для UI/REST/DSL.

## Две категории (CC-RULE-009 / §12.1.1)

- **release-configured** профили — поставляются через Dev/Test/UAT,
  версионируются механизмом конфигурации (план 60). Оператор их не меняет.
- **runtime-configurable** подмножество — правит уполномоченная роль через
  UI/API с черновиком/публикацией/аудитом; хранится в рантайм-таблице с версиями,
  эффективными датами и статусом. Универсальный rule-engine **не** делаем —
  только объявленные категории (CC-ARCH-006).

## Синтаксис / метаданные

Объект `validation/<имя>.yaml` (release-configured; runtime-профили — та же
схема, но живут в таблице и правятся из UI):

```yaml
name: ЗаявкаВыездная
applies_to: Документ.Заявка
owner: Руководитель КЦ
select:
  when: "Тип = 'Выездная'"        # DSL-условие активности профиля
rules:
  - code: REQ-STREET
    level: block
    field: Улица
    require: not_empty
    message: "Укажите улицу"
    effective_from: 2026-08-01
  - code: REQ-PHONE
    level: block
    field: Телефон
    require: format_phone
    message: "Телефон в неверном формате"
  - code: WARN-DUP
    level: warn
    check: "ЕстьВозможныйДубль(ЭтотОбъект)"   # выражение DSL
    message: "Возможный дубль заявки за 24 ч"
```

- `require:` — быстрые полевые операторы (`not_empty`, `in:[…]`, `format_phone`,
  `min_len` …), вычисляются напрямую;
- `check:` — произвольное выражение DSL (через интерпретатор), для сложных
  правил; результат `Ложь` → срабатывание уровня;
- `level:` — `block` (останавливает запись) / `warn` / `info` (не блокируют);
- `effective_from`/`effective_to` — период действия версии правила.

DSL для прикладных сценариев:
```
Результат = ПроверитьПоПравилам(ЭтотОбъект);   // массив {код, уровень, поле, текст}
```

## Хранилище / SQL

- `_validation_profiles` — голова профиля: имя, `applies_to`, `select`, владелец,
  категория (release/runtime);
- `_validation_rule_versions` — версии правил: профиль, `code`, `level`, `field`,
  предикат/выражение, `message`, владелец, `effective_from`/`effective_to`,
  `status` (draft/published/retired), `decision_id`, `published_at`/`published_by`;
- разрешение активной версии = последняя `published` с
  `effective_from ≤ бизнес-время < effective_to` — тот же приём, что срез
  последних периодического регистра;
- **история неизменяема** (CC-RULE-003): публикация/откат добавляют строки, не
  апдейтят старые.

## Движок — `internal/validation` (новый пакет)

- `ResolveProfiles(entity, businessTime, ctxObject) → []Profile` (по `select`);
- `Evaluate(profiles, object) → []CheckResult{Code, Level, Field, Message}`;
  - `require:` — прямые операторы; `check:` — через `dsl/interpreter`;
- `BLOCK` останавливает запись **до** записи (в `entityservice`), `WARN`/`INFO`
  собираются и отдаются UI без блокировки;
- lifecycle: `Publish(profile, versionSet, decisionID)` — новая опубликованная
  версия; `Rollback(profile, toVersion)` — републикует прежнюю как новую активную
  (новая версия не удаляется — CC-RULE-007);
- `Preview(draft, sampleObject)` и `RulesAsOf(profile, date)` — тестовый
  предпросмотр и «что действует на дату» (CC-RULE-008).

## Интеграция

- `internal/entityservice.Save` — прогон `BLOCK`-правил перед записью; сбор
  `WARN`/`INFO`; при `BLOCK` — отказ с массивом кодов/текстов;
- `internal/ui` (managed-формы) — показ результатов по уровням (блок красным,
  warn жёлтым, info серым); **редактор runtime-профилей**: черновик, diff с
  прошлой версией, предпросмотр на образце, публикация, откат;
- `internal/api` — REST `POST /v2/{kind}/{entity}/validate` возвращает результаты
  без записи;
- `internal/dsl/interpreter` — builtin `ПроверитьПоПравилам(Объект)`;
- `internal/configcheck` — lint профилей.

## Права и аудит (CC-RULE-005)

- публикует/откатывает только уполномоченная роль; обычный оператор — никогда;
- публикация/откат пишут `AuditEvent` (`_audit`, действия `publish`/`rollback`,
  ссылка на `decision_id`);
- при недоступной/конфликтной версии — остановить зависимую операцию либо
  документированный safe default, но не молча (CC-RULE-006).

## Тесты

- разрешение профиля по контексту (`select.when`); несколько активных профилей;
- **effective-dating**: правило действует только внутри периода; смена активной
  версии по бизнес-времени;
- `BLOCK` останавливает запись; `WARN`/`INFO` — нет;
- публикация создаёт новую версию, история неизменяема; откат восстанавливает
  прежнюю, не удаляя новую;
- предпросмотр гоняет черновик без публикации; `RulesAsOf(date)` корректен;
- RBAC: оператор не публикует; публикация/откат в аудите;
- `ПроверитьПоПравилам` из DSL возвращает те же результаты, что запись.

## Verification

```bash
onebase run --project examples/callcenter --sqlite cc.db --port 8080
# Профиль ЗаявкаВыездная: REQ-STREET block c effective_from завтра.
# 1. Сегодня принять заявку без улицы → проходит (правило ещё не действует).
# 2. Сдвинуть бизнес-дату/effective_from на прошлое → та же заявка блокируется с кодом REQ-STREET.
# 3. Уполномоченной ролью изменить текст правила → новая версия; старые документы в истории не меняются.
# 4. Откат на прежнюю версию → действует прежний текст, новая версия сохранена; обе видны в аудите.
# 5. Оператором опубликовать правило → 403.
```

## Эстимейт

**Заход 0 — прикладной паттерн (рекомендуемый первый шаг, БЕЗ правки движка):**
справочник `ПравилаВалидации` + хук `OnWrite` + периодический регистр «активная
версия» + пример в `examples/` (напр. заявка в `callcenter`/`trade`). Уже
закрывает CC-RULE-008 демонстрацией на конфигурации. **≈ 2–3 дня, силами
конфигуратора.**

**Заход 1 — платформенный вариант** (отдельным решением, если паттерн приживётся):

- 89A модель `validation/*.yaml` + lint + `Resolve`/`Evaluate` (require+check),
  BLOCK/WARN/INFO, хук в `entityservice`: 4–5 дней;
- 89B effective-dating + версии + хранилище + разрешение активной версии: 3–4 дня;
- 89C runtime-редактор (черновик/публикация/откат/предпросмотр) + аудит + RBAC:
  4–5 дней;
- 89D REST `validate` + builtin `ПроверитьПоПравилам` + тесты + пример: 2–3 дня.

MVP (89A–B) ≈ 1.5 недели; полный контур (89A–D) ≈ 2.5–3 недели.

**Порядок работ по проекту КЦ: сначала план 88 (истинный пробел, в движок), затем
заход 0 плана 89 (прикладной паттерн); заход 1 — по факту потребности.**
