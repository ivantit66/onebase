# План 87 — Реакция интерфейса на real-time события (аналог «УведомленияКлиента» 1С)

**Статус:** 📝 предложение (обсуждение 2026-07-17)

## Контекст

План 74 дал платформе шину «сервер → браузер»: `realtime.Hub`, SSE `GET /ui/events`,
builtin `ОтправитьУведомление(Кому, Событие, Данные)` и JS-мост, который превращает
каждый кадр в window-событие `CustomEvent('onebase:<имя>')` (с ретрансляцией во
вкладки-фреймы через postMessage — issues #322/#323). Из коробки на шину подписаны
два потребителя:

- событие `"уведомление"`/`"notify"` со строкой — простой тост (`ui.js`);
- событие `"звонок.входящий"` — карточка звонка (план 75), **захардкожена в `ui.js`**.

Чего нет: у прикладной конфигурации нет способа заставить конкретную форму или
список **отреагировать** на событие. В 1С 8.3.24 это закрывает
`УведомленияКлиента.ПодключитьОбработчик("Имя", Обработчик)` + клиентская процедура
с `ПоказатьОповещениеПользователя(...)`. Прямой перенос в OneBase невозможен и не
нужен:

1. **Клиентского DSL нет** — весь код исполняется на сервере. «Обработчик формы на
   событие» в лоб означает обратный POST от каждого окна каждого адресата: событие
   на `"*"` при 50 онлайн-пользователях — 50+ одновременных запросов. В 1С проблемы
   нет, потому что обработчик исполняется на клиенте; у нас это фундаментальное
   ограничение архитектуры, и обходить его надо дизайном, а не смягчениями.
2. **Легального места для пользовательского JS нет** — страницы (план 66) при
   санитизации вырезают `<script>`/`on*`/`javascript:` (`page_builtins.go`),
   и это осознанная защита.
3. Захардкоженная карточка звонка — симптом: первый же реальный кейс «форма
   реагирует на событие» пришлось встраивать в движок. Второй кейс (сканер,
   очередь заявок, монитор обмена) снова потребует правки движка, если не дать
   точку расширения.

Главный практический кейс — **«живой список»**: у оператора открыт список
заказов/заявок; сейчас изменения видны только по F5. Тост «появилась заявка» без
обновления списка — полумера.

## Цель

Дать конфигурации декларативный и командный способ реагировать интерфейсом на
события шины — закрыть ~80% ценности «УведомленийКлиента», **не создавая обратного
шторма запросов** (ни одного нового обязательного запроса клиент → сервер) и не
открывая поверхность XSS. Полноценный серверный обработчик формы — зафиксировать
эскизом с условиями старта, не реализовывать.

## Архитектура — три ступени

| Ступень | Что | Обратные запросы | Статус в этом плане |
|---------|-----|------------------|---------------------|
| A | «Живой список»: декларативная подписка `list_refresh_on` + автопубликация `notify_changes` | перечитывание списка (тот же GET, что F5; debounce, только видимая вкладка) | реализуем |
| B | Клиентские команды `ui.*`: издатель шлёт готовую команду, клиент исполняет без кода | нет | реализуем |
| C | Серверный обработчик формы `ПриУведомлении` через существующий form-event канал | POST на каждое событие (opt-in) | только эскиз |

Ступени A и B независимы и могут идти отдельными PR (A — первым: он несёт
основную ценность).

## Ступень A — «Живой список»

### YAML (метаданные сущности)

Рядом с существующими `list_mode`/`list_form` (`metadata.Entity`, `types.go`):

```yaml
# documents/Заказ.yaml
list_refresh_on:            # список перечитывается при любом из событий
  - заказ.изменён           # прикладное событие (шлёт конфигурация)
  - данные.заказ            # служебное событие автопубликации (см. ниже)
notify_changes: true        # opt-in: платформа сама публикует "данные.заказ"
```

- `ListRefreshOn []string` — новое поле `metadata.Entity`; пустое = поведение
  как сегодня.
- `NotifyChanges bool` — opt-in автопубликации служебных событий.

### Автопубликация служебных событий

Без неё «живому списку» нечего слушать, пока автор не допишет
`ОтправитьУведомление` в каждый модуль записи. При `notify_changes: true`
платформа публикует после **успешной** записи/проведения/отмены/удаления:

- имя события: `данные.<lower(имя сущности)>` (например `данные.заказ`);
- payload: `{ "id": "<uuid>", "действие": "записан|проведён|отменён|удалён" }` —
  **без значений полей** (см. Безопасность);
- точки публикации: `entityservice.Save` (`service.go:161`) — основной путь
  записи/проведения; путь DSL `Документы.X.Записать()/Провести()`
  (`ui/dsl_documents.go`) — он отдельный от entityservice; платформенное
  удаление в UI-хендлерах;
- адресация: роли, имеющие право чтения сущности (RBAC, план 04), — по кадру на
  роль `роль:<Имя>`; если в конфигурации права не описаны — `"*"`.

Инжекция издателя — тем же путём, что `Notifier` в `dslvars` (план 74): интерфейс
объявляется на стороне потребителя, `*realtime.Hub` подставляет слой сборки
сервера; nil → тихий no-op (тесты, headless `procrun`, `migrate`).

### Рендер и клиент

- Рендер списка (autogen `templates.go` и managed-форма `kind: list`) помечает
  контейнер списка атрибутом
  `data-ob-refresh-on="заказ.изменён данные.заказ"` (имена через пробел) и
  `data-ob-refresh-src` (URL для перечитывания, по умолчанию — текущий).
- `ui.js`, один универсальный слушатель (рядом с существующим мостом плана 74):
  - собирает множество имён событий из всех `[data-ob-refresh-on]` на странице;
  - на `onebase:<имя>`: помечает контейнеры-подписчики dirty;
  - **debounce 700 мс** на контейнер (пачка событий → одно обновление; семантика
    «показать актуальное состояние» от склейки не страдает — в отличие от
    обработчиков с логикой, дебаунсить которые нельзя без потери событий);
  - обновляет **только при `document.visibilityState === 'visible'`**; скрытая
    вкладка остаётся dirty и обновляется на `visibilitychange` — фоновые вкладки
    не создают нагрузки;
  - обновление: `fetch` того же URL → распарсить ответ → заменить контейнер
    (сохранение позиции скролла; страничный/feed-режим `list_mode` перечитывает
    текущее окно данных).
- Во вкладках-фреймах события уже доставляются postMessage-ретрансляцией —
  дополнительной работы нет.
- **Бонус после обкатки (не в MVP):** разрешить атрибут `data-ob-refresh-on` в
  санитайзере страниц (`page_builtins.go`) — страницы плана 66 получат живые
  блоки без единой строки JS. Значение атрибута фильтруется по
  `[\p{L}\p{N}._\- ]`, так что вектор XSS не появляется.

### Почему шторм не возникает

Перечитывание — тот же GET, который пользователь и так делает по F5, но реже
(debounce) и только в видимых вкладках. Сервер не получает **ни одного** нового
обязательного запроса; для события на `"*"` при 50 видимых списках — 50 GET,
размазанных дебаунсом, что эквивалентно «все нажали F5», штатной нагрузке
серверного рендеринга. Издержка управляемая и пропорциональна реально смотрящим.

## Ступень B — клиентские команды `ui.*`

Зарезервированный namespace имён событий `ui.*`: издатель (DSL на сервере — он и
есть «обработчик, исполненный один раз в момент публикации») шлёт **готовую
команду**, JS-мост её исполняет. Обратных запросов нет вообще.

### Команды MVP

- **`ui.оповещение`** — богатый тост, полный аналог
  `ПоказатьОповещениеПользователя`:
  `{ "заголовок": "...", "текст": "...", "важность": "обычное"|"важное", "ссылка": {"вид":"document","сущность":"Заказ","id":"<uuid>"} }`.
  «Важное» не автоскрывается (закрытие крестиком); клик по тосту со ссылкой
  открывает форму объекта во вкладке. Картинку из 1С не тащим; опционально —
  имя Lucide-иконки (`иконка: "phone"`), спрайт уже вшит (план 73).
- **`ui.обновитьСписок`** — `{ "сущность": "Заказ" }`: дёргает те же контейнеры,
  что ступень A, независимо от YAML-декларации (императивный push «обнови сейчас»).
- **`ui.открытьФорму`** — `{ "вид": "document|catalog|processor|report|page", "сущность": "...", "id": "<uuid>|пусто" }`:
  оболочка открывает вкладку. Кейс телефонии «открыть карточку по звонку» решается
  конфигурацией, а не хардкодом.

### DSL-сахар

Builtin **`ПоказатьОповещениеПользователя(Кому, Заголовок, Текст, Ссылка, Важность)`**
(алиас `ShowUserNotification`) — тонкая обёртка над
`ОтправитьУведомление(Кому, "ui.оповещение", …)`. Имя намеренно 1С-ное; отличие
(первый параметр `Кому` — потому что вызов серверный) зафиксировать в langref и
доке. Правки по паттерну плана 74: `notify_builtins.go` + `KnownBuiltinNames` +
`langref/functions.go`; `AGENTS.md` подхватится через `ai-guide`.

`ui.обновитьСписок`/`ui.открытьФорму` шлются обычным `ОтправитьУведомление` —
отдельный сахар не нужен.

Существующие контракты (`"уведомление"`/`"notify"`, `"звонок.входящий"`) не
трогаем; перевод карточки звонка на `ui.*` — возможный следующий шаг, не здесь.

## Ступень C — серверный обработчик формы (только эскиз, НЕ реализуем)

Для кейсов, где нужна логика с доступом к **состоянию конкретной формы** (подсветить
строку, дозаполнить несохранённые поля). Эскиз, чтобы будущая реализация не была
наивной:

- подписка в `form.yaml`: `subscriptions: [{событие: "Собака", обработчик: "ПриПолученииСообщения"}]`;
  процедура в `.form.os` получает `(Форма, ИмяСобытия, МассивДанных)`;
- транспорт — существующий канал `POST /ui/{kind}/{entity}/form-event`
  (`handlers_managed_events.go`): `_event=ПриУведомлении` + данные события +
  текущие значения формы; ответ — штатный `formEventResponse`
  (values/tableparts/messages);
- обязательные ограничители: только видимая вкладка; jitter 0–1.5 с; коалесинг
  очереди в `МассивДанных` (обработчик видит все события, но вызывается один раз);
  минимальный интервал между вызовами на окно (~2 с); **запрет подписки на
  широковещание `"*"`** — только адресные события (логин/роль), это отсекает
  сценарий «50 окон × одно событие» по построению;
- условие старта работ: живой прикладной кейс, который ступени A и B не покрыли.

## Безопасность

- Доставка уже ограничена identity подписчика (`hub.matches`: логин/роль/`*`) —
  ступени A и B ничего не расширяют: клиент реагирует только на то, что ему и так
  доставлено.
- Автопубликация — **opt-in** (`notify_changes`), payload без значений полей
  (только id + действие): не обходит строковый доступ RLS (план 79) — утекает
  только факт изменения, и только ролям с правом чтения сущности.
- `ui.открытьФорму`/`ссылка` тоста: URL собирается **на клиенте** из
  валидированных частей (`вид` — белый список, `сущность`/`id` —
  `encodeURIComponent`), сырые URL от издателя не принимаются — нет open
  redirect и `javascript:`.
- `data-ob-refresh-on` в санитайзере страниц — значение по белому списку символов
  `[\p{L}\p{N}._\- ]`; другие `data-*` атрибуты по-прежнему режутся.
- Потолки шины прежние (план 74): неблокирующая публикация, replay-окно,
  отключение медленного подписчика.

## Тесты

- `metadata`: загрузка `list_refresh_on`/`notify_changes` из YAML; `omitempty`
  не ломает существующие конфигурации.
- `entityservice`: `Save` при `notify_changes: true` публикует `данные.<имя>` с
  верными действием/адресацией; при `false` — не публикует; отдельно путь
  `Документы.X` (`dsl_documents`) и удаление.
- `ui`: рендер autogen-списка и managed-списка содержит `data-ob-refresh-on`
  из метаданных; `static_test.go` — `ui.js` содержит обработчики `ui.*`
  (по образцу существующих проверок статики).
- `interpreter`: `ПоказатьОповещениеПользователя` → Publisher получает
  `ui.оповещение` с верной структурой; nil-Notifier → no-op;
  `page_sanitize_test.go` — `data-ob-refresh-on` переживает санитизацию
  (когда включим для страниц), прочие `data-*` — нет.
- Приёмка: `onebase check` зелёный на `examples/crm` с новыми ключами.

- Ступень A: YAML + автопубликация (оба пути записи) + рендер + `ui.js` + тесты ≈ 2–2.5 дня.

## Открытые вопросы

1. Имена YAML-ключей: плоские `list_refresh_on`/`notify_changes` (в стиле
   `list_mode`) или секция `realtime: {refresh_on: [...], notify_changes: true}` —
   решить на ревью.
2. Служебное событие: `данные.<имя>` — достаточно ли одного имени, или сразу
   разделить `данные.<имя>.записан` / `.проведён` (подписка по префиксу)?
3. `ui.оповещение`: нужна ли иконка (Lucide) в MVP или отложить.
