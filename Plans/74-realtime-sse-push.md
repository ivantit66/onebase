# План 74 — Realtime-push в основном UI (шина уведомлений «сервер → браузер», SSE)

**Статус:** ✅ Сделано — `internal/realtime` Hub, SSE `/ui/events`, builtin `ОтправитьУведомление`, JS-мост в layout. (langref-описание builtin'а добавлено в стеке плана 75.)

## Контекст

В основном UI onebase нет канала «сервер → браузер»: формы — классический серверный рендеринг (План 31 прямо фиксирует: «В основном UI push-канала (SSE/WebSocket) нет»). Из-за этого невозможны: всплывающие уведомления, живые дашборды/KPI, индикаторы прогресса долгих обработок, оповещения о новых задачах/документах и — мотивирующий кейс — «всплывашка входящего звонка» (план 75, телефония/CTI).

При этом инфраструктура для SSE в платформе уже есть и проверена:
- эталонная реализация — `internal/deviceagent/agent.go` (`text/event-stream` + `http.Flusher`, кадры `data: …\n\n`, см. строки ~405–419);
- в `internal/api/server.go` `ReadTimeout/WriteTimeout` НАМЕРЕННО не выставлены (в комментарии упомянут «SSE-стрим отладчика») — длинные стримы не обрываются таймаутом;
- роутер — chi, UI монтируется в защищённой (session-auth) группе через `uiSrv.Mount(r)`.

Не хватает **общего, переиспользуемого механизма**: внутрипроцессной шины, HTTP-эндпоинта подписки и DSL-функции публикации. Это и закрывает план.

## Цель

Дать конфигурации возможность из встроенного языка отправлять уведомление конкретному пользователю / роли / всем и доставлять его в браузер подписанного пользователя в реальном времени (SSE), с готовым JS-мостом, на который вешает обработчик любая страница/панель.

## Архитектура

Внутрипроцессная шина `realtime.Hub` (горутино-безопасная: подписчики-каналы, маршрутизация по identity подписчика). Один SSE-эндпоинт `GET /ui/events` в защищённой группе подписывает текущего пользователя и стримит события, адресованные ему / его роли / всем. Публикация — из DSL встроенной функцией `ОтправитьУведомление`, доступной в обработках, HTTP-сервисах, обработке проведения и регламентных заданиях. Однопроцессно (как `БлокировкаДанных`); многоузловой брокер — отдельная будущая работа.

## Синтаксис / UX

### DSL-публикация
```bsl
// Кому: логин пользователя | "роль:ИмяРоли" | "*" (всем онлайн)
// Событие: произвольное имя; Данные: Структура/Соответствие/Массив/строка/число
ОтправитьУведомление("ivan", "звонок.входящий",
    Новый Структура("номер, клиент", "+79990001122", "ООО Ромашка"));

ОтправитьУведомление("роль:Оператор", "очередь.переполнена",
    Новый Структура("очередь, ожидают", "Продажи", 7));

ОтправитьУведомление("*", "уведомление", "Сервер будет перезагружен в 18:00");
```
Английский псевдоним: `PublishNotification(Кому, Событие, Данные)`.

Событие `"уведомление"` / `"notify"` со строковым телом — зарезервированное: встроенный JS-мост показывает его тостом без кода на странице.

### JS-мост (встроен в общий layout)
```js
// Любая страница/панель слушает события шины:
window.addEventListener('onebase:звонок.входящий', e => {
  const d = e.detail;            // {номер, клиент}
  показатьПанельВходящего(d);    // код панели/страницы
});
```
`EventSource('/ui/events')` подключается автоматически на каждой странице UI; переподключение — штатное для SSE. Доставляются только события, адресованные текущему пользователю / его роли / всем.

## Изменения в коде

**Новый пакет `internal/realtime`:**
- `hub.go` — типы `Hub`, `Subscriber{ID, UserID, Login string; Roles []string}`, `Event{Name string; Data any}`;
  - `Subscribe(userID, login string, roles []string) (id string, ch <-chan Event, cancel func())`;
  - `Publish(target string, ev Event)` — `target` ∈ `<login>` | `роль:<Role>` | `*`; матчинг по подписчику; **неблокирующая** отправка (канал переполнен → кадр дропается, издатель не блокируется);
  - буфер канала (например 32), счётчик подписчиков с потолком, `sync.Mutex`.
- `hub_test.go` — фанаут (`*` доходит всем подписчикам), адресация по логину/роли (не доходит чужому), отписка освобождает подписчика, дроп при переполнении не блокирует `Publish`.

**Новый `internal/ui/handlers_events.go`:**
- `func (s *Server) eventsStream(w http.ResponseWriter, r *http.Request)` — берёт `auth.UserFromContext`; аноним → `401`; `Subscribe(...)`; заголовки `text/event-stream`/`Cache-Control: no-cache`/`Connection: keep-alive`, `flusher.Flush()`; цикл `select` по `ch` (кадр `event: <Name>\ndata: <json>\n\n`), `time.Ticker` 25s (heartbeat-комментарий `: ping\n\n`) и `r.Context().Done()` (выход + `cancel()`). Зеркалит паттерн `deviceagent/agent.go`.
- `handlers_events_test.go` — `Content-Type: text/event-stream`; опубликованное в hub событие приходит в тело ответа; без сессии — `401`.

**`internal/ui/server.go`:**
- поле `hub *realtime.Hub`; инициализация в `ui.New(...)`; геттер `Hub() *realtime.Hub` (нужен издателям-builtins).
- в `Mount(r)` — `r.Get("/ui/events", s.eventsStream)` (внутри защищённой группы — session-auth уже применён в `api/server.go`).

**Новый `internal/dsl/interpreter/notify_builtins.go`:**
- `NewNotifyFunctions(pub Publisher) map[string]any` с `ОтправитьУведомление`/`PublishNotification`; интерфейс `Publisher{ Publish(target, name string, data any) }` реализует `*realtime.Hub` через тонкий адаптер, сериализующий `data` в JSON.
- `notify_test.go` — вызов билтина → `Publisher` получил верные target/name/data; JSON-сериализация `Данные`.

**Точечные правки (паттерн как у HTTP-сервисов, План 61):**
- `internal/dslvars/dslvars.go` — добавить `NewNotifyFunctions(...)` в общий namespace DSL.
- `internal/dsl/interpreter/builtins.go` — `ОтправитьУведомление`/`PublishNotification` в `KnownBuiltinNames` (иначе `onebase check` ругнётся «неизвестная функция»).
- проброс `Hub` в места сборки dslVars (UI-обработки, HTTP-сервисы, scheduler) — туда, где сегодня собираются builtins.

**Клиент и документация:**
- `internal/ui/templates.go` — в общий layout: `EventSource('/ui/events')` → на каждое сообщение `JSON.parse` → `window.dispatchEvent(new CustomEvent('onebase:'+name, {detail:data}))`; дефолтный обработчик события `notify`/`уведомление` рисует тост.
- `docs/features.md` — секция «Уведомления в реальном времени» (`<!-- status: testing -->`).
- `AGENTS.md` — функция попадёт автоматически (через `ai-guide`, источник — `KnownBuiltinNames`).

## Безопасность

- Подписка только аутентифицированным (эндпоинт в session-группе; аноним → 401).
- Пользователь получает строго своё: матчинг по его `UserID/Login/Roles`; чужой `<login>` / `роль:` не доставляется. Широковещание `*` — только не приватные сообщения (в доке предупредить).
- Потолок одновременных подписчиков и размера кадра; неблокирующая публикация (backpressure дропом — защита от медленного клиента, без утечки горутин).
- `ОтправитьУведомление` — серверная функция (как и весь DSL); на песочничной/недоверенной поверхности издаётся в тех же рамках, что прочие builtins.
- Однопроцессность: область действия — один процесс onebase (как у `БлокировкаДанных`). Горизонтальное масштабирование требует внешнего брокера (Redis/NATS) — отдельный план.
