## Контекст

Сейчас есть интеграционные примитивы для обмена с **внешними** системами:
вебхуки (план 29), HTTP-сервисы (план 61), HTTP-клиент (план 19), REST API v2
(план 26). Чего нет — механизма синхронизации данных **между базами OneBase**:
центральная база ↔ филиалы, обмен «сайт ↔ учётная база», периодическая
консолидация. В 1С это «Планы обмена»: подписка узлов, регистрация изменений,
пакеты выгрузки/загрузки, разрешение конфликтов.

Без этого распределённый сценарий приходится собирать вручную поверх REST, без
гарантий «выгружено ровно изменённое» и без разрешения конфликтов.

## Цель

- декларативный **план обмена**: состав данных (какие объекты синхронизируются)
  и узлы (участники);
- **регистрация изменений** по узлам: что изменилось с последней успешной
  синхронизации;
- обмен **пакетами** (выгрузка изменений → загрузка на приёмнике) поверх
  существующего транспорта (HTTP-сервисы/REST + токены);
- разрешение конфликтов по правилу (приоритет узла / по времени / хук на DSL);
- идемпотентность и устойчивость к повторной доставке.

## Синтаксис / метаданные

Объект `exchange/<имя>.yaml`:

```yaml
name: ФилиалыЦентр
content:                       # состав обмена
  - Справочник.Номенклатура
  - Справочник.Контрагенты
  - Документ.Реализация
nodes:
  - { code: center, name: Центральная }
  - { code: fil01,  name: Филиал 1 }
conflict: by_node_priority     # by_node_priority | by_time | hook
```

DSL для ручного/регламентного цикла:

```
Пакет = ПланыОбмена.ФилиалыЦентр.ВыгрузитьИзменения("fil01");
ПланыОбмена.ФилиалыЦентр.ЗагрузитьПакет(Пакет);
```

## Хранилище / SQL

- `_exchange_nodes` — узлы плана, this-node, адреса/токены (секреты — план 83);
- `_exchange_changes` — регистрация изменений: план, объект, id, версия, набор
  узлов, которым ещё не отправлено (как «таблица регистрации изменений» 1С);
- запись в регистрацию — в той же транзакции, что и изменение объекта
  (`entityservice`), по составу плана;
- формат пакета — версионированный JSON (переиспользовать сериализацию из REST
  v2 / экспорта конфигурации), с номером сообщения и подтверждением приёма.

## Дизайн

- **Регистрация:** запись/проведение объекта из состава плана добавляет строку в
  `_exchange_changes` для всех узлов, кроме источника изменения.
- **Выгрузка:** «изменения для узла N» = незаконфирмованные строки; пакет несёт
  номер сообщения; после подтверждения приёма строки для N снимаются.
- **Загрузка:** идемпотентная запись по (объект,id,версия); конфликт версий →
  правило `conflict` (приоритет узла / позже победил / DSL-хук
  `ПриКонфликтеОбмена`).
- **Транспорт:** поверх HTTP-сервисов (план 61) с Bearer-токенами (план 26);
  офлайн-вариант — файловый пакет (выгрузить/загрузить `.obx`).
- **Права:** пакеты формируются под сервис-учёткой узла; строковые политики
  (план 79) на границе обмена не должны молча резать состав — обмен идёт
  доверенным каналом с явной сервис-ролью.

## Изменения в коде

- `internal/metadata/` — модель `ExchangePlan` (состав, узлы, правило);
- `internal/exchange/` — регистрация изменений, сборка/разбор пакета, номера
  сообщений/подтверждения, разрешение конфликтов;
- `internal/entityservice/` — хук регистрации изменений в транзакции записи;
- `internal/dsl/interpreter/` — объект `ПланыОбмена.X.…`;
- `internal/httpservice/` — эндпоинты приёма/подтверждения пакетов;
- `internal/ui/` — монитор обмена: узлы, очередь изменений, ошибки, ручной запуск;
- `internal/cli/` — `exchange dump/load` (файловый вариант).

## Тесты

- регистрация: изменение объекта из состава ставит записи всем узлам, кроме
  источника; вне состава — не ставит;
- цикл двух баз (in-process/SQLite): выгрузка→загрузка синхронизирует данные,
  повторная доставка идемпотентна;
- конфликт версий: каждое правило (`by_node_priority`/`by_time`/`hook`);
- подтверждение приёма снимает очередь; потеря пакета → повтор без дублей.

## Verification

1. Две базы, план обмена с общим составом; изменить объект в A → выгрузить в B →
   данные совпали.
2. Изменить один объект в обеих → правило конфликта отработало ожидаемо.
3. Повторно загрузить тот же пакет → без дублей и ошибок.
