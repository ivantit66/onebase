# План 77: лексическая изоляция области видимости процедур DSL

**Статус:** фаза 1 включена по opt-in (аудит 2026-07-08).

## Контекст

В `c73dce7` (fix/212-collections-scope) корректно убрали **сквозную запись** в
родительский scope: присваивание в вызванной процедуре больше не молча меняет
переменные вызывающей. Но осталась дуальная проблема — **сквозное чтение**.

`callUserProc(proc, callEnv, args)` создаёт кадр вызова как `callEnv.child()`
(`internal/dsl/interpreter/interpreter.go:764`, вызовы на 662/669/678/711), то
есть родителем окружения процедуры становится окружение **места вызова**. А
`env.get` идёт вверх по цепочке родителей (`env.go:90`). Итог: вызванная
процедура **видит локальные переменные вызывающей**, которых сама не объявляла.
Это динамическая видимость чтения, а не лексическая 1С-семантика (процедура
должна видеть только свои локальные + параметры + переменные модуля + глобальные
инжекции).

### Воспроизведение (подтверждено)

```
Функция Тест()          Функция Помощник()
  Секрет = 42;            Возврат Секрет;   // не объявлена здесь
  Возврат Помощник();    КонецФункции
КонецФункции
```

`Помощник()` возвращает `42` вместо `Неопределено`. Достижимо в реальных
`.posting.os` / `.proc.os` с функциями-помощниками (резолв через
`LookupSiblingProc`, `interpreter.go:667`).

### Почему это грабли

Помощник, случайно сославшийся на необъявленное имя, молча подхватывает значение
из вызывающей в одном пути вызова и ломается/ведёт себя иначе в другом. Ошибка не
ловится ни компилятором, ни `onebase check`, проявляется только в рантайме на
конкретных данных.

## Стратегия ввода (без резких поломок)

Смена на лексическую видимость — **изменение семантики**: часть конфигураций
может (случайно) полагаться на утечку. Поэтому вводим поэтапно.

### Фаза 0 — детектор (СДЕЛАНО)

Линт `dsl.cross-scope-read` (`internal/configcheck/lint.go`,
`lintCrossScopeReads`) помечает чтение идентификатора, который не объявлен в
текущей процедуре, но является локальной (параметром / `Перем` / переменной
цикла / целью присваивания) другой процедуры того же модуля. Предупреждение, не
меняет рантайм. На всех примерах (`examples/*`) — 0 срабатываний, значит
эталонные конфигурации уже чисты и готовы к строгому режиму.

### Фаза 1 — строгий режим по opt-in (СДЕЛАНО)

- Флаг базы `dsl.strict_lexical_scope: true` в `config/app.yaml` пробрасывается в
  `Interpreter` в CLI/server/offline entrypoint-ах.
- В строгом режиме кадр процедуры больше не наследует окружение места вызова:
  родителем становится per-run **module-env** с `Перем` модуля и затем root-env
  запуска (инжектированные платформенные объекты/factories/extraVars), а не
  `callEnv`.
- `Перем` модуля теперь действительно живут в отдельном module-env одного
  запуска: helper-процедуры читают и пишут их явно через module scope, без
  опоры на динамическую цепочку caller-а и без глобального состояния между
  конкурентными запусками.
- Значения по-прежнему передаются только через параметры и результат функции.
- `onebase check` при включённом флаге повышает `dsl.cross-scope-read` из lint
  warning до блокирующей ошибки.

### Фаза 2 — строгий режим по умолчанию

После того как строгий режим обкатан и `onebase lint` в реальных базах чист,
делаем изоляцию поведением по умолчанию, оставляя явный «unsafe»-переключатель на
переходный период.

## Эскиз реализации (фаза 1)

1. Разделить «окружение места вызова» и «лексический родитель». Сейчас оба — это
   `callEnv`. Нужен доступ к корневому окружению запуска (создаётся в
   `newEnv`/`Run`, держит `this` и инжекции) и per-run module-env для `Перем`
   модуля.
2. В `callUserProc`: при строгом режиме `child` получает родителем module-env
   текущего модуля (если есть `Перем`) или root-env; в legacy-режиме caller-chain
   сохраняется, но module vars читаются/пишутся через module-env.
3. Учесть особые случаи: `Вычислить(...)` намеренно исполняется в текущем
   окружении (видит локальные) — это отдельная фича, её не трогаем.
4. Проверить пути `LookupProc` / `LookupSiblingProc` / `LookupModuleProc` /
   процедуры формы (`__form_procs__`) — все идут через `callUserProc`, значит
   правка одна.
