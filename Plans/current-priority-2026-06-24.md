# Текущий приоритет развития onebase

Дата фиксации: 2026-06-24.

## Рекомендуемый порядок

1. **План 60A — UI истории конфигурации.**
   Довести уже начатое версионирование конфигурации в БД до пользовательской
   фичи: список версий, diff, rollback, сообщение версии и группировка
   batch-сохранений. Это главный следующий шаг после визуального конфигуратора,
   AI-генерации и форм: пользователь должен иметь понятный откат.

2. **План 74 — `onebase fmt` и общий `aicontract`.**
   Стабилизировать формат YAML/DSL-артефактов и машинный контракт для AI,
   конфигуратора и будущего MCP. Это уменьшит шумные diff'ы и сделает AI apply
   предсказуемее.

3. **План 56 — `onebase lint` поверх `check`.**
   Добавить предупреждения по конфигурациям: неиспользуемые переменные DSL,
   объекты без прав в ролях, мёртвые процедуры, слабые поля. Практическая польза
   выше, чем у большой миграции логирования.

4. **План 43.2 — graceful shutdown scheduler.**
   Дренировать активные регламентные задания при остановке и фиксировать
   прерывание в `_scheduled_runs`. Атомарные бэкапы уже сделаны, этот пункт
   закрывает оставшийся надёжностный зазор.

5. **План 26 — REST API v2.**
   Делать при наличии конкретного внешнего потребителя: мобильный клиент, Excel
   или интеграция. Без такого потребителя это крупная работа с меньшей
   немедленной отдачей, потому что HTTP-сервисы уже закрывают часть сценариев.

## Не в топе сейчас

- **План 28** и **план 70** фактически реализованы.
- **План 55, этап 3** полезен как рефакторинг, но не блокирует продукт.
- **План 46** относится скорее к упаковке и маркетингу PWA.
