# План 48: ИИ-помощник в OneBase (бизнес-ассистент на LLM)

## Контекст

Цель — встроить ИИ-помощника так, чтобы **в настройках указывался API-ключ**, а затем
LLM **использовался из конфигурации** для реальных бизнес-задач — прежде всего анализа
запасов и продаж с рекомендациями «что закупить и почему».

Важно отделить от уже существующих планов **38/41 (`fill-on-based`/`ai-dev-tools`* и
`ai-dev-tooling`)** — те про ИИ **для разработчика** (CLI `check/describe/ai-guide`, MCP).
Настоящий план — про ИИ **для бизнес-пользователя и прикладного разработчика конфигурации**:
помощник работает с данными базы (номенклатура, регистры, документы), а не с исходниками
платформы.

Платформа уже даёт всё необходимое как фундамент:
- **Настройки** — key-value таблица `_settings` (`internal/storage/settings.go`) и секции
  `app.yaml` (`internal/project/loader.go:80`, рядом с `EmailConfig`).
- **HTTP-клиент из DSL** — паттерн `dslHTTPConnection` (`internal/dsl/interpreter/http_builtins.go`),
  по нему строится LLM-клиент.
- **Данные для анализа** — компилятор запросов `query.Compile` с виртуальными таблицами
  `.Остатки(дата)` / `.Обороты(нач,кон,период)` (`internal/query/query.go`) и API регистров
  `storage.GetBalances/GetMovements` (`internal/storage/register.go`).
- **Регистрация builtins** — `KnownBuiltinNames()` и группы builtins
  (`internal/dsl/interpreter/builtins*.go`).
- **Фоновые задания** — планировщик `internal/scheduler/scheduler.go` (для регулярного анализа).
- **Формы/команды/диалоги** — `internal/ui/handlers_managed_events.go`, `internal/ui/picker.go`
  (для кнопок «Спросить ИИ» и панели чата).

Решения по развилкам (согласовано):
- Точки интеграции: **DSL-builtin + чат в UI + помощник в конфигураторе + умная загрузка документов**.
- Доступ к данным: **фазами** — сперва конфигурация сама готовит выборку и кладёт в промпт,
  затем tool-use (ИИ сам выполняет запросы через инструменты).
- Провайдер: **мультипровайдерный, с маршрутизацией по задачам и фолбэком**, не один ключ:
  - **Зрение/распознавание документов** → `gemini-2.5-flash`, при исчерпании лимитов —
    фолбэк на `gemini-2.0-flash` (Google, мультимодальный ввод).
  - **Прочие задачи** (анализ, чат, конфигуратор) → **GLM** через Anthropic-совместимый
    endpoint `https://api.z.ai/api/anthropic`.
  - Это значит: конфиг строится не как «провайдер+ключ», а как **профили задач** —
    каждая задача указывает упорядоченную цепочку моделей (фолбэк по лимитам/ошибкам),
    плюс нужен **мультимодальный (vision) ввод** для документов.

## Часть A. Полезные кейсы применения (зачем это бизнесу)

Сгруппированы по типу ценности и привязаны к реальным данным платформы.

### 1. Управление запасами и закупками (главный кейс из запроса)
- **Рекомендация к закупке.** Вход: остатки (`.Остатки`), продажи за период (`.Обороты`),
  средний дневной расход, срок поставки, точка заказа. Выход: список «что и сколько закупить»
  с обоснованием (товар кончится через N дней, продажи растут на X%, до неликвида далеко).
- **ABC/XYZ-анализ номенклатуры.** Классификация по выручке/стабильности спроса → политика
  запаса по группам. ИИ объясняет классификацию словами и предлагает действия.
- **Неликвиды и пересортица.** Найти позиции без продаж M месяцев при ненулевом остатке →
  предложить уценку/акцию/списание с текстом обоснования для руководителя.
- **Прогноз спроса (лёгкий).** По истории оборотов — ожидаемый спрос на следующий период
  с учётом сезонности (ИИ подмечает тренды в переданных рядах).

### 2. Анализ продаж и клиентов
- **Сводка «что происходит».** Естественно-языковой пересказ дашборда: топ-товары, динамика,
  аномалии, провалы. Кнопка «Объясни отчёт» рядом с виджетом.
- **Отток/реактивация клиентов.** Контрагенты, переставшие покупать → текст для менеджера.
- **Кросс-сейл.** «С этим товаром обычно берут…» по совместным продажам в документах.

### 3. Ввод и обработка документов («умная загрузка»)
- **Распознавание входящих.** Текст/скан накладной или письма → черновик документа
  «Поступление» с подбором номенклатуры по справочнику (fuzzy-match). Сильно ускоряет ввод.
- **Нормализация наименований.** Сопоставление «как написал поставщик» ↔ карточка номенклатуры.
- **Автозаполнение реквизитов** документа по контексту (контрагент, договор, ставка НДС).

### 4. Помощь внутри интерфейса (чат-ассистент)
- **Вопрос-ответ по данным базы** на естественном языке: «сколько товара X на складе?»,
  «кто должен больше всех?» — ИИ формирует запрос и отвечает (фаза tool-use).
- **Объяснение и навигация:** «как провести возврат», «где настроить нумерацию».

### 5. Помощь разработчику конфигурации (в конфигураторе)
- Генерация обработчиков `.form.os`/`.proc.os` и запросов из описания на русском.
- Объяснение/ревью существующего DSL-кода, подсказка builtins.
  (Переиспользует контракт `describe`/`ai-guide` из планов 38/41 как контекст для LLM.)

### Что НЕ берём в этот план
Автономные действия без подтверждения (авто-проведение, авто-отправка заказов поставщику),
дообучение моделей, голосовой интерфейс. Все мутации — только черновик + подтверждение человека.

## Часть B. Архитектура (слоями)

### Слой 1 — Провайдеры, модели и маршрутизация по задачам (настройки)
Конфиг строится как **3 сущности**, а не один ключ:

1. **Endpoints (провайдеры/ключи).** Именованные подключения:
   - `kind`: `gemini` | `anthropic` | `openai` | `compatible`.
   - `base_url` (для z.ai/локальных), `api_key`, опц. заголовки, таймаут.
   - Пример: `z_ai` = `{kind: anthropic, base_url: https://api.z.ai/api/anthropic, api_key: …}`;
     `google` = `{kind: gemini, api_key: …}`.

2. **Models.** `{name, endpoint, vision: bool, max_tokens}` — например `gemini-2.5-flash`
   (`google`, vision=true), `gemini-2.0-flash` (`google`, vision=true),
   `glm-4.6` (`z_ai`).

3. **Task profiles (маршрутизация).** Каждая задача → **упорядоченная цепочка моделей**
   (фолбэк по лимитам/ошибкам):
   - `документы` → `[gemini-2.5-flash, gemini-2.0-flash]` (требует vision).
   - `анализ` / `чат` / `конфигуратор` → `[glm-4.6]` (можно добавить фолбэк).
   Профиль по умолчанию для неуказанных задач — настраивается.

Пакет `internal/llm/`:
- `Endpoint`, `Model`, `Profile`, общий `Config` (всё перечисленное), `Settings.Resolve(task)`
  → упорядоченный список `Model` для исполнения с фолбэком.
- `Client` интерфейс: `Complete(ctx, ChatRequest) (ChatResponse, error)` и
  `CompleteWithTools(...)`; реализации `gemini_client.go`, `anthropic_client.go`
  (он же покрывает z.ai/GLM сменой `base_url`), `openai_client.go` (compatible/локальные).
- `ChatRequest` поддерживает **мультимодальный контент**: текст + изображения
  (base64/файл) — для распознавания документов (см. Слой 6).
- **Фолбэк-движок** `Run(ctx, task, req)`: идёт по цепочке профиля; при ошибках
  лимита/квоты (HTTP 429, провайдер-специфичные коды) и временных 5xx — переключается на
  следующую модель; ошибки валидации/контента — не ретраит. Логирует, какая модель ответила.
- HTTP — по образцу `dslHTTPConnection` (`http_builtins.go`): таймауты, обработка ошибок.

Хранение настроек per-база → `_settings`. Чтобы не плодить десятки ключей, хранить весь
LLM-конфиг **одним JSON-значением** под ключом `llm.config` (endpoints+models+profiles+enabled);
геттер/сеттер в `internal/storage/settings.go` (по образцу `GetAuditSettings`). Ключи API —
секрет: не отдавать в `describe`/экспорт `.obz`, в UI маскировать. Опционально дефолты в
`app.yaml` → `LLMConfig` (`loader.go:80`) с `${env:…}`-подстановкой ключей.

UI настроек («ИИ-помощник» в конфигураторе): таблицы Endpoints / Models / Task-profiles,
маски ключей, флаг «включён», кнопка «Проверить» по каждой модели (тестовый вызов, для
vision-моделей — с тестовой картинкой).

### Слой 2 — Доступ к данным (фазами)
- **Фаза 1 — конфигурация готовит данные.** Разработчик в DSL собирает выборку штатным
  `Запрос()` (виртуальные таблицы `.Остатки/.Обороты`), формирует компактную таблицу/JSON и
  передаёт в промпт builtin'у. Никакого автономного доступа — детерминированно и безопасно.
- **Фаза 2 — tool-use.** LLM получает набор инструментов (read-only по умолчанию):
  `выполнить_запрос(текст_запроса, параметры)` поверх `query.Compile`, `получить_остатки`,
  `найти_номенклатуру`, `описание_конфигурации` (срез `describe`). ИИ сам решает, что тянуть.
  Мутации (создать черновик документа) — отдельные инструменты, только с подтверждением в UI.
  Контроль доступа: инструменты исполняются в сессии пользователя и уважают RBAC.

### Слой 3 — Builtin DSL `ЗапросИИ(...)`
Новый файл `internal/dsl/interpreter/llm_builtins.go` (по образцу `http_builtins.go`/
`email_builtins.go`), регистрация в общем списке builtins:
- `ЗапросИИ(Промпт)` / `ЗапросИИ(Промпт, Параметры)` — синхронный вызов, возвращает строку.
  `Параметры` (Структура): `Задача` (профиль маршрутизации, по умолчанию `анализ`),
  `Температура`, `Формат="json"`, `Система` (system-prompt). Конкретная модель и фолбэк
  выбираются профилем (Слой 1) — конфигурация не привязана к имени модели.
- `ЗапросИИДжейсон(Промпт, Схема)` — структурированный ответ (валидируется), возвращает
  Структуру/Массив — удобно для «дай список к закупке полями …».
- `РаспознатьДокумент(Файл, Промпт)` — vision-вызов (задача `документы`): принимает
  изображение/PDF, прогоняет через профиль с vision-моделями (gemini-2.5→2.0 flash).
- Конфиг и маршрутизация берутся из `_settings` базы; вызовы идут через фолбэк-движок
  `llm.Run(task, …)`; ошибки — через `userError` (как в HTTP-builtin).
- Регистрация имени в `KnownBuiltinNames()` и в группе builtins для `ai-guide`/`check`
  (иначе `onebase check` пометит функцию неизвестной — см. план 41, `CheckDSLCalls`).

Пример использования в обработке `закупки.proc.os`:
```
Данные = ВыборкаОстатковИПродаж();           // штатный Запрос()
Промпт = "Проанализируй остатки и продажи, предложи что закупить. Данные: " + ВJSON(Данные);
Ответ = ЗапросИИ(Промпт, Новый Структура("Формат", "json"));
Сообщить(Ответ);
```

### Слой 4 — Чат-ассистент в UI
- Серверный эндпоинт `POST /ui/ai/chat` (`internal/ui/` — рядом с `handlers_managed_events.go`):
  принимает историю сообщений, гоняет цикл tool-use (Слой 2, фаза 2), возвращает ответ + следы
  вызванных инструментов. Сессия пользователя → RBAC соблюдается.
- Клиент: выезжающая панель чата (по образцу модального диалога подбора `openItemPicker`,
  `internal/ui/templates.go`), доступна с рабочего стола; кнопка «Объясни отчёт» возле виджетов.
- История чата в служебной таблице (`_ai_chat` или в памяти сессии на старте — решить при реализации).

### Слой 5 — Помощник в конфигураторе
- Эндпоинт `POST /configurator/ai-assist` (`internal/launcher/server.go` + новый handler),
  в контекст LLM кладётся срез `describe` + список builtins с сигнатурами (план 38, A2) +
  редактируемый фрагмент. Возвращает сгенерированный/исправленный `.os`/YAML.
- Кнопка «Сгенерировать/Объяснить» в редакторе форм и модулей (`configurator_tmpl.go`).

### Слой 6 — Умная загрузка документов (vision)
- Обработка-приёмник: скан/фото/PDF/текст → `РаспознатьДокумент(Файл, Промпт)` (задача
  `документы` → `gemini-2.5-flash`, фолбэк `gemini-2.0-flash`) с инструкцией «извлеки
  позиции» + срез справочника номенклатуры для сопоставления → структурированный JSON →
  черновик документа через `entityservice.Save` (IsNew, без проведения). Для файлов
  переиспользует `ПрочитатьФайл`/`ОбработкаExcel`; изображения уходят в мультимодальный
  `ChatRequest`. Результат — всегда черновик на проверку человеком.

### Сквозное: безопасность, стоимость, наблюдаемость
- **Секреты:** ключ только в `_settings`/env, исключить из `describe`, экспорта `.obz`, логов.
- **PII/конфиденциальность:** предупредить, что данные уходят внешнему провайдеру; для
  чувствительных баз — поддержать локальный `compatible` base_url (данные не покидают контур).
- **Лимиты/стоимость:** ограничение размера контекста, `max_tokens`, таймаут, при желании —
  суточный лимит запросов в `_settings`; кэш одинаковых промптов (опц.).
- **Аудит:** писать факты вызова ИИ (кто, когда, токены, без полного промпта) — переиспользовать
  журнал регистрации (`AuditSettings`).
- **Деградация:** при выключенном/несконфигурированном ИИ builtin даёт понятную ошибку,
  UI-панель прячется.

## Verification

1. **Настройки/маршрутизация:** заводятся endpoints (`google`, `z_ai`), модели и профили
   задач; «Проверить» по модели успешно; ключи не видны в `onebase describe` и экспорте `.obz`.
   При искусственном 429 от первой модели профиля движок переключается на следующую (фолбэк
   `gemini-2.5-flash`→`gemini-2.0-flash` логируется).
2. **Builtin (F2):** обработка `закупки.proc.os` вызывает `ЗапросИИ(...)` с реальными остатками
   из `TradeFull.db` и печатает связную рекомендацию к закупке; `onebase check` зелёный
   (функция распознана). Go-тест с мок-провайдером (HTTP-заглушка) на `internal/llm`.
3. **Чат (F3/F4):** в UI вопрос «сколько товара X на складе и что закупить» → корректный ответ;
   в режиме tool-use в ответе видны вызванные инструменты; RBAC соблюдается (нет доступа к
   запрещённым объектам).
4. **Конфигуратор (F5):** «сгенерируй обработчик, ставящий текущую дату при открытии формы» →
   валидный `.form.os`, проходящий `onebase check`.
5. **Загрузка документов (F6, vision):** фото/скан накладной → `РаспознатьДокумент` через
   `gemini-2.5-flash` → черновик «Поступление» с подобранной номенклатурой; документ не
   проводится автоматически; при недоступности 2.5-flash отрабатывает фолбэк на 2.0-flash.
6. **Сборка/тесты:** `go build ./...`, существующие тесты зелёные, новые тесты `internal/llm`
   и builtin'а проходят на мок-провайдере (без реальных сетевых вызовов в CI).

## Эстимейт

Суммарно ≈ 3–4 недели на все фазы. Минимально полезный срез (кейс «что закупить»):
**F1 + F2 ≈ 5–8 дней**.

## Реализовано (F1 + F2)

- **Ядро `internal/llm`** — `config.go` (endpoints/models/profiles, `Resolve(task)`,
  `Redacted()` для маскировки ключей), `types.go` (мультимодальный `ChatRequest`,
  классификация ошибок), `run.go` (фолбэк-движок: 429/5xx/сеть → следующая модель,
  vision-модели пропускаются для текста и наоборот), провайдеры `anthropic.go`
  (он же z.ai/GLM по `base_url`), `gemini.go`, `openai.go`, общий `httpjson.go`.
  Тесты `run_test.go` (фолбэк на 429, отсутствие фолбэка на 400, vision-skip,
  маскировка ключей) — на `httptest`, без реальной сети.
- **Настройки** — `storage.GetLLMConfig`/`SaveLLMConfig` (весь конфиг одним JSON в
  `_settings`, ключ `llm.config`).
- **Builtin DSL** — `internal/dsl/interpreter/llm_builtins.go`: `ЗапросИИ`,
  `ЗапросИИДжейсон`, `РаспознатьДокумент` через интерфейс `AIAssistant`
  (interpreter не зависит от llm/storage). Адаптер `internal/aiassist` строит
  `llm.Runner` из конфига базы; подключён в `internal/dslvars` (доступен и в UI, и
  в регламентных заданиях). Имена зарегистрированы в `KnownBuiltinNames()` и
  `ai-guide` (группа «ИИ-помощник») — `onebase check` их знает. Тесты `llm_test.go`.
- **UI настроек** — страница «ИИ-помощник» в админ-меню конфигуратора
  (`internal/launcher/ai_handlers.go`): JSON-редактор конфига, заготовка для пустой
  базы (Gemini vision + z.ai GLM), кнопки «Сохранить» и «Проверить» (пробный запрос
  по выбранной задаче). Маршруты в `server.go`, пункт меню в `configurator_tmpl.go`.
- **Документация** — `docs/ai-assistant.md` (настройка, функции, пример «что закупить»).

`РаспознатьДокумент` читает файл с диска напрямую — на F6 переиспользовать sandbox
файловых builtin'ов.

## Реализовано (F3 — чат в UI)

- **Эндпоинты** `internal/ui/ai_chat.go`: `GET /ui/ai/enabled` (показывать ли кнопку) и
  `POST /ui/ai/chat` (история переписки → ответ по профилю `чат`, под auth/RBAC `/ui`).
- **Панель чата** инжектится глобально в `tplHead` (`internal/ui/templates.go`) — как
  панель сообщений: плавающая кнопка 🤖 (показывается только если помощник настроен),
  выезжающее окно с историей и вводом, Enter — отправить. CSS там же.
- На фазе F3 чат отвечает на общие вопросы и рассуждает по тексту переписки; прямого
  доступа к данным базы пока нет — это F4 (tool-use). Тесты `ai_chat_test.go`
  (enabled до/после конфига, ответ через mock-провайдер, 400 на пустой запрос).

## Реализовано (F4 — tool-use)

- **Ядро** `internal/llm/tools.go` (`Tool`/`ToolCall`/`ToolResult`/`ToolExecutor`,
  лимит итераций) + `anthropic_tools.go` — агентный цикл по Anthropic Messages API
  (`tool_use`/`tool_result`), покрывает GLM через z.ai. `Runner.RunWithTools`
  (`run.go`): фолбэк по anthropic-моделям; при отсутствии таковых или пустом наборе
  инструментов деградирует до обычного `Run`. Тесты `tools_test.go` (полный цикл
  вызова инструмента через mock, делегирование при пустых tools, деградация на
  openai-модели). **Tool-use для OpenAI/Gemini-протоколов — follow-up.**
- **Инструменты данных** `internal/ui/ai_tools.go` (read-only, только для админа —
  как консоль запросов): `описание_данных` (интроспекция справочников/документов/
  регистров с полями) и `выполнить_запрос` (через `query.Compile` + `store.QueryAll`,
  потолок 100 строк, результат в JSON). Подключены в чат: `RunWithTools` с
  системным промптом, поощряющим «сначала описание_данных, затем выполнить_запрос,
  числа — только из результатов». Тесты `ai_tools_test.go` (схема, валидный/битый/
  пустой запрос). **RBAC: гейт по `isAdmin`; объектный RBAC на произвольные запросы —
  follow-up.**

## Реализовано (F5 — помощник в конфигураторе)

- **Эндпоинты** `internal/launcher/ai_assist.go`: `GET …/configurator/ai-enabled` и
  `POST …/configurator/ai-assist` (инструкция + опц. текущий код → ответ по профилю
  `конфигуратор`). Системный промпт включает список известных builtin'ов
  (`interpreter.KnownBuiltinNames()`), чтобы модель не выдумывала функции.
- **UI** — плавающая панель 🤖 в шаблоне конфигуратора (`configurator_tmpl.go`,
  показывается только если помощник настроен): ввод задачи, опция «добавить текущий
  код из активного редактора» (берёт значение из Monaco/textarea), вывод кода с
  кнопкой «Копировать». Конфигуратор уже под админ-доступом.
- Контекст пока без среза `describe` конкретной базы (метаданные объектов) — это
  follow-up; модель опирается на builtins, общий промпт и переданный фрагмент.

## Реализовано (F6 — умная загрузка документов)

- **Builtin `РаспознатьИзображение(Base64, MIME, Промпт)`** (`llm_builtins.go`) —
  vision из данных в памяти (загруженный файл/вложение), без записи на диск;
  дополняет `РаспознатьДокумент` (с диска). Оба идут по профилю `документы`
  (`gemini-2.5-flash` → `gemini-2.0-flash`). Тест `TestРаспознатьИзображение`.
- **Паттерн умной загрузки** задокументирован в `docs/ai-assistant.md`: распознать
  скан → `ПарсJSON` → создать черновик документа (`Документы.X.Создать` +
  `Записать` без проведения) с подбором номенклатуры/контрагента обычным `Запрос`.
- Платформенные кирпичи готовы; конкретная обработка-приёмник и upload-UI зависят от
  модели документов базы и пишутся в конфигурации (демо-обработка для examples/trade —
  follow-up). Прямой upload-эндпоинт в UI с авто-созданием черновика не делался: это
  per-base, чище реализуется DSL-обработкой поверх builtin'а.
