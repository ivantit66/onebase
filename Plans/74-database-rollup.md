## Контекст

Свёртка информационной базы (как в 1С: Конфигуратор → Администрирование →
Свёртка) — административная операция: на выбранную дату `D` остатки всех
регистров «сворачиваются» в опорные записи, а старая история движений (и,
опционально, документы) удаляется. Польза — размер БД и скорость виртуальных
таблиц `.Остатки()` (меньше строк сканировать), быстрее пересчёты, архивация.

Механизм **платформенный** (один движок для всех конфигураций) и живёт в
**Конфигураторе** (лаунчере), а не в пользовательском UI запущенной базы —
операция базового уровня, требует монопольного доступа.

## Ключевые решения (зафиксировано в обсуждении)

1. **Регистратор — синтетический** `_СвёрткаБазы` (+ журнал `_rollup`), а НЕ
   документ «ВводОстатков». Не требует правок конфигурации, остаётся
   платформенным, ложится на уже orphan-толерантный код регистров.
2. **Три независимые операции**, чтобы опциональность падала сама:
   (а) сформировать опорные остатки → (б) удалить свёрнутые движения →
   (в) удалить старые документы (отдельный тумблер). (а)+(б) всегда вместе.
3. **Документы — тумблер.** Дефолт — удалять (свёртка без shrink почти
   бессмысленна; объём БД делают табличные части документов, а не движения).
   Режим «оставить» — снять проведение + защитить **датой запрета проведения**
   (иначе перепроведение старого дока вернёт движения `< D` и даст двойной счёт).
   Вариант «оставить проведёнными» невалиден.
4. **UI — страница админ-меню Конфигуратора** (по образцу «Параметры базы»),
   не в `internal/ui/admin.go`.
5. **Страховка решает судьбу фичи:** обязательный бэкап (существующий механизм) +
   монопольный доступ + вся операция в одной транзакции + **пост-чек
   «остатки до == остатки после»** на дату свёртки, иначе откат.

## Текущее состояние кода (выверено чтением)

**Регистры накопления.** Таблица `рег_*`, колонки
`id, recorder, recorder_type, line_number, period, вид_движения` + измерения +
ресурсы + реквизиты (`internal/storage/register.go:163` `WriteMovements`).
Баланс = знаковая сумма `SUM(CASE вид_движения='Приход' THEN +ресурс ELSE -ресурс)`
(`register.go:310` `GetBalances`). В `metadata.Register` (`internal/metadata/types.go:188`)
**нет вида Остатки/Обороты** — все регистры считаются остаточными.

- `GetBalances(ctx, regName, reg, RegFilter{To})` уже даёт «остатки на дату»,
  `To` **включительно** (`register.go:307-310`) — переиспользуем для опорных.
- Массовое удаление движений — готовый шаблон `DeleteOrphanMovements`
  (`register.go:122`); там же `OrphanMovements` (`:72`) — её надо научить считать
  `_СвёрткаБазы` «известным» регистратором, чтобы не помечала опорные сиротами.
- `вид_движения` по умолчанию `'Приход'` (`register.go:176`) — опорную строку
  пишем одним `Приход` с нетто-значением ресурса (может быть отрицательным).

**Удаление документов.** Колонка `deletion_mark` + «Удалить помеченные» уже есть

## Архитектура

### 1. Ядро — `internal/storage/rollup.go` (новый)

```go
type RollupOptions struct {
    Date            time.Time // дата свёртки D (00:00)
    Registers       []string  // включённые регистры накопления
    DeleteDocuments bool      // удалить документы с датой < D
}
type RollupReport struct {
    PerRegister []struct{ Name string; FoldedMovements, OpeningRows int;
                          TotalsBefore, TotalsAfter map[string]any }
    DeletedDocs int
    DanglingRefs []string // v2
}
```

- `RollupPreview(ctx, regs []*metadata.Register, ents []*metadata.Entity, opts) (RollupReport, error)`
  — без записи: для каждого включённого регистра считает сворачиваемые движения
  (`period < cutoff`), опорные строки (`GetBalances`), итоги «до»; список
  документов к удалению. Для UI-предпросмотра и `--dry-run`.
- `Rollup(...)` — в **одной транзакции**:
  1. по каждому включённому регистру: `bal := GetBalances(To: cutoff.Add(-time.Second))`;
     вставить опорные движения (`recorder_type='_СвёрткаБазы'`,
     `recorder=<uuid операции>`, `period=cutoff`, `вид_движения='Приход'`),
     **нулевые комбинации измерений пропускать**; затем
     `DELETE FROM рег_X WHERE period < cutoff AND recorder_type <> '_СвёрткаБазы'`.
  2. `DeleteDocuments` ? удалить документы с датой `< cutoff` (+ их ТЧ;
     переиспользовать purge помеченных) : снять проведение (`SetPosted(false)`)
     у этих документов.
  3. записать **дату запрета проведения** = `cutoff` (см. §2).
  4. журнал `_rollup` (id, cutoff, who, at, folded_movements, opening_rows,
     deleted_docs, summary JSON).
  5. **пост-чек:** пересчитать остатки на `cutoff` (и на дате `> cutoff`) — должны
     совпасть с зафиксированными «до»; иначе `ROLLBACK` + ошибка.
- **Граница периода — единый момент** `cutoff = D 00:00:00`: сворачиваем
  `period < cutoff`, опорные на `period = cutoff`, оставляем `>= cutoff`. Так
  пересчёт остатков на любую дату `>= cutoff` не меняется — это и проверяет
  пост-чек. (GetBalances inclusive → звать с `To = cutoff − 1s`; выверить на
  тестах рядом с `internal/query/moment_test.go`.)
- `EnsureRollupTable(ctx)` создаёт `_rollup`.

### 2. Дата запрета проведения (малая отдельная фича — нужна «keep»-режиму)

- UI — добавить поле в «Параметры базы» (`cfgAdminSettings`), чтобы менять/снимать

## Безопасность / необратимость

- Обязательный бэкап (кнопка inline → существующий механизм лаунчера).
- Транзакция + пост-чек «до == после», иначе откат.
- Монопольный доступ.
- Журнал `_rollup` + запись в журнал регистрации (audit) действия `rollup`.

## Фазы

- **v1:** регистры накопления; опорные + удаление свёрнутых движений; тумблер
  удаления/сохранения документов; дата запрета проведения; мастер в Конфигураторе

## Тестирование

- **storage (SQLite):** свернуть регистр с движениями по обе стороны `cutoff`;
  остатки до==после на `cutoff` и на дате `> cutoff`; нулевые комбинации не
  порождают опорных; повторный `RollupPreview` идемпотентен.
- **граница:** движение ровно на `cutoff` остаётся (не сворачивается).
- **keep-режим:** после свёртки перепроведение старого документа отклоняется
  датой запрета.
- **delete-режим:** документы `< cutoff` и их ТЧ удалены, движения `>= cutoff`
  целы, опорные на месте.
- **UI:** render-тест страницы (как `configurator_render_test.go`).

### Добито 2026-06-24 (ранее отложенное)
- **Обрезка периодических инфо-регистров** `инфо_*` (опционально, по явному выбору):
  на каждую комбинацию измерений остаётся последний срез `period < cutoff`, более
  ранние удаляются (`DELETE … WHERE period < cutoff AND EXISTS(<более поздний срез той
  же комбинации>)`, NULL-safe по измерениям). `СрезПоследних` на дату `>= cutoff` не
  меняется (пост-чек: после обрезки на комбинацию ≤1 строка до cutoff и всего = числу
  комбинаций — иначе откат). Непериодические пропускаются с пометкой. `СрезПервых` и
  историю срезов до даты обрезка ломает — поэтому off по умолчанию (CLI
  `--trim-info-registers` = все периодические; в UI — чек-лист, по умолчанию сняты).
  Ядро: `selectRollupInfoRegs`, `infoTrimCounts`, `infoRegTrimReport`, `trimInfoReg`.
- **Жёсткий гейт по повисшим ссылкам**: в delete-режиме свёртка откатывается, если на
  удаляемый документ ссылается СОХРАНЯЕМАЯ запись. `countDanglingRefs` сделан точным
  (считает только ссылки от выживающих записей: шапки документов с датой `>= cutoff`,
  справочники, и строки ТЧ — через join к шапке-родителю по `parent_id`; ссылки между
  самими удаляемыми не висят). Безопасный выход — keep-режим. Override намеренно НЕ
  добавлен: все ссылочные поля шапок/ТЧ имеют FK, поэтому «удалить вопреки» всё равно
  упёрлось бы в FK.
- **Отложенная проверка FK** при удалении (`deferFKChecks`): SQLite `PRAGMA
  defer_foreign_keys=ON` внутри транзакции свёртки — массовое удаление документов,
  ссылающихся друг на друга (оба до даты), больше не зависит от порядка удаления; на
  коммите FK проверяются (дубль гейта на уровне БД). На PostgreSQL no-op (FK не
  DEFERRABLE) — там порядок миграции + гейт.
- Тесты: `TestRollup_TrimInfoRegister`, `TestRollup_InfoRegister_NonPeriodicSkipped`,
  `TestRollup_DanglingRefsGate`, `TestRollup_DanglingRefs_BetweenDeleted`. E2E на
  `examples/finance` (периодический `КурсыВалют`): обрезка сохраняет `СрезПоследних`.
