# 108. Атомарность побочных записей хука проведения/записи

Статус: **выполнено**
Родитель: issue #360, PR #365.

## Результат

`entityservice.Save` выполняет `OnWrite`/`OnPost` внутри той же транзакции, что:

- provisional-запись шапки нового объекта для валидности обратных FK-ссылок;
- записи справочников и документов, созданных из DSL-хука;
- табличные части и движения;
- признак проведения и регистрация обмена.

Ошибка DSL возвращается как `DSLError` и откатывает весь набор изменений. Удаление
шапки в режиме best-effort больше не используется. Финальное сохранение полей,
изменённых хуком, выполняется без второго увеличения `_version`, поэтому новый объект
коммитится с версией 1.

`buildDSLVars` получает транзакционный context. Вложенный явный
`НачатьТранзакцию()` внутри уже атомарного hook использует SAVEPOINT, а не второе
соединение; это исключает зависание SQLite с `SetMaxOpenConns(1)`.

## Ограничение

Транзакция остаётся открытой на всё время выполнения hook. HTTP, AI, email и команды
ОС внутри `OnWrite`/`OnPost` поэтому не рекомендуются: долгие внешние операции
удерживают блокировки. Такие эффекты следует переносить в post-commit webhook/outbox.

## Регрессии

- `TestSaveNewDocument_PostWithSelfReference` — hook видит provisional-родителя,
  создаёт FK-child, родитель коммитится с `_version=1`.
- `TestSaveNewDocument_HookChildAndFailureRollBackTogether` — ошибка после записи
  child не оставляет ни parent, ни child.
- Тесты `entityservice`, UI и DSL транзакций проверяют прежние сценарии проведения,
  отмены проведения и вложенных транзакций.
