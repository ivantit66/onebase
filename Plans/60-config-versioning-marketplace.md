## Контекст

Конфигурация уже умеет жить **в БД** (`--config-source database`, `internal/configdb`).
Это открывает две связанные возможности, которых ждут от 1С-подобной платформы:

1. **Версионирование (хранилище конфигурации, §4.6).** История изменений (кто/когда/что),
   diff между версиями, откат. Аналог «Хранилища конфигурации» 1С + мост к git.
2. **Marketplace (§4.5).** Реестр готовых YAML-конфигураций + кнопка «Установить» в
   лаунчере. Шаблоны проектов уже есть (план 24, `onebase init --template`) — marketplace
   расширяет их до устанавливаемых из сети.

Обе опираются на один фундамент: конфигурация как версионируемый, проверяемый,
устанавливаемый артефакт. **Предусловие marketplace — песочница DSL (план 57, этап 0)**,
иначе ставить чужой код из интернета опасно.

## Часть A — Версионирование конфигурации в БД

### Хранилище

```sql
-- история версий (снимки конфигурации)
CREATE TABLE _config_versions (
  id          TEXT PRIMARY KEY,        -- uuid
  created_at  TIMESTAMP NOT NULL,
  author_id   TEXT, author_login TEXT,
  message     TEXT,                    -- комментарий к изменению
  snapshot    BLOB NOT NULL            -- сжатый набор объектов конфигурации (как .obz-конфиг-часть)
);
```

Снимок — переиспользовать сериализацию конфигурации из универсального бэкапа
(`internal/backup/universal.go`, формат `.obz` v2 уже кросс-движковый, JSON).

### Поведение
- Каждое сохранение конфигурации в конфигураторе создаёт версию (с сообщением).
- Каждый `onebase deploy --project <dir> --db <dsn> --message <text>` создаёт
  одну релизную версию после импорта конфигурации и применения миграций.
- Страница «История конфигурации»: список версий, **diff** между любыми двумя (по объектам/
  полям/тексту DSL), **откат** к версии (создаёт новую версию = «откат к N», не стирает историю).
- Экспорт версии в `.obz` и в git-дружественный набор файлов (для внешнего versioning).

| Файл | Изменение |
|---|---|
| `internal/configdb/versions.go` (новый) | DDL + create/list/get/diff/rollback версий |
| `internal/configdb` | хук «сохранение → новая версия» |
| `internal/cli/deploy.go` | релизный снимок при накатывании файловой конфигурации в БД |
| `internal/launcher/config_history.go` (новый) | UI истории: список/diff/откат |
| `internal/backup/universal.go` | переиспользовать сериализацию снимка |

## Часть B — Marketplace конфигураций

### Реестр
- Простой git/HTTP-реестр: `index.json` (имя, описание, версия, автор, URL архива,
  контрольная сумма, требуемая версия платформы) + архивы конфигураций (`.obz`-конфиг-часть).
- Старт — официальный реестр на базе `examples/` (trade/crm/tasks/...); позже — сторонние.

### Установка из лаунчера
1. Просмотр каталога (поиск/категории) → выбор конфигурации.
2. Скачивание + проверка контрольной суммы + проверка совместимости версии.
3. **`onebase check`** на загруженной конфигурации (в песочнице) перед применением.
4. Создание новой базы из неё (как `init --template`, но источник — реестр) **или**
   установка в существующую через версионирование (часть A → новая версия + diff на подтверждение).

| Файл | Изменение |
|---|---|
| `internal/marketplace/registry.go` (новый) | модель index, загрузка/проверка checksum/версии |
| `internal/launcher/marketplace_handlers.go` (новый) | UI каталога + установка |
| `internal/cli/install.go` (новый) | `onebase install <id>` (CLI-аналог) |
| `internal/configcheck` | обязательный прогон на устанавливаемой конфигурации |

**Безопасность (критично):**
- DSL устанавливаемой конфигурации исполняется только в **песочнице** (план 57, этап 0):
  ограничение FS/сети/email/CPU, явное согласие на сетевые builtin'ы.
- Проверка контрольной суммы и (в перспективе) подписи автора.
- Предупреждение «конфигурация из стороннего источника исполняет код — установить?».

## Тесты

- `versions_test.go` — создание версии, diff (добавлен/удалён/изменён объект и поле),
  откат создаёт новую версию, история не теряется.
- `registry_test.go` — парсинг index, отказ при несовпадении checksum/несовместимой версии.
- установка: загруженная конфигурация проходит `check` в песочнице → применяется; не
  проходящая → отказ без изменения базы.
- песочница: DSL из marketplace с сетевым вызовом блокируется без согласия.

## Verification

1. Изменить конфигурацию в конфигураторе → в «Истории» появилась версия с автором/сообщением.
2. Diff двух версий показывает изменённые объекты/поля; откат возвращает состояние новой версией.
3. В лаунчере: каталог marketplace → установить «trade» → новая база стартует.
4. Установка повреждённого архива (битый checksum) → отказ.
5. Конфигурация из реестра с сетевым DSL → запрос согласия/блокировка в песочнице.
