## PR 1 — `fix/48-converter-import` (ишью #48 целиком)

### 1.1 Плоские XML-объекты (пропуск объектов без папки-компаньона)

**Проблема.** В выгрузке 1С (v8.3) объект без подчинённых элементов (форм, макетов,
модулей) представлен одним файлом `<Имя>.xml` без каталога-компаньона. Почти все
парсеры в `internal/converter/parser1c/metadata.go` начинаются с
`if !e.IsDir() { continue }` и молча теряют такие объекты. Подтверждено
тестировщиком: потеряны 2 справочника и 4 константы. Для перечислений этот же баг
уже чинился точечно (issue #16, `parseEnumerations`).

**Решение.** Общий хелпер в `parser1c/metadata.go`:

```go
// objectNames возвращает имена объектов раздела выгрузки: объединение имён
// подкаталогов и одиночных *.xml-файлов (без расширения), без дубликатов.
func objectNames(dir string) []string
```

Логика — как в текущем `parseEnumerations` (metadata.go:556–571). Применить во всех
парсерах: `parseCatalogs`, `parseDocuments`, `parseRegisters`, `parseInfoRegisters`,
`parseAccountingRegisters`, `parseChartsOfAccounts`, `parseConstants`,
`parseDataProcessors`, `parseCommonModules`, `parseScheduledJobs`.
`parseEnumerations` переписать на хелпер (поведение не меняется).

**Тесты.** Регрессионные по образцу `parsedir_test.go` (`TestParseDirEnumerations`):
плоский `Имя.xml` без каталога — минимум для справочника, константы, регистра
сведений. Проверять и счётчики отчёта конвертации.

### 1.2 Препроцессорные директивы `#Область` / `#Если`

**Проблема.** `WriteModules` (`writer/dsl.go:73`) пишет исходник `.bsl` в
`.module.os` дословно. Лексер DSL не знает символа `#` → загрузка конфигурации
падает: `expected Procedure or Function, got "#"`.

**Решение, два уровня:**

1. **Конвертер** — функция `sanitizeBSL(src string) string` в `writer/dsl.go`:
   удаляет строки-директивы `#Область`, `#КонецОбласти`, `#Если … Тогда`,
   `#ИначеЕсли … Тогда`, `#Иначе`, `#КонецЕсли` (регистронезависимо, с учётом
   ведущих пробелов; англоязычные варианты `#Region`/`#EndRegion`/`#If`/`#EndIf`
   тоже). Содержимое блоков `#Если` сохраняется. Применить во всех местах, где
   `.bsl` попадает в `.os`: `WriteModules`, `WriteDSLStubs` (там исходник идёт в
   комментарий — не критично, но единообразно), обработки (`writer/yaml.go:391`).
2. **Лексер DSL** (`internal/dsl/lexer`) — символ `#` в позиции начала лексемы
   приводит к пропуску до конца строки (как комментарий `//`). Защищает любой
   копипаст кода из 1С, не только конвертер.

**Тесты.** Юнит на `sanitizeBSL`; тест лексера (`#Область Текст` между процедурами
не ломает разбор); интеграционный — модуль с `#Область` парсится парсером DSL.

### 1.3 Перечисления: `EnumRef.X` → `enum:X`

**Проблема.** `parser1c/typemap.go:38` деградирует `EnumRef.X` /
`ПеречислениеСсылка.X` в `string` с предупреждением «перечисление → string», хотя
платформа поддерживает тип `enum:Имя` (`metadata/yaml.go:230,246`), а конвертер сам
пишет `enums/*.yaml`.

**Решение.** Маппить в `"enum:" + ИмяПеречисления` (имя — через `extractRefName`,
с учётом `RefObject`). Предупреждение убрать.

**Тесты.** Юнит на typemap; интеграционный — поле справочника с `EnumRef.Статус` →
в YAML `type: enum:Статус`.

### 1.4 Формы обработок

**Проблема.** `ProcessorMeta` (`parser1c/types.go:119`) не имеет поля `Forms` —
формы обработок не импортируются вовсе. Платформа при этом managed-формы обработок
поддерживает: `loadProcessorForms` (`internal/project/loader.go:284`) грузит их из
`forms/<имя-обработки-в-нижнем-регистре>/`.

**Решение.**
- `ProcessorMeta.Forms []FormSource`; в `parseDataProcessors` — вызов `scanForms`
  (как у справочников/документов, metadata.go:319).
- В `convert.go` шаг импорта форм (строка ~149, через `onec_forms`) включает формы
  обработок; целевой каталог — `forms/<lower(имя обработки)>/<форма>.form.yaml`.
- Комментарий к `FormSource.Entity` обновить («справочник/документ/обработка»).
- Счётчик форм в отчёте учитывает формы обработок.

**Тесты.** Выгрузка с обработкой и формой → появляется
`forms/<обработка>/<форма>.form.yaml`, счётчик отчёта корректен.

### 1.5 Макеты обработок → layout-скаффолд

**Проблема.** Все макеты идут в `printforms/<owner>_<name>.yaml` с
`document: <владелец>`, но `PrintForm.Document` резолвится только в
документ/справочник — для обработки привязка мёртвая («МАКЕТЫ ИМПОРТИРОВАНЫ БЕЗ
ПРИВЯЗКИ К ОБРАБОТКЕ»).

**Решение.** В `writer/templates.go` макеты с владельцем-обработкой исключаются из
`printforms/`. Вместо этого на обработку генерируется один скаффолд
`src/<имя>.proc.layout.yaml` (его подхватывает `FindLayoutFile` для
`src/<имя>.proc.os`): каждый макет 1С → именованная область с одной TODO-ячейкой
(`имя области = имя макета`). Исходники макетов копируются рядом:
`src/<имя обработки>_<макет>.src.<ext>`. DSL-привязка работает сразу:
`Макет.Область("ИмяМакета")`. В отчёте конвертации — отдельная строка о созданных
layout-заглушках. Макеты документов/справочников/отчётов продолжают идти в
`printforms/` как раньше.

**Тесты.** Выгрузка с обработкой и двумя макетами → один `*.proc.layout.yaml` с
двумя областями + два `.src.*`; в `printforms/` ничего не попало.

## PR 2 — `fix/49-i18n-messages` (ишью #49)

### 2.1 Лаунчер: Accept-Language

**Проблема.** `resolveLang` (`internal/launcher/handlers.go:638`) вызывает
`i18n.Resolve("", "ru", acceptHeader, …)`, а `Resolve` (`internal/i18n/i18n.go:139`)
возвращает `baseLang="ru"` сразу, не доходя до Accept-Language. Лаунчер всегда
русский, хотя шаблоны переведены (45 t-вызовов) и словарей 16, включая `de.json`.

**Решение.** `i18n.Resolve("", "", r.Header.Get("Accept-Language"), launcherBundle)`
— финальный фолбэк `"ru"` уже встроен в `Resolve`. Захардкоженный
`<html lang="ru">` в `templates.go`, `configurator_tmpl.go`, `cfgauth.go`,
`forms_tmpl.go` → `{{.Lang}}` там, где язык доступен в данных шаблона.

**Тесты.** `resolveLang` с `Accept-Language: de` → `"de"`; без заголовка → `"ru"`;
с неизвестным языком → `"ru"`.

### 2.2 Пакет `i18nerr`

Новый пакет `internal/i18n/i18nerr`. Ошибка несёт русский шаблон-ключ и аргументы
раздельно:

```go
func Errorf(key string, args ...any) error   // key — русский fmt-шаблон («неизвестная таблица %s»)
func New(key string) error
func Localize(err error, b *i18n.Bundle, lang string) string
```

- `Error()` рендерит шаблон по-русски — поведение в логах/CLI не меняется.
- `Localize`: если ошибка (или любая в цепочке `errors.Unwrap`) — `i18nerr`,
  перевести шаблон через `Bundle.T(lang, key)` и подставить аргументы; иначе —
  exact-match `T(lang, err.Error())`. Ключ словаря = русский шаблон с `%s`/`%d`.
- Совместимость с `errors.Is/As/Unwrap` (поддержка обёртывания `%w` не нужна —
  шаблоны конечные; при необходимости оборачивать снаружи `fmt.Errorf("…: %w", е)`).

`tools/i18ncheck` сейчас собирает только шаблонные ключи `{{t $.Lang "..."}}` —
расширить его сканированием Go-исходников на вызовы `i18nerr.Errorf("...")` и
`i18nerr.New("...")`, чтобы pre-commit ловил непереведённые ключи ошибок.

### 2.3 Прогон сообщений через i18nerr

Конвертировать русскоязычные `fmt.Errorf`/`errors.New`/строки сообщений в
`i18nerr` в слоях, чьи сообщения доходят до браузера (счёт по состоянию на
2026-06-11): `internal/storage` (~31), `internal/ui` (~40),
`internal/dsl` (~17, пути `userError`/`dslStop`), `internal/query` (~2),
`internal/launcher` (~24). На границах HTTP-ответов (`internal/ui` хендлеры, REST,
лаунчер) — вывод через `i18nerr.Localize(err, bundle, lang)`.

**Сознательные исключения (фиксируются здесь, чтобы не было спора на ревью):**
- CLI-вывод (`onebase check`, `migrate`, `procrun`, …) — инструмент разработчика,
  остаётся русским.
- Сообщения прикладных конфигураций (`Error("Номер обязателен")` из `.os`-кода,
  тексты в YAML) — ответственность конфигурации, не платформы.
- Внутренние ошибки, никогда не доходящие до UI (например, ошибки инициализации в
  `main`) — по ситуации, без фанатизма.

### 2.4 Словари

Все новые ключи получают полный перевод в `en.json` и `de.json` (тестировщик из
#49 — немецкоязычный). Остальные 14 локалей — фолбэк на русский ключ; добивка
отдельной задачей позже. Это удовлетворяет pre-commit `i18ncheck` (требует перевода
ключа хотя бы в одной нерусской локали).

**Тесты.** Юнит `i18nerr` (рендер по-русски, Localize en/de, фолбэк, цепочки
Unwrap); интеграционный — HTTP-запрос с `Accept-Language: en` получает переведённое
сообщение об ошибке storage.
