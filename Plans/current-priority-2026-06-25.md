# Текущий приоритет развития onebase

Дата ревизии: 2026-06-25.

## Рекомендуемый порядок

1. **Стабилизационный спринт: lint-clean shipped examples/templates.**
   Новый `onebase lint` уже выявляет предупреждения в поставляемых
   `examples/` и `templates/`: неизвестные YAML-ключи, объекты без RBAC-ролей,
   мёртвые DSL-процедуры. Пока примеры не приведены к нулю предупреждений,
   нельзя включать lint-gate в CI для всего набора: пользователи быстро
   привыкнут игнорировать вывод инструмента.

2. **План 76 — готовность к многопользовательской нагрузке.**
   Первый срез: RBAC в текущем REST API, лимиты/пагинация REST list, атомарная
   optimistic locking запись, индексы под табличные части и большие списки,
   server-side reference picker. Это самый практичный шаг, если база пойдёт к
   100+ активным пользователям.

3. **План 34 F3 — автодоступ к полям ссылок (`this.X.Y`, `Стр.X.Y`).**
   В examples уже встречается `this.Задача.Проект`, но текущий runtime требует
   обходного `ЗначениеРеквизитаОбъекта(Ссылка, "Реквизит")`. Это небольшой, но
   важный разрыв между DSL и ожидаемой 1С-подобной моделью.

4. **План 60B — marketplace конфигураций.**
   Версионирование конфигурации в БД и UI истории уже реализованы. Marketplace
   целесообразно брать после lint-clean examples/templates, потому что именно
   они станут первыми устанавливаемыми конфигурациями.

5. **План 26 — REST API v2.**
   Делать после guardrails из плана 76: v2 должен сразу наследовать RBAC,
   пагинацию, токены, OpenAPI и лимиты. Без этого новый API закрепит старые
   многопользовательские риски.

## Зафиксированные находки аудита

- **План 60A уже не нужно брать.** Ядро `_config_versions`, snapshots, diff,
  rollback, UI истории и export ZIP/OBZ уже есть в коде; впереди только
  marketplace-часть.
- **План 43.2 уже закрыт готовым PR #203.** После merge он должен уйти из
  активного списка.
- **План 76 стоит зафиксировать как следующий guardrail-план.** Текущий REST API,
  большие списки, reference-options, конкурентная запись и тяжёлые отчёты
  требуют ограничителей до полноценного REST API v2.
- **`onebase check` зелёный, но `onebase lint` вскрывает качество examples.**
  Основные коды: `metadata.unvalidated-key`, `rbac.object-without-role`,
  `dsl.dead-procedure`.
- **Особая находка:** `examples/tasks/src/учёт_времени.posting.os` привязан к
  имени файла с подчёркиванием и использует `this.Задача.Проект`; это хороший
  тест-кейс для нормализации имён posting-файлов и/или Plan 34 F3.
