# 104 — Табличные части и хук ПриЗаписи у объектов справочников в DSL

## Проблема

`Справочники.X.Создать()` и `Ссылка.ПолучитьОбъект()` возвращали
`interpreter.CatalogRecordWriter`, который умеет только поля шапки:

- табличные части недоступны (`Кл.Контакты.Добавить()` → «Метод Добавить
  вызван у Неопределено»);
- запись шла через `storage.WriteCatalogRecord` **мимо** DSL-хука `ПриЗаписи`
  (`<справочник>.os`), веб-хуков и обычного пути сохранения.

У документов всё это давно есть (`docWriter` в `internal/ui/dsl_documents.go`).
Асимметрия блокировала обработки импорта (заполнение ТЧ справочников из
DSL) и делала поведение хуков зависимым от того, откуда записан объект.

## Решение

1. **`interpreter.CatalogObjectFactory`** (`internal/dsl/interpreter/catalogs_proxy.go`) —
   необязательная фабрика объектных обёрток. Подключается цепочкой
   `WithObjectFactory(...)` к `CatalogsRoot` (и к standalone `CatalogProxy` —
   менеджеру ссылок). Если фабрика задана, `Создать()` и `LoadObject()`
   делегируются ей; без фабрики поведение прежнее (`CatalogRecordWriter`).

2. **`ui.catWriter`** (`internal/ui/dsl_catalogs.go`) — объект справочника
   уровня ui, симметричный `docWriter`: `runtime.Object` с ТЧ (`tpProxy`),
   методы `Записать()/Прочитать()/ЭтоНовый()/УстановитьЗначение()`.
   Сохранение — через **`entityservice.Save`**: тот же путь, что у веб-формы
   (ПриЗаписи, ТЧ, движения, веб-хуки, планы обмена, RLS-проверки).

3. `tpProxy` обобщён: держит `*runtime.Object` вместо `*docWriter` и
   обслуживает ТЧ обоих видов объектов.

4. Подключение: `buildDSLVars` (обработки, procrun, формы) и
   `refManagerFor` (ссылки из колонок БД) передают фабрику
   `s.catObjectFactory(ctxSrc)`.

5. `entityservice.Save` присоединяется к уже открытой DSL-транзакции через
   `WithTxIfNeeded`. Побочные эффекты после записи (webhook) ставятся в
   post-commit callback внешней транзакции и не публикуются при rollback.

## Тесты

`internal/ui/dsl_catalogs_test.go`:
- создание справочника с ТЧ из DSL → строки в БД, хук ПриЗаписи видит ТЧ
  через `this` и пишет вычисленное поле;
- `ПолучитьОбъект()` возвращает объект с загруженной ТЧ; очистка/дозапись
  строк при повторной записи (обновление, не дубль);
- без фабрики — прежний `CatalogRecordWriter` (обратная совместимость).
- явная `НачатьТранзакцию`: rollback не оставляет запись, commit сохраняет её
  без второй/зависшей SQLite-транзакции.
