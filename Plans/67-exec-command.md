# Этап 67 — Выполнение команд ОС из DSL (`ВыполнитьКоманду`) за флагом AllowExec

**Статус:** 🔲 Запланировано

> Источник: предложение пользователя — «выполнение команды системы с ожиданием
> выполнения».

## Контекст

В DSL сегодня **нет** запуска внешних процессов: `os/exec` используется только в
инфраструктуре (`internal/launcher`, `internal/backup`, `internal/cli`,
`cmd/onebase`), но не в интерпретаторе. Аналог 1С — `ЗапуститьПриложение` /
`КомандаСистемы(..., ОжиданиеЗавершения=Истина)`, который ждёт завершения и
возвращает код возврата. Запрошен именно синхронный вариант («с ожиданием»).

Реализация самого вызова тривиальна (`exec.CommandContext(...).Run()` уже ждёт
завершения и даёт код возврата). **Главное в этом плане — не вызов, а граница
безопасности.** Это самая мощная возможность из обсуждавшихся: запуск
произвольной команды ОС = **исполнение произвольного кода на сервере (RCE)**.

### Модель угроз

`ВыполнитьКоманду` **строго опаснее** уже гейтящихся в движке сети и файлов, и
обнуляет существующую песочницу:

- наличие запуска процессов делает бессмысленными запреты `AllowNet`/`AllowFile`
  (`internal/dsl/interpreter/sandbox.go`) и сетевой предохранитель (план 62):
  всегда можно вызвать `curl`/`powershell`/`bash` и обойти их;
- под удар попадают **недоверенные** источники кода: ИИ-генерация (план 57,
  `RestrictedProfile`), marketplace конфигураций (план 60), demo-режим;
- инъекция команд, если принимать «строку для shell»;
- зависший процесс держит воркер/запрос.

В desktop-однопользовательском режиме (где DSL пишет сам владелец машины — ср.
комментарий в `internal/dsl/interpreter/builtins_files.go`) запуск команд
приемлем. В серверном/многопользовательском/demo — это критический риск.

**Вывод:** делаем, но строго как **gated-возможность**, зеркально устройству
файлового сэндбокса и сетевого предохранителя (план 62, `internal/ui/network_guard.go`
+ `internal/storage/network_switch.go`).

## Синтаксис / DSL

```bsl
// Команда и аргументы — РАЗДЕЛЬНО (без shell), таймаут обязателен (сек).
Рез = ВыполнитьКоманду("git", Массив("status", "--porcelain"), 30);

Сообщить("Код возврата: " + Рез.КодВозврата);
Сообщить(Рез.СтандартныйВывод);
Если Рез.КодВозврата <> 0 Тогда
    Сообщить("Ошибка: " + Рез.ОшибочныйВывод);
КонецЕсли;
```

Сигнатура:

```
ВыполнитьКоманду(Команда, Аргументы = Неопределено, Таймаут = 30,
                 РабочийКаталог = "", ПеременныеСреды = Неопределено)
```

- `Команда` — путь/имя исполняемого файла (без shell-интерпретации).
- `Аргументы` — Массив строк (каждый аргумент — отдельный элемент; **не** склейка).
- `Таймаут` — секунды; по истечении процесс (и его группа) убивается, поднимается
  ошибка (ловится `Попыткой`). Согласуется с `MaxWallClock` песочницы.
- `РабочийКаталог` — необязательный; при включённом файловом сэндбоксе
  ограничивается рабочим каталогом базы.
- `ПеременныеСреды` — необязательное Соответствие; иначе — минимальное окружение.

Если возможность выключена (см. ниже) — вызов сразу поднимает пользовательскую
ошибку «выполнение команд ОС запрещено в этом режиме», перехватываемую `Попыткой`
(как `checkNet` для сети).

## Объекты DSL

`РезультатКоманды` — Структура результата:

| Поле | Тип | Описание |
|---|---|---|
| `.КодВозврата` | число | exit code (0 — успех) |
| `.СтандартныйВывод` | строка | stdout (обрезается по лимиту) |
| `.ОшибочныйВывод` | строка | stderr (обрезается по лимиту) |
| `.Завершилась` | булево | `Ложь`, если убита по таймауту |

## Безопасность (ядро плана)

1. **Выключено по умолчанию.** Новый флаг `AllowExec bool` в `SandboxProfile`
   (zero value = `false`). `RestrictedProfile()` — всегда `false`.
2. **Глобальный переключатель базы.** Возможность включается явной настройкой
   (`exec.enabled` в `_settings`, по образцу `net.enabled` плана 62) — только для
   доверенных/локальных баз; админ-гейт на переключение; **сбрасывается при
   restore** (как предохранитель сети). Опционально дополнительный «потолок» через
   env (`ONEBASE_ALLOW_EXEC`).
3. **Без shell.** Команда и аргументы — раздельно (`exec.CommandContext`), без
   `sh -c`/`cmd /c`. Это убирает инъекцию команд. Shell-режим — только отдельным
   явным параметром в будущем, не в этом плане.
4. **Обязательный таймаут** через `context` + kill process group; защищает воркер.
5. **Лимит вывода** (например 1 МиБ на поток) — против OOM.
6. **Аудит.** Каждый вызов пишется в журнал (`internal/storage/audit.go`,
   таблица `_audit`): пользователь, команда, аргументы, код возврата.
7. **Согласование с сэндбоксом.** `РабочийКаталог` уважает файловый сэндбокс;
   demo-режим (недоверенный код) — `AllowExec=false` принудительно.

## Хранилище / SQL

- Флаг `exec.enabled` в существующей таблице настроек `_settings` (как
  `net.enabled`). Отдельной схемы не требуется.
- Запись в существующий `_audit` на каждый вызов.

## Изменения в коде

**`internal/dsl/interpreter/exec_builtins.go`** (новый) — по образцу
`http_builtins.go`:
- тип-гейт `ExecGuard func() error` (как `NetGuard`); `checkExec(guard)` —
  паникует userError при запрете;
- `NewExecFunctions(guard ExecGuard) map[string]any` → `ВыполнитьКоманду`/
  `ExecuteCommand`; реализация на `exec.CommandContext` с таймаутом, kill group,
  лимитом вывода, сборкой `РезультатКоманды`.

**`internal/dsl/interpreter/sandbox.go`**:
- добавить `AllowExec bool` в `SandboxProfile`;
- в `Vars()` при `!AllowExec` внедрить deny-`ExecGuard` поверх exec-функций
  (как уже сделано для сети/файлов/ИИ) — мерж после обычных переменных;
- `RestrictedProfile()` оставляет `AllowExec=false` (явно, для читаемости).

**`internal/dslvars/dslvars.go`**:
- поле `ExecGuard func() error` в `Common` (рядом с `NetGuard`); внедрять
  `NewExecFunctions(c.ExecGuard)` в общий namespace. По умолчанию guard
  запрещает (nil → deny), чтобы фоновые задания/сервисы не получали exec без
  явного включения.

**`internal/dsl/interpreter/builtins.go`**:
- добавить `ВыполнитьКоманду`/`ExecuteCommand` в `KnownBuiltinNames` — чтобы
  `onebase check` знал функцию и не ругался «неизвестное имя».

**Переключатель и guard (по образцу плана 62):**
- `internal/storage/settings.go` (+ `network_switch.go`-аналог) — чтение/запись
  `exec.enabled`.
- `internal/ui/exec_guard.go` (новый, как `network_guard.go`) — guard, читающий
  настройку; прокидывается в `dslvars.Common.ExecGuard` во всех путях запуска DSL
  (обработки/сервисы/страницы/задания).
- админ-UI: переключатель «Разрешить выполнение команд ОС» рядом с
  предохранителем сети; по умолчанию выключен; предупреждение о риске.

**Сброс при restore** — добавить `exec.enabled` к тому, что обнуляется при
восстановлении из бэкапа (там же, где `net.enabled`).

**Доки:** `onebase ai-guide`/`AGENTS.md` — раздел про `ВыполнитьКоманду` с явным
предупреждением; `DEVELOPER.md`/`CLAUDE.md` (DSL-грабли) — что по умолчанию
выключено.

## Тесты

- `internal/dsl/interpreter/exec_test.go`:
  - успешный запуск кросс-платформенной команды (например `go env` или
    `cmd /c echo` под Windows / `echo` под *nix через `runtime.GOOS`-ветку в
    тесте) → `КодВозврата=0`, stdout захвачен;
  - ненулевой код возврата читается; stderr захвачен;
  - таймаут убивает долгую команду, `Завершилась=Ложь`, ошибка ловится `Попыткой`;
  - **deny при `AllowExec=false`** (профиль/без guard) → userError, ловится
    `Попыткой`; `RestrictedProfile` запрещает;
  - инъекция: аргументы не интерпретируются shell (`ВыполнитьКоманду("echo",
    Массив("$(rm -rf /)"))` печатает литерал, ничего не исполняя).
- `internal/storage` — флаг `exec.enabled` (default false, read/write, сброс при
  restore).
- `onebase check` — `ВыполнитьКоманду` распознаётся (нет «unknown builtin»).

## Verification (вручную, end-to-end)

```bash
# По умолчанию выключено:
onebase procrun --project examples/trade --sqlite trade.db --proc ТестКоманды
#  → «выполнение команд ОС запрещено» (ловится Попыткой)

# Включить в админке (или _settings exec.enabled=1), затем:
onebase procrun --project examples/trade --sqlite trade.db --proc ТестКоманды
#  → команда отрабатывает, в журнале аудита видна запись о вызове

# restore из бэкапа → exec.enabled снова выключен (как предохранитель сети)
```

## Дальнейшее развитие (необязательно)

- Явный **shell-режим** отдельным параметром (для пайпов) — с удвоенными
  предупреждениями.
- **Асинхронный** запуск (без ожидания) + опрос статуса.
- **Белый список** разрешённых команд в настройках базы (точечнее, чем общий
  флаг).
- Потоковый вывод (стрим) для долгих команд.
