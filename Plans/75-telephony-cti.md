# План 75 — Телефония / CTI: панель софтфона, события звонков, click-to-call

**Статус:** ✅ Сделано — режим «монитор» (WebRTC отложен в отдельную работу). Реализация: `examples/callcenter` + общий модуль `Телефония` (события/скрин-поп/click-to-call) + панель `pages/Телефония` + глобальная всплывашка в `internal/ui/templates.go`. Длительность — из события hangup (без CDR/регламента).

## Контекст

Запрос «из жизни»: легаси-конфигурация 1С содержит развитый блок колл-центра на базе **МИКО «Простые звонки» / Asterisk** — нативная панель-софтфон (внешний компонент тонкого клиента), связь с АТС по AJAM/AMI, история звонков прямым SQL к CDR-базе (`PT1C_cdr`/`cel`), SOAP-сервисы `CallMonitoring`/`asteriskEvent`, очереди/операторы/обзвоны. При переносе на onebase нативную панель портировать нельзя, а прямого SQL к чужой БД у onebase нет. Зато всё общение с АТС там идёт через **стандартные интерфейсы Asterisk** — значит, onebase может говорить с АТС напрямую.

Цель плана — дать onebase **обобщённый CTI-механизм** (не порт проприетарной панели, а полезный всему сообществу): приём событий звонка, всплывашка-«скрин-поп», click-to-call и веб-панель софтфона. Generic-стек: Asterisk **ARI/AMI** или **любой облачный провайдер с webhooks/REST**; аудио — браузерный **WebRTC** (SIP.js). onebase выступает системой учёта и REST-бэкендом.

Базис уже есть: HTTP-сервисы (План 61, входящие события), HTTP-клиент (План 19, originate в ARI), Страницы с действиями (План 66), real-time-шина (План 74, скрин-поп), готовый документ `Звонок` в `examples/crm`.

## Цель

Эталонный переиспользуемый колл-центр на onebase: входящий звонок создаёт/находит `Звонок` и подсвечивает оператору карточку клиента в реальном времени; из карточки клиента — звонок в один клик; веб-панель показывает состояние вызова. Без проприетарных компонентов и без прямого доступа к CDR-БД.

## Архитектура

Нормализованные события звонка (ringing/answered/hangup) приходят на HTTP-сервис onebase `POST /hs/telephony/event` — от dialplan Asterisk (AGI / `CURL()` / Stasis) **или** от webhook провайдера. DSL-обработчик резолвит номер → `Клиент`, пишет `Звонок`, публикует уведомление целевому оператору (План 74) → скрин-поп. Click-to-call — DSL-хелпер через HTTP-клиент в Asterisk ARI `POST /ari/channels` (originate). Панель — веб-компонент: **режим «монитор»** (по умолчанию; аудио на стороне АТС/телефона, панель показывает состояние и кнопки) и опциональный **режим WebRTC** (браузер = софтфон, SIP.js к `chan_pjsip` по WSS). CDR/длительность — регламентным заданием по REST (ARI/адаптер), не прямым SQL.

## Синтаксис / UX

### 1. Входящие события — `services/telephony.yaml`
```yaml
name: Телефония
root_url: telephony
auth: hmac                 # X-Webhook-Signature = hex(HMAC-SHA256(тело, secret))
secret: "${env:TELEPHONY_SECRET}"
templates:
  - template: /event
    methods: { POST: СобытиеЗвонка }
  - template: /lookup
    methods: { GET: КтоЗвонит }
```

### 2. Обработчик — `src/Телефония.service.os`
```bsl
Функция СобытиеЗвонка(Запрос) Экспорт
    С = Запрос.ТелоJSON();                 // {state, direction, from, to, operator, call_id}
    Клиент = НайтиКлиентаПоНомеру(С.Получить("from"));
    Если С.Получить("state") = "ringing" И С.Получить("direction") = "in" Тогда
        ОтправитьУведомление(С.Получить("operator"), "звонок.входящий",
            Новый Структура("номер, клиент, id",
                С.Получить("from"),
                ?(Клиент = Неопределено, "", Клиент.Наименование),
                ?(Клиент = Неопределено, "", СсылкаНаОбъект(Клиент))));
    КонецЕсли;
    Если С.Получить("state") = "hangup" Тогда
        ЗафиксироватьЗвонок(С, Клиент);     // создать/обновить документ Звонок
    КонецЕсли;
    Возврат Новый Структура("ok", Истина);
КонецФункции
```

### 3. Click-to-call — `src/Телефония.module.os` (хелпер на HTTP-клиенте → ARI)
```bsl
Функция Позвонить(КудаНомер, ОператорЭкстеншен) Экспорт
    Соединение = Новый HTTPСоединение(ХостARI(), 8088, Ложь, 10);
    Ресурс = "/ari/channels?endpoint=PJSIP/" + КудаНомер
        + "&extension=" + ОператорЭкстеншен + "&context=from-internal&priority=1"
        + "&api_key=" + ПользовательARI() + ":" + ПарольARI();
    Запрос = Новый HTTPЗапрос(Ресурс);
    Возврат Соединение.ОтправитьДля(Запрос, "POST").КодСостояния = 200;
КонецФункции
```
Кнопка «Позвонить» на карточке клиента — серверным действием Страницы (`КнопкаДействие`, План 66).

### 4. Панель — страница `pages/Телефония`
- **Режим «монитор»** (по умолчанию): подписка JS на `onebase:звонок.входящий` (План 74) → карточка входящего с кнопкой «Открыть клиента»; список активных звонков; click-to-call.
- **Режим WebRTC** (опционально): SIP.js-софтфон в браузере (приём/сброс/перевод), `chan_pjsip` WSS; на INVITE — `GET /hs/telephony/lookup?number=` для скрин-попа.

**Опционально — глобальная всплывашка входящего:**
- `internal/ui/templates.go` — маленький слушатель `onebase:звонок.входящий` в общем layout (тост «Входящий … / Открыть карточку»), чтобы скрин-поп работал на любой странице, не только на панели. Переиспользует JS-мост Плана 74.

> **Сознательно НЕ делаем в ядре:** проприетарную панель МИКО; прямой SQL к CDR-БД (нет «Внешних источников данных» — длительность тянем по REST); SOAP (вместо `CallMonitoring`/`asteriskEvent` — HTTP-сервис). Это и делает фичу generic и апстрим-пригодной.

## Безопасность
- `services/telephony` — `auth: hmac`/`token`, `secret` из `${env:…}`, rate-limit (всё из Плана 61); тело — `MaxBytesReader`.
- `Позвонить`/originate — гейт по роли `Оператор`; креды ARI — в `Константы`/env, не в коде.
- Панель и документы звонков видны только роли `Оператор` (ролевой гейт Страниц/сервисов).
- Скрин-поп шлётся **только целевому оператору** (адресная публикация Плана 74), не широковещанием.
- WebRTC: WSS + TLS; STUN/TURN при необходимости; SIP-креды выдавать по сессии, не зашивать в JS.

## Тесты
- `examples/callcenter` проходит `onebase check` (валидны сервис, страница, модули, запросы).
- Юнит на `src/Телефония.service.os` в стиле `internal/ui/services_test.go`: POST `state=ringing,direction=in` → опубликовано уведомление целевому оператору (мок `Publisher`); POST `state=hangup` → создан `Звонок` с верным номером/направлением.
- `lookup` возвращает клиента по номеру (200/JSON) и 404 для неизвестного.
- При WebRTC-вендоринге — `assets_test.go` отдаёт `/vendor/sip/sip.min.js`.

## Verification (вручную, end-to-end)
```bash
# Тестовый Asterisk (Docker): ARI :8088, chan_pjsip (+WSS :8089 для WebRTC),
# CDR включён, dialplan на входящем делает CURL → POST /hs/telephony/event.
onebase run --project examples/callcenter --sqlite cc.db --port 8080

# 1) Имитируем входящий (или звоним софтфоном на экстеншен оператора):
curl -X POST http://localhost:8080/hs/telephony/event \
  -H "X-Webhook-Signature: sha256=<hmac>" \
  -d '{"state":"ringing","direction":"in","from":"+79990001122","operator":"ivan"}'
# → у оператора ivan в браузере всплыла карточка входящего.

# 2) Click-to-call с карточки клиента → поднимается трубка на экстеншене оператора (ARI originate).
# 3) hangup-событие → создан документ Звонок; регламент проставил длительность.
```

## Связанные планы
- План 74 — real-time-шина (скрин-поп). **Обязательная зависимость.**
- План 61 — HTTP-сервисы (приём событий); План 19 — HTTP-клиент (originate в ARI); План 29 — вебхуки (исходящие оповещения по звонкам).
- План 66 — Страницы (панель, кнопки-действия); План 31 — `device-agent` (альтернатива: телефонный драйвер с локальным AMI/ARI, если нужен коннектор на месте оператора).

## Эстимейт
HTTP-сервис событий + скрин-поп + пример: ~2 дня. Click-to-call (ARI-хелпер) + панель-монитор: ~2 дня. Сверка длительности (REST): ~1 день. Режим WebRTC (вендоринг SIP.js + софтфон + lookup): ~3–4 дня (опционально). **Итого ≈ 1 неделя (+ WebRTC ≈ ещё неделя).**
