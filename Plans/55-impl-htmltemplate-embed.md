# План 55 (импл.) — конфигуратор: html/template + вынос фронта в go:embed

Дизайн-спека к этапу 2 плана [55](55-monolith-split-embed-frontend.md). Закрывает
XSS-долг плана [47](47-audit-fixes-2026-06-06.md) §1.3 и выносит статику фронта
конфигуратора в файлы. Авторство задания — Иван Титов.

**Статус:** дизайн утверждён 2026-06-19; реализация — Фаза 1 первой, Фаза 2 после.

---

## 1. Контекст и цель

`internal/launcher/configurator_tmpl.go` — 6730 строк HTML/JS/CSS в виде Go
backtick-литералов, собираемых в **один** набор `text/template` (единственный
`.Parse(cfgCSS + … + cfgFoot)` на строке 105; единственный продакшен-рендер —
`renderCfg` → `cfgTmpl.ExecuteTemplate(w, "cfg-main", data)` в `configurator.go:2067`).

Две проблемы:
- **Безопасность.** `text/template` не экранирует. Имена конфигурации
  (сущности/поля/отчёты) и эхо `r.FormValue("entity")` в блоке ошибки
  попадают в HTML и в `<script>` сырыми → XSS (план 47 §1.3, «отложено»).
- **DX.** ~5K строк HTML/JS в литералах: нет подсветки, eslint, форматирования;
  опечатка в JS ловится только в браузере.

Замер из разведки: ~62% файла **по сути статика** (`cfgCSS` 100%; гигантский
`<script>` в `cfgFoot` ~3842 строки — статичен на ~97%, лишь 12 интерполяций
данных + 91 i18n), ~36% — действительно динамический рендер дерева конфигурации
(`cfgTabTree` ~1700 строк).

**Цель:** закрыть XSS-долг и ужать монолит, **не меняя поведения** (главный риск —
регресс рендера).

## 2. Подход и последовательность

Движок — один набор шаблонов, смешать `text/template` и `html/template` в нём
нельзя → переключение возможно только целиком. Отсюда фазы:

- **Фаза 1 (безопасность, первой):** переключить конфигуратор на `html/template`.
  Меньше и нужнее, тематически продолжает security-работу. Контейнерный дифф.
- **Фаза 2 (DX, после Фазы 1):** вынести статику CSS/JS в `internal/launcher/static/`
  + `go:embed`, динамику кормить bootstrap-блобом.

Можно остановиться после Фазы 1. Этап 3 плана 55 (inline-JS в `ui/templates.go`) —
**вне рамок**, отдельный план.

Отвергнуто: (B) поэтапная миграция по под-шаблонам — невозможна без предварительного
раскола набора (это и есть работа Фазы 2); (C) ручное `{{.X | h}}` в `text/template` —
762 сайта, легко пропустить, игнорирует контекст-зависимость.

## 3. Фаза 1 — миграция на html/template

Одно согласованное изменение `configurator_tmpl.go`; поведение сохраняется, кроме
того, что интерполируемые данные начинают экранироваться по контексту.

1. **Движок.** `text/template` → `html/template` (строка 6). Конструкция
   `template.Must(New("cfg").Funcs(...).Parse(...))` по API идентична. html/template
   контекстно-зависим: экранирует по окружению (тело HTML, атрибут, `<script>`, URL).

2. **Фикс funcmap `js` (ловушка двойного экранирования).** Хелпер `js` (строка 32)
   возвращает `string` из `json.Marshal`; под html/template обычная строка в
   `<script>` экранируется ещё раз → битый JSON. Фикс: возвращать `template.JS`.
   Проверить сайты `{{js}}` (`:5866`, `:5985`) на побайтовое совпадение вывода.

3. **Аудит существующих `template.JS`-полей** (`QBSchema` `:385`, `LayoutMeta` `:388`,
   `InlineJSYaml` `static.go`). html/template распознаёт тип `template.JS` и не
   экранирует повторно → остаются корректными; подтвердить diff'ом рендера.

4. **Закрытие XSS (автоматически).** Сайты из разведки экранируются по контексту:
   - JS: `_cfgEntityNames=['{{.}}']` (`:884`/`:885`), `directId='{{.SelectedTreeID}}'`
     (`:1717`), `'<option…{{.}}>'` (`:3091`) → экранирование JS-строки.
   - HTML/атрибут/URL: `{{.Error}}` (из `r.FormValue("entity")`),
     `«{{.FieldsSavedEntity}}»`, `<title>{{.AppName}}`, ~322 интерполяции `cfgTabTree`
     (`{{.Name}}`/`{{.Title}}`/`data-id="e-{{.Name}}"`/`?entity={{.Entity}}&name={{.Name}}`).
   - Кириллица html/template'ом не экранируется (только `< > & ' "`) → обычные имена
     рендерятся без изменений, правок тестов минимум.

5. **Чистка.** Удалить мёртвую конст `cfgAdminOverlay` (`:474`; `cfg-main` её инлайнит).

6. **Аудит «намеренной разметки».** Grep по интерполяциям, которые должны выдавать
   HTML/JS. Разведка таких (и полей `template.HTML`) не нашла; найденное — обернуть
   явным обоснованным `template.HTML`/`template.JS`.

### Тесты Фазы 1
- **Регрессии:** существующие рендер-тесты `cfgTmpl.ExecuteTemplate(&buf, name, data)`
  (`configurator_tabs_render_test.go`, `langref_render_test.go`,
  `layout_editor_render_test.go` и др.) — зелёные; ассерты на данных с `'`/`&`/`<`
  правятся под экранированную форму (большинство — кириллица, не затронуты).
- **Новый `configurator_xss_test.go`:** рендер `tab-tree`/`cfg-main` с сущностью
  по имени `<img src=x onerror=alert(1)>` и `'+alert(1)+'` → payload экранирован и
  в HTML, и в `<script>` (нет сырого `<img`, нет вырывающейся кавычки).
- **Новый `configurator_render_test.go`:** smoke-рендер вкладок (tree/convert/files/
  backup) + head/foot → нет ошибки шаблона, есть якорный элемент.

### Критерий готовности Фазы 1
`text/template` ушёл из конфигуратора; `configurator_xss_test` доказывает
экранирование; рендер-тесты зелёные; ручной обход без визуальных регрессий.

## 4. Фаза 2 — вынос статики в /static + go:embed

Цель — ужать `configurator_tmpl.go` до ~2400 строк. После Фазы 1.

1. **CSS → файл (тривиально).** `cfgCSS` → `internal/launcher/static/configurator.css`;
   `<style>{{template "css"}}</style>` → `<link rel="stylesheet" href="/static/configurator.css">`.
   Каталог `static/` уже встроен `//go:embed static` и отдаётся на `/static/*`.

2. **Статический JS → файл + bootstrap-блоб (основная работа).** `<script>` из
   `cfgFoot` (~3842 строки, ~97% статика) → `internal/launcher/static/configurator.js`.
   12 значений данных + 91 i18n-строку отдавать bootstrap-объектом перед загрузкой
   внешнего скрипта:
   ```html
   <script>
     window.__cfg     = {{.Bootstrap}};  // template.JS(json): base id, имена сущностей/перечислений, флаги…
     window.__cfgI18n = {{.I18n}};        // template.JS(json): карта переводов
   </script>
   <script src="/static/configurator.js"></script>
   ```
   В вынесенном JS `{{.X}}` → `window.__cfg.X`, `{{t "ключ"}}` → `__cfgI18n["ключ"]`
   (хелпер `t()`). ~100 точек трансформации — самая трудоёмкая часть; пошагово
   распишется в плане Фазы 2.

3. **Остаётся шаблоном.** `cfgTabTree` и HTML-хром вкладок — динамические, остаются
   в `html/template` (там Фаза 1 уже навела экранирование).

4. **Порядок выноса** (низкий→высокий риск): CSS → хвостовые скрипты → главный
   скрипт + bootstrap. Мелкими коммитами, проверка страницы после каждого.

5. **Бонус CSP.** `websec.SecurityHeaders` уже подключены; вынос inline-JS открывает
   путь к строгому CSP без `inline-script` (отдельный шаг, вне рамок).

6. **eslint/prettier** на `assets/*.js` — отложено в план 56.

## 6. Риски

| Риск | Смягчение |
|---|---|
| html/template сломает место с намеренной разметкой | Grep-аудит (п.3.6); полей `template.HTML` нет |
| Двойное экранирование `template.JS`-полей и `js`-хелпера | П.3.2–3.3 + diff рендера |
| Регресс рендера при выносе JS (Фаза 2) | Golden-render + bootstrap через `template.JS(json)`, не текстовая склейка |
