# План 50: пометка на удаление ↔ проведение

**Дата:** 2026-06-09
**Issue:** [#36 — Пометка на удаление/отмена проведения](https://github.com/ivanarama/onebase/issues/36)
**Ветка:** `feature/36-deletion-mark-posting`
**Статус:** ✅ Реализовано

## Контекст

Issue #36 + комментарий тестировщика описывают три симптома вокруг взаимодействия
пометки на удаление и проведения документа:

1. **Движения остаются.** При пометке проведённого документа на удаление его движения
   регистров (`рег_*`/`инфо_*`/`акк_*`) не очищаются → остатки/обороты остаются
   «раздутыми», пока документ физически не удалят через «Удалить помеченные».
2. **Нет отмены проведения.** *(жалоба)* Пользователь не нашёл способа отменить
   проведение.
3. **Помеченный документ перепроводится.** Документ с пометкой на удаление успешно
   проводится/перепроводится — а не должен.

### Текущее состояние кода (выверено чтением)

- **Пометка на удаление** существует как колонка `deletion_mark BOOL NOT NULL DEFAULT false`
  на всех таблицах сущностей (создаётся в `storage.Migrate`, `internal/storage/migrate.go:303`).
  `storage.MarkForDeletion` (`internal/storage/deletion.go:25`) только переключает флаг —
  **движения не трогает**. → корень симптома 1.
- **Проведение** имеет три точки входа, сходящиеся на `storage.SetPosted`
  (`internal/storage/crud.go:631`) и тройке писателей движений
  (`WriteMovements`/`WriteAccountMovements`/`WriteInfoMovements`, каждый делает
  DELETE-by-registrar, при `rows=nil` — только DELETE):
  1. `ui.Server.postDocument` (`internal/ui/handlers.go:886`) — роут `POST /…/{id}/post`;
  2. `entityservice.Service.Save` ветка `isPosting` (`internal/entityservice/service.go:150`) — сабмит формы `_action=post`;
  3. `ui/dsl_documents.go` `docWriter.post` (`:428`) — DSL `Документы.X.Провести()`.
  **Ни одна** не проверяет `deletion_mark`. → корень симптома 3.
- **Отмена проведения** уже реализована на уровне формы:
  `ui.Server.unpostDocument` (`internal/ui/handlers.go:962`) + роут `POST /…/{id}/unpost`
  (`internal/ui/server.go:149`) + кнопка «Отменить проведение» в autogen
  (`templates.go:1030`) и managed-формах (`templates_managed.go:336`, добавлено
  коммитом `70625eb`, уже в `main`), право `unpost`. `unpostDocument` зовёт
  `clearMovements` (`handlers.go:943` — чистит ВСЕ рег_/инфо_/акк_ по recorder) +
  `SetPosted(false)`. → симптом 2 на уровне формы закрыт; недостаёт **списочного**
  действия и **DSL**-метода.

### Решение (выбрано с пользователем)

Опорный инвариант, по аналогии с 1С: **«проведён» и «помечен на удаление» —
взаимоисключающие состояния документа.** Из него:

- **Семантика пометки:** авто-отмена проведения. Пометка проведённого документа на
  удаление в одной транзакции снимает проведение (чистит все движения, `posted=false`)
  и затем выставляет пометку. Снятие пометки проведение **не** возвращает.
- **Объём:** полный — два корректностных исправления + списочные «Отменить
  проведение»/«Снять пометку» + DSL-методы.

**Архитектура — Вариант A** (точечные guard'ы + переиспользование примитивов).
Отвергнутый Вариант B (стейт-машина состояния в `storage`) требует прокинуть метаданные
регистров в storage-слой (Registry сейчас живёт в ui) и рефактора двух оркестраторов
проведения — несоразмерно объёму #36.

## Изменения по компонентам

### 1. Инвариант проведения — запрет проведения помеченного (симптом 3)

**storage.** Новый хелпер + типизированная ошибка:

```go
// internal/storage/deletion.go
var ErrPostingDeletionMarked = errors.New("документ помечен на удаление: проведение невозможно")

// IsMarkedForDeletion возвращает значение deletion_mark для записи.
func (db *DB) IsMarkedForDeletion(ctx context.Context, entityName string, id uuid.UUID) (bool, error)
```

**Ранний guard** (до запуска хука `ОбработкаПроведения` и до записи) в каждой из трёх
точек входа проведения — вернуть `ErrPostingDeletionMarked`:

- `ui/handlers.go` `postDocument` — после загрузки документа, до `runOnPostCtx`. В UI
  редиректить с понятным сообщением (как существующий `?posting_error=`).
- `entityservice/service.go` `Save` — в ветке `isPosting`, **до** `Upsert`/хука, чтобы
  не терять правки полей и не гонять хук впустую.
- `ui/dsl_documents.go` `docWriter.post` — до `runOnPostCtx`; ошибка пробрасывается в DSL
  (перехватывается `Попытка`).

**Backstop в storage.** В `SetPosted` при переходе в `posted=true` — повторная проверка
`deletion_mark`; если помечен → `ErrPostingDeletionMarked`. Страховка от будущей 4-й
точки входа; в норме не срабатывает, т.к. точки входа сторожат раньше.

**UI.** Прятать кнопки «Провести»/«Провести и закрыть», когда документ помечен
(`templates.go:1010-1073`, `templates_managed.go:316-386`) — добавить условие на
`deletion_mark` рядом с существующими `{{if $.CanPost}}`. «Записать» остаётся доступной
(помеченный документ можно редактировать и сохранять без проведения).

### 2. Авто-отмена проведения при пометке (симптом 1)

Общий метод на `*Server` (переиспользует существующие `clearMovements` + `SetPosted` +
`MarkForDeletion`):

```go
// internal/ui/handlers.go
// markForDeletion помечает/снимает пометку; при пометке проведённого документа
// сперва снимает проведение (чистит движения по всем регистрам, posted=false).
func (s *Server) markForDeletion(ctx context.Context, entity *metadata.Entity, id uuid.UUID, mark bool) error
```

Логика в одной транзакции (`store.WithTx`):
1. если `mark && entity.Posting` и текущий `posted==true` → `clearMovements(ctx, entity.Name, id)` + `SetPosted(ctx, entity.Name, id, false)`;
2. `store.MarkForDeletion(ctx, entity.Name, id, mark)`.

`deleteRecord` (ветка пометки, `handlers.go:1006-1016`) переключается на `markForDeletion`.
Результат: помеченный документ → 0 движений → остатки корректны немедленно. `SetPosted(false)`
уже пишет аудит-запись `unpost`, так что авто-отмена логируется.

**Право.** Пометка проведённого документа требует существующего права `delete`;
авто-снятие проведения — следствие пометки, отдельного `unpost` не требуем.
*(Решаемо: если решим строже — добавить требование `unpost` для пометки проведённого.)*

### 3. Списочные действия (симптом 2 — полное закрытие)

- `listCtxMenu` (`templates.go:945-992`) — два новых пункта:
  - **«Отменить проведение»** — для строк с `posted` и при `CanUnpost`; POST на
    существующий `/…/{id}/unpost` (хендлер/роут уже есть).
  - **«Снять пометку на удаление»** — для строк с `deletion_mark`; POST на `/…/{id}/delete?mark=0`.
- `list`-хендлер (`handlers.go`, рендер списка) — прокинуть `CanUnpost` и
  per-row `data-unpost-url`/`data-unmark-url`; `posted`/`deletion_mark` уже в строке.
- `deleteRecord` — новая ветка `mark=0` → `store.MarkForDeletion(false)` (без возврата
  проведения).
- Форма: кнопка **«Снять пометку»** когда документ помечен — симметрично существующей
  «Отменить проведение» (`templates.go`/`templates_managed.go`).

### 4. DSL-методы (полный объём)

В `internal/ui/dsl_documents.go`, симметрично существующему `.Провести()` —
тонкие обёртки над теми же хелперами `Server`:

- `Документы.X.ОтменитьПроведение(Ссылка)` → `clearMovements` + `SetPosted(false)`
  (та же логика, что `unpostDocument`).
- `Документы.X.ПометитьНаУдаление(Ссылка)` → `s.markForDeletion(…, true)`.
- `Документы.X.СнятьПометку(Ссылка)` → `s.markForDeletion(…, false)`.

Добавляются в `docProxy.CallMethod` (`:79-129`) и/или `docWriter.CallMethod` (`:256-294`),
регистронезависимо (как остальные методы DSL).

## Хранилище / SQL

Схема не меняется — `deletion_mark` и `posted` уже существуют. Новое только в Go:
`IsMarkedForDeletion` (SELECT), `ErrPostingDeletionMarked`. Учитывать диалектную
особенность SQLite (BOOL = INTEGER 0/1) — как в существующих `deletion.go:103-106`.

## Вне области

- **query/ не трогаем.** Помеченный документ авто-распроведён → движений нет → в
  Остатки/Обороты он уже не попадает; отдельная фильтрация по `deletion_mark` не нужна.
- **Миграция данных не нужна.** Существующие «помечен + проведён» документы (данные с
  багом) лечатся при первом действии: расп­роведение, повторная пометка, либо «Удалить
  помеченные» (путь физического удаления уже чистит движения).
- *Заметка (отдельный баг, вне #36):* REST `DELETE` (`internal/api/handlers.go:251-257`)
  чистит только накопительные регистры, не инфо/акк.

## Тесты

SQLite через `ConnectSQLite(ctx, tmpfile)` (PostgreSQL не нужен):

1. **Пометка проведённого → очистка движений.** Провести документ с движениями в
   рег_/инфо_/акк_; пометить на удаление; проверить: 0 движений во всех трёх,
   `posted=false`, `deletion_mark=true`.
2. **Запрет проведения помеченного.** Для каждой из трёх точек входа: пометить →
   попытаться провести → ожидать `ErrPostingDeletionMarked`, `posted` остаётся false,
   движений нет.
3. **Снятие пометки.** Снять пометку → `deletion_mark=false`, `posted` остаётся false,
   движений нет.
4. **DSL.** `Документы.X.ОтменитьПроведение/ПометитьНаУдаление/СнятьПометку` дают то же
   поведение, что и UI-пути.
5. **Регресс отмены проведения формы** (`/unpost`) не сломан.
6. `onebase check --project examples/trade` проходит.

## Verification (вручную, end-to-end)

1. `onebase run --project examples/trade --sqlite /tmp/t.db --port 8080`.
2. Создать и провести документ → убедиться, что появились движения (свёрнутые
   «таблеточки» под формой и в регистрах).
3. В списке: ПКМ по проведённому документу → «Отменить проведение» → движения исчезли,
   значок «проведён» снят.
4. Провести снова; в списке пометить на удаление → строка зачёркнута, движения
   исчезли, `posted` снят.
5. Попытаться провести помеченный (кнопка скрыта на форме; через прямой POST `/post`) →
   ошибка, состояние не изменилось.
6. ПКМ → «Снять пометку» → зачёркивание ушло, документ остаётся непроведённым.
