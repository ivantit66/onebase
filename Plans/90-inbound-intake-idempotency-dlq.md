# План 90 — Входной шлюз: идемпотентность и DLQ/карантин произвольных входящих

Дата проектирования: 2026-07-22.
Статус: ⬜ Проект (мотивирован оценкой ТЗ КЦ ICE-32, пробел **G6**:
`OneService/onebase-cc-platform-traceability-v0.1.md`). Закрывает
CC-DATA-105, CC-INT-006/007/010, CC-INT-003, CC-INT-1C-003, CC-FR-P03-007/008,
CC-ARCH-004/010. Связан с пробелом **G4** (мост RabbitMQ→HTTP) — этот план даёт
приёмную сторону, к которой мост подключается.

> **Честная оценка приоритета (в отличие от планов 89/85).** Планы 89 (правила) и
> 85 (задачи) — это «сборка» из готовых кирпичей, платформизация опциональна. Здесь
> наоборот: **гарантии приёмки — истинная платформенная ценность**, тот же класс,
> что уже реализованный `internal/exchange` (план 86). Прикладным паттерном можно
> *показать форму* контракта, но нельзя корректно сделать главное — атомарный
> «insert-if-new» ключа идемпотентности под конкурентностью и транзакционную
> связку «обработчик + отметка обработки». Поэтому рекомендуемый маршрут:
> **заход 0 (прикладной) — проверить контракт и поток; заход 1 (движок) — цель.**
> Транспорт (HTTP сейчас, AMQP потом) остаётся на краю за интерфейсом-швом.

## Контекст

OneBase умеет **принимать по HTTP** (HTTP-сервисы, план 61: `/hs/<корень>/…`,
auth `token`/`hmac`, rate-limit) и **звать наружу** (HTTP-клиент план 19, вебхуки
план 29). Чего нет — **надёжной приёмки произвольного входящего события**:

- один и тот же внешний вызов может прийти **дважды** (RabbitMQ fanout из G4 —
  at-least-once; ретраи отправителя; повтор моста после сбоя) → нужен **ключ
  идемпотентности**, чтобы не завести две заявки на одно событие;
- «битое» событие (нераспознанная схема, ошибка обработчика, конфликт) не должно
  теряться или бесконечно ретраиться → нужен **карантин (DLQ)** с безопасным
  ручным **replay**;
- надо различать **транспортный ack** (сообщение принято) и **бизнес-результат**
  (что с ним стало) — CC-INT-1C-003, CC-ARCH-004/010;
- периодическая **сверка** количества/ID/состояний с источником (CC-INT-007).

Идемпотентность уже есть в `internal/exchange` (план 86), но **только для
OneBase↔OneBase**: повторное применение пакета обмена ничего не меняет, потому что
пакет контент-адресуемый (ID объекта + версия). Для **произвольных входящих команд
из 1С/сайта/партнёров** такого стора нет. Этот план его обобщает.

### Единый конверт (CC-INT-003)

Вход нормализуется в конверт (JSON), независимо от транспорта:

```json
{
  "event_id":       "site-2026-07-22-000123",  // ключ идемпотентности
  "schema_version": "1",
  "correlation_id": "…",                        // сквозная трассировка
  "source":         "site",                     // компонент scope
  "aggregate":      "Заявка",                   // тип бизнес-объекта
  "occurred_at":    "2026-07-22T10:00:00Z",
  "payload":        { … }                       // тело события
}
```

## Заход 0 — прикладной паттерн (проверить контракт, БЕЗ правки движка)

Собирается на готовых кирпичах, демонстрирует форму и поток в `examples/callcenter`:

- **Транспорт:** HTTP-сервис (план 61) `/hs/site/lead`, `auth: hmac`, rate-limit.
- **Обработчик** `.os`: парсит конверт, создаёт `Заявка` из `payload` (стык с
  планами 89/85 — заявка сразу попадает в валидацию и маршрут согласования).
- **Идемпотентность:** справочник `ОбработанныеСобытия` (Ключ, ХэшТела, СсылкаНа
  Результат). Обработчик: `НайтиПоРеквизиту("Ключ", event_id)` → если есть, вернуть
  прежний результат; иначе создать заявку и записать событие.
- **Карантин:** документ `КарантинСобытий` (сырое тело, причина, попытки) +
  обработка `ПовторитьИзКарантина`.

**Честное ограничение захода 0 (мотивирует заход 1):**
- `НайтиПоРеквизиту + Создать` — **гонка TOCTOU**: два параллельных дубля оба не
  найдут ключ и оба создадут заявку. Пробел закрывается только storage-уровневым
  атомарным «insert-if-new» (уникальный индекс + `ON CONFLICT`).
- Обработчик и отметка «обработано» **не в одной транзакции** через HTTP-границу →
  сбой между ними даёт дубль или потерю.

Заход 0 закрывает CC-INT-006/007/010 **демонстрацией** (счастливый путь, дубль,
карантин, replay — headless-проверяемо), но под нагрузкой корректен не полностью.

## Заход 1 — платформенный примитив `internal/intake/` (цель)

### Объявление (объект конфигурации `intake/<имя>.yaml`)

```yaml
name: SiteLead
transport: http            # http (эталон); amqp — за швом, план G4/Б
endpoint: /hs/site/lead    # для http-транспорта
schema_version: "1"
idempotency:
  key: event_id            # поле конверта — ключ
  scope: [source, aggregate]   # пространство ключа
  ttl: 30d                 # окно хранения записи идемпотентности
handler: SiteLead          # процедура Обработать(Конверт) → результат
dlq:
  on: [handler_error, schema_mismatch, key_conflict]
  max_retries: 3
```

### Ядро

`intake.Ingest(name, конверт) → Результат{Статус, СсылкаРезультата, БизнесРезультат}`,
где `Статус ∈ {Принято, Дубль, Карантин, Отклонено}`:

1. канонизировать `payload` → `payload_hash` (sha256);
2. **атомарный insert-if-new** строки идемпотентности по `(intake, scope, key)` —
   уникальный индекс + `INSERT … ON CONFLICT DO NOTHING` (тот же приём, что
   `storage.AdvisoryXactLock`/optimistic_lock);
3. если строка **новая** → в **одной транзакции** (`storage.WithTx`): запустить
   `handler` (создаёт бизнес-объект) → отметить `processed` + сохранить
   `result_ref`/`business_result`. Ошибка обработчика → откат + **карантин** →
   `Статус=Карантин`;
4. если строка **есть** и `payload_hash` совпал → вернуть прежний результат
   (`Статус=Дубль`, без повторной обработки);
5. если ключ есть, а `payload_hash` **не** совпал → `key_conflict`/`schema_mismatch`
   → карантин (CC-DATA-105 «mismatch»).

`intake.Replay(intake, key)` — прогнать запись карантина заново через `Ingest`
(идемпотентность соблюдается). `MessageSource` — **шов транспорта**: `http`
реализован первым; `amqp` (нативный consumer из G4/пути Б) позже подключается сюда,
не трогая ядро.

## Хранилище / SQL

- `_intake_log` — стор идемпотентности: `intake`, `scope`, `key`, `payload_hash`,
  `status` (received/processed/quarantined), `result_ref`, `business_result`,
  `correlation_id`, `received_at`, `ttl_expires_at`. **UNIQUE(intake, scope, key)**.
- `_intake_dlq` — карантин: `intake`, `key`, `payload` (сырое), `reason`, `error`,
  `attempts`, `quarantined_at`, `replay_state`. Прообраз — `storage/webhook_log.go`.
- индексы под «открытый карантин» и очистку по TTL.

## Изменения в коде

- `internal/metadata/intake.go` (новый) — модель `Intake` (transport/endpoint/
  idempotency/handler/dlq) + валидация;
- `internal/intake/` (новый) — `Ingest`/`Replay`, канонизация+хэш, `MessageSource`
  (интерфейс), `httpSource` (эталон); зависит от `storage` и `dsl/interpreter`;
- `internal/httpservice` — маршрут intake вызывает `Ingest`, маппит
  `Результат → HTTP` (200 Принято/Дубль; 422 Отклонено; 202/5xx для карантина —
  политика ack) так, чтобы мост/отправитель корректно делал ack/nack;
- `internal/exchange` — переиспользовать `intake` для внешнего входящего контура
  (обобщение пакетной идемпотентности плана 86);
- `internal/scheduler` — регламентная **сверка** (CC-INT-007): количество/ID/
  состояния против источника;
- `internal/ui` — список **«Карантин»** + кнопка **«Повторить»** (replay), под RBAC;
- `internal/configcheck` — валидация intake (уникальность endpoint, наличие
  процедуры `Обработать`, корректность idempotency/scope).

## Права и аудит (CC-SEC, план 04)

- replay/чистку карантина делает только уполномоченная роль; обычный оператор — нет;
- события `intake.quarantine` и `intake.replay` пишутся в `_audit` (с
  `correlation_id`); сырое тело в карантине — под RBAC (потенциально ПДн → стык с
  маскированием плана 88).

## Тесты

- happy-path: конверт → создан бизнес-объект, `status=processed`, ответ 200;
- **идемпотентность**: тот же `event_id` дважды → один объект, второй `Дубль`
  (без повторной обработки); **конкурентно** (горутины) → ровно один объект
  (проверка атомарного insert-if-new);
- `payload_hash` mismatch того же ключа → карантин, не тихий дубль;
- ошибка обработчика → откат бизнес-объекта + карантин (нет полу-записи);
- replay из карантина → обработка, идемпотентность соблюдена;
- ack-семантика: маппинг Результат→HTTP-код; сверка (scheduler) находит расхождение.

## Verification

```bash
onebase run --project examples/callcenter --sqlite cc.db --port 8080
# 1. POST /hs/site/lead конверт с event_id=E1 → 200, создана Заявка.
# 2. POST тот же E1 → 200 «Дубль», второй заявки нет.
# 3. POST E1 с изменённым payload → карантин (mismatch), в списке «Карантин».
# 4. POST заведомо битого → карантин с причиной handler_error.
# 5. «Повторить» из карантина после починки → создаётся заявка, запись в аудите.
# 6. Конкурентно 10× один event_id → ровно одна Заявка.
```

## Связь с G4 (мост)

Приёмный `endpoint` intake — это ровно тот HTTP, на который постит **мост
RabbitMQ→HTTP** (G4, путь А). `MessageSource` — шов, куда позже встраивается
**нативный AMQP-consumer** (G4, путь Б), не переписывая приёмную логику. Таким
образом «настоящий брокер» вносится позже как *транспорт за интерфейсом*, повторно
используя гарантии этого плана, а не как отдельная плумбинг-ветка (см. обсуждение
топологии в оценке ТЗ, разделы G4/G6).

## Эстимейт

**Заход 0 (прикладной, проверка контракта):** HTTP-сервис + справочник
идемпотентности + документ карантина + replay + пример в `examples/callcenter`.
**≈ 2–3 дня**, силами конфигуратора; закрывает CC-INT-006/010 демонстрацией.

**Заход 1 (платформенный примитив):**

- 90A модель `intake/*.yaml` + `internal/intake` ядро (`Ingest`, канонизация+хэш,
  атомарный insert-if-new) + транзакционный обработчик: **4–5 дней**;
- 90B карантин (`_intake_dlq`) + `Replay` + UI «Карантин» + аудит: **3–4 дня**;
- 90C HTTP-транспорт + ack-семантика + `configcheck` + `MessageSource` шов: **2 дня**;
- 90D переиспользование в `exchange` + регламентная сверка + тесты: **2–3 дня**.

MVP (90A–B) ≈ 1.5 недели; полный контур (90A–D) ≈ **2–2.5 недели**. Нативный
AMQP-транспорт (G4/Б) — отдельно, только за швом `MessageSource`.

**Порядок по проекту КЦ:** заход 0 (демонстрация контракта, стык с планами 89/85) →
заход 1 (примитив в движке) → мост G4 на приёмник → нативный AMQP по потребности.
